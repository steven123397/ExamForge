# 第三版第二阶段计划

## 1. 目标

第三版第二阶段聚焦测试基线补强和 API service 提取。完成后，后续数据库迁移、仓储实现和调度器合同变更应有可重复运行的自动化验证，不再依赖手工 PostgreSQL 路径验证或只覆盖内存仓储的 API 测试。

本阶段不改变产品功能主流程，优先把现有能力变成可测试、可维护的工程基线。

## 2. 范围

本轮包含：

- 建立 PostgreSQL 集成测试入口，覆盖真实数据库仓储的核心路径。
- 增加迁移从空库顺序执行验证，确认 `packages/db/drizzle/*.sql` 与迁移状态表可从零运行。
- 增加 scheduler CLI 契约测试，验证 API 调用 Python CLI 的输入输出和错误边界。
- 提取 API service，优先拆出已发布方案受众查询、通知预览、CSV 导出和审计过滤辅助函数。
- 为新增 service 和测试入口补齐 package 脚本与文档状态。

本轮不包含：

- 不做软约束入 CP-SAT，该内容留到第三版第三阶段。
- 不重构 PostgreSQL schema 的 JSONB 数组为关联表。
- 不引入 Redis、BullMQ、SSE、WebSocket 或 FastAPI scheduler。
- 不改 Web 视觉和交互结构，除非 API service 提取导致合同命名必须同步。
- 不做生产级认证、refresh token、用户管理或多租户权限。

## 7. 计划边界说明

本阶段优先处理“后续改动能否被自动验证”的问题。软约束入 CP-SAT 是第三版的重要目标，但它会同时影响算法建模、评分解释和测试数据设计，应在第二阶段测试基线稳定后单独执行。
