# 第四版第四阶段计划：CI 质量门禁与交付收尾

## 1. 阶段目标

把第四版已经稳定的本地验证入口固化为 GitHub Actions 自动质量门禁，使类型检查、跨语言测试、生产构建、真实 PostgreSQL、迁移和浏览器 E2E 能在干净环境中重复执行。

本阶段只建设工程验证与交付治理，不继续扩展排考业务能力。阶段完成后，第四版进入全量代码审查，不在本计划内提前处理审查尚未发现的问题。

## 2. 执行策略

### 2.1 触发策略

本项目以个人开发和直接维护 `main` 为主，采用以下分层策略：

- 快速门禁在所有 `push`、少量可能出现的 `pull_request` 以及手动触发时运行。
- 完整门禁只在 `main` 的 `push` 和手动触发时运行，不为通常不存在的 PR 重复消耗完整环境成本。
- 同一分支的新运行取消尚未完成的旧运行，避免连续推送形成无效排队。

### 2.2 门禁分层

| 门禁 | 主要内容 | 目标时长 | 阻断范围 |
| --- | --- | --- | --- |
| 快速门禁 | 变更区间检查、仓库卫生、类型检查、API 测试、scheduler 测试、生产构建 | 15 分钟内 | 所有受支持触发 |
| PostgreSQL 门禁 | PostgreSQL 16、空库迁移、迁移幂等与结构检查、迁移测试、API PostgreSQL 集成 | 15 分钟内 | `main` 与手动触发 |
| 完整演示门禁 | 从空卷构建 Compose、迁移、seed、smoke、API 重启持久化和 Playwright 主链 | 30 分钟内 | `main` 与手动触发 |

### 2.3 运行环境

- Node.js 使用项目 Docker 基线一致的主版本，依赖通过 `npm ci` 和根 `package-lock.json` 安装。
- scheduler 使用 Python 3.12 和 `uv`，依赖以 `apps/scheduler/uv.lock` 为准。
- 数据库门禁使用干净的 PostgreSQL 16 service，不复用开发机数据库。
- 完整演示门禁使用 GitHub Linux runner 自带的 Docker Compose，并为每次运行设置隔离的 Compose project name。
- GitHub Actions 第三方步骤在实施时核对官方当前稳定主版本；工作流权限默认收敛为 `contents: read`。

## 3. 范围与边界

### 3.1 本阶段包含

- GitHub Actions 工作流、触发条件、并发取消、超时和最小权限。
- 可在本地测试的仓库治理脚本，包括禁止跟踪产物、提交消息格式和变更区间空白错误检查。
- Node.js、Python/uv、PostgreSQL 16 和 Docker Compose 的干净环境验证。
- 失败时上传 Playwright 报告和测试结果，成功时不长期保留无用产物。
- README 中的 CI 使用说明，以及设计、状态、索引和计划历史的阶段收尾。

### 3.2 本阶段不包含

- Bearer token、进程内异步作业、HTTP 轮询、scheduler CLI 或 JSONB 兼容字段的业务改造。
- 独立 scheduler 服务、持久化队列、SSE/WebSocket、真实认证或新的 Web 功能。
- 自动发布、容器镜像推送、生产部署、云数据库或密钥管理平台接入。
- 自动修改 GitHub 仓库设置、分支保护、远端 secrets 或执行未经用户确认的 `push`。
- 将已知 CR-002 `npm audit` moderate 公告设为阻断项；该问题继续由 `docs/status/code_review_status.md` 跟踪。
- 第四版完成后的全量代码审查及其问题修复。

## 4. 任务清单

### 任务 1：建立可测试的 CI 治理合同

**目标：** 先把适合自动阻断的仓库规则变成稳定脚本，避免在工作流 YAML 中散落难以测试的 shell 片段。

**预期文件：**

- 新增 `scripts/ci/` 下的仓库卫生、提交区间或统一门禁脚本及其测试。
- 修改根 `package.json`，提供语义明确的本地入口，例如 `check:ci` 和对应脚本测试入口。

**实现要求：**

1. 先以临时 Git 仓库或受控 fixture 写红灯测试，覆盖禁止路径被跟踪、合法路径通过、合法/非法提交消息和多提交区间。
2. 禁止跟踪规则至少覆盖 `.codegraph/`、`node_modules/`、`.pytest_cache/`、`__pycache__/`、`.venv/`、`dist/`、`build/`、`coverage/`、`playwright-report/` 和 `test-results/`。
3. 提交消息沿用当前中文 Conventional Commits 形式，允许既有类型、可选作用域和非空主题；正确处理 GitHub 自动产生的合并提交，不扫描本次触发区间之外的历史提交。
4. 空白错误检查必须针对本次 push 或 PR 的实际提交区间；首次推送或缺少基准提交时采用可解释的回退，不能在干净 checkout 中直接运行无效的裸 `git diff --check`。
5. 脚本失败时输出具体文件或提交，不只返回退出码。

**最窄验证：**

```bash
npm run test:ci
npm run check:ci
git diff --check
```

**提交边界：** 纳入第四阶段统一提交。

### 任务 2：建立快速 GitHub Actions 门禁

**目标：** 在干净 runner 中复现不依赖数据库和浏览器的日常验证。

**预期文件：**

- 新增 `.github/workflows/ci.yml`。
- 必要时调整任务 1 的脚本或根 `package.json`。

**实现要求：**

1. 配置 `push`、`pull_request` 和 `workflow_dispatch`，设置最小权限、并发取消和 15 分钟超时。
2. checkout 必须保留计算提交区间所需的历史；治理脚本接收明确的 base/head，而不是猜测当前工作树。
3. 使用锁文件安装 Node.js 与 Python 依赖，并启用与锁文件绑定的缓存；缓存失效不能影响正确性。
4. 顺序执行治理检查、`npm run typecheck`、`LOG_LEVEL=silent npm test`、`npm run test:scheduler` 和 `npm run build`。
5. 任一步骤失败立即阻断；不得以 `continue-on-error` 隐藏真实失败。

**最窄验证：**

```bash
npm run test:ci
npm run check:ci
npm run typecheck
LOG_LEVEL=silent npm test
npm run test:scheduler
npm run build
git diff --check
```

工作流语法需通过本机可用的 YAML/Actions 静态检查；若本机没有对应工具，应记录限制，并以首次 GitHub Actions 运行作为最终托管环境证据。

**提交边界：** 纳入第四阶段统一提交。

### 任务 3：建立真实 PostgreSQL 与迁移门禁

**目标：** 在 PostgreSQL 16 干净实例上自动验证迁移、schema 约束和真实仓储行为。

**预期文件：**

- 修改 `.github/workflows/ci.yml`，新增独立 PostgreSQL 作业。
- 必要时新增仅负责数据库创建、隔离或清理的 CI 辅助脚本。

**实现要求：**

1. PostgreSQL service 必须包含健康检查，凭据只使用无生产价值的 CI 固定测试值。
2. 迁移检查从空库开始，并证明第二次执行不再应用迁移；复用现有 `db:check-migrations` 的双次执行、关键表、约束和关联表一致性检查。
3. 迁移测试与 API PostgreSQL 集成测试使用明确隔离或重置后的测试数据库，不能依赖作业执行顺序留下的偶然状态。
4. 执行 `npm run test:migrations`、`npm run db:check-migrations`、`npm run db:migrate` 和 `npm run test:postgres` 对应的真实库路径，并确保任何一步失败都会阻断。
5. 作业设置 15 分钟超时，日志不得打印真实凭据；本阶段不接入远端数据库 secrets。

**最窄验证：**

```bash
TEST_DATABASE_URL=postgres://examforge:examforge@localhost:5432/examforge_test npm run test:migrations
TEST_DATABASE_URL=postgres://examforge:examforge@localhost:5432/examforge_test npm run db:check-migrations
DATABASE_URL=postgres://examforge:examforge@localhost:5432/examforge_test npm run db:migrate
TEST_DATABASE_URL=postgres://examforge:examforge@localhost:5432/examforge_test npm run test:postgres
git diff --check
```

**提交边界：** 纳入第四阶段统一提交。

### 任务 4：建立完整 Compose 与 Playwright 门禁

**目标：** 在 `main` 和手动运行中，从空环境复现第四版第三阶段的完整演示验收。

**预期文件：**

- 修改 `.github/workflows/ci.yml`，新增完整演示作业。
- 必要时增强 `scripts/test-demo-e2e.sh` 的 CI 隔离、诊断或清理行为。

**实现要求：**

1. 安装与当前 Playwright 版本匹配的 Chromium 及系统依赖。
2. Compose project name 绑定 GitHub run id/attempt，避免重跑或并发运行共享容器、网络和卷。
3. 直接复用 `npm run test:e2e:demo`，覆盖空卷迁移、seed、真实 scheduler、smoke、Web/API、API 重启持久化和全部 Playwright 场景。
4. 无论成功、失败或取消，均执行兜底 `docker compose down --volumes --remove-orphans`；不得在 runner 留下影响后续步骤的环境。
5. 失败时上传 `playwright-report/` 和 `test-results/`，设置短保留期；缺失报告目录不能掩盖原始失败。
6. 作业设置 30 分钟超时，并只在 `main` push 或 `workflow_dispatch` 中运行。

**最窄验证：**

```bash
npm run test:e2e:demo
git diff --check
```

本地验证已经通过且之后未修改相关实现时，提交前不重复执行；首次 GitHub Actions 完整作业成功是托管 CI 的最终验收证据。

**提交边界：** 纳入第四阶段统一提交。

### 任务 5：托管验收、文档收尾与第四版交付

**目标：** 用真实 GitHub Actions 运行证明门禁有效，并完成第四版活动计划生命周期。

**预期文件：**

- 更新 `README.md`，说明 CI 分层、触发条件、手动运行和本地等价命令。
- 更新 `docs/design/第四版实现内容设计.md` 的已落地 CI 边界。
- 更新 `docs/status/project_status.md`、`docs/plan/history_plan.md` 和 `docs/index.md`。
- 删除完成后的 `docs/plan/第四版第四阶段计划.md`。

**实现要求：**

1. 完成本地全量验证后形成任务提交；不要因为提交动作重复运行提交前已经通过且之后未修改代码的验证。
2. 未经用户明确确认不执行 `push`。如果仍保持本地-only，应将“托管工作流尚未运行”如实记录为外部验收待办，不得声称 CI 已在 GitHub 通过。
3. 获得 push 授权后，确认快速、PostgreSQL 和完整演示三个作业在 GitHub 托管环境中通过；如失败，按系统化调试流程修复并重新验证。
4. 阶段完成后把日期、提交、工作流证据、验证结果和后续影响追加到 `history_plan.md`，删除活动计划并同步索引和状态。
5. 将“第四版完成后执行一次全量代码审查”设为唯一下一步；审查发现进入 `code_review_status.md`，修复另建计划。

**阶段验证：**


**提交边界：** 与任务 1 至任务 4 统一提交。

## 5. 验收标准

满足以下条件时，第四版第四阶段视为完成：

1. 快速门禁能在干净 runner 中稳定完成治理检查、类型检查、API 测试、scheduler 测试和生产构建。
2. PostgreSQL 门禁能从空库应用全部迁移，证明迁移幂等，并通过迁移与真实仓储集成测试。
3. 完整演示门禁能从空卷启动全栈环境，通过 smoke、API 重启持久化检查和全部 Playwright 场景，并在结束后清理环境。
4. CI 失败能定位到具体规则、测试或作业，并在浏览器失败时提供短期诊断产物。
5. 本地验证入口与 CI 使用同一批项目脚本，不维护两套行为不同的命令。
6. 真实 GitHub Actions 运行已经通过；如果用户选择不 push，则阶段只能记录为“本地实现完成、托管验收待完成”，不能提前归档计划。
7. 正式文档已同步，活动计划按仓库规则归入 `history_plan.md`，下一步明确为第四版完成后的全量代码审查。

## 6. 交付边界与实施约束

- 实施继续采用单线程；只有出现需要大量独立读取且明显侵占主上下文的工作时，才重新评估子代理。
- 功能或修复遵循最窄红灯、实现、回归的顺序；工作流 YAML 也必须由可本地执行的项目脚本支撑。
- 任务 1 至任务 5 连续完成并验证后统一提交一次；提交前已经验证且之后未改代码时，不重复执行相同验证。
- 不回滚用户已有修改，不提交缓存、依赖、构建产物、测试报告或 `.codegraph/`。
- 不以 CI 建设为由处理 CR-002、CR-003、CR-007、CR-008 或扩张第四版业务范围。
