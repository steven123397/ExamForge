# 第五版第六阶段计划：正式部署与交付收尾

> **面向 AI 代理的工作者：** 使用 `executing-plans` 逐任务实施本计划。默认在当前会话单线程推进；每项代码或配置行为先建立最窄失败证据，再实现并回归。任务 1 至 5 仅操作本地和 CI；任务 6 至 8 必须在用户明确授权目标主机、域名、镜像仓库、凭据方式和维护窗口后执行。

## 1. 阶段目标

把第五阶段已经通过真实 PostgreSQL/Redis 与浏览器回归的应用，交付为可追溯、可恢复、资源受控的正式部署。GitHub 托管 job 执行质量门禁，本地 WSL self-hosted Runner 生成不可变镜像和供应链证据，服务器只从广州 TCR 按 digest 拉取镜像、执行迁移并启动服务；nginx 统一承载 HTTPS，PostgreSQL/Redis/scheduler/Worker 不暴露公网端口，备份、恢复、日志、健康检查、150 场基准和镜像回滚均取得真实证据。

阶段完成后，ExamForge 应在腾讯云 4 核 4 GB Ubuntu 服务器上通过正式域名完成四角色核心闭环，并能回答“部署了哪个提交与镜像、数据如何备份恢复、故障如何发现、版本如何回滚、资源是否足够”。本阶段最后执行第五版完成后的全量代码审查和交付归档；在远程证据、CR-002 处置或回滚演练缺失时，不得宣称第五版整体完成。

## 4. 核心决策

### 4.1 发布物与供应链

- CI 对 API、Web、Worker 和 scheduler 生成以完整 Git SHA 标识的镜像；部署清单记录仓库、tag、digest、构建时间、源提交和 SBOM，服务器按 digest 拉取，不使用浮动 `latest`。
- GitHub 托管 job 只执行质量门禁；正式镜像构建、探针、SBOM、漏洞扫描和 TCR 推送由带 `examforge-release` 标签的本地 WSL Runner 执行。Runner 必须使用独立检出目录并显式使用本机 Docker Engine `default` Buildx builder，不读取开发仓库的未提交内容，也不在 job 启动时依赖远端 BuildKit bootstrap。
- Web 的请求路径本身已经包含 `/api/*`，因此 `NEXT_PUBLIC_API_BASE_URL` 在构建时固定为正式 HTTPS origin，不再附加 `/api`。不同环境必须生成不同 Web digest，不能在运行时假装修改已写入客户端的公开变量。
- 发布 workflow 与部署 workflow 分离。发布只生成镜像和清单；远程部署使用手动批准的版本，不由普通 push 自动修改服务器。
- 广州 TCR 是正式制品单一来源，不因使用本地 Runner 而取消。Runner 不持有北京服务器 SSH 私钥或生产 secrets；生产服务器不访问 GitHub、不从源码构建。
- 所有 GitHub Action 固定到完整提交 SHA；镜像仓库 token 只授予推送目标仓库所需的最小权限。
- Node.js 构建链固定为 Node.js 22.22.2 与 npm 12.0.1；需要安装脚本的依赖采用精确版本 allowlist，并由 CI 拒绝未批准的新增脚本。PostCSS 临时覆盖固定为 8.5.19，测试约束其唯一父依赖链为 Next；Next 上游解除精确旧依赖后应优先移除该覆盖。

### 4.2 生产网络与身份

- `compose.production.yml` 为独立生产定义，不继承演示文件中的宿主机数据库/Redis/scheduler/Worker 端口和本地 build。Web/API 只绑定 `127.0.0.1`，由宿主 nginx 反代；其他服务仅加入内部 Docker 网络。
- API 固定 `NODE_ENV=production`、`EXAMFORGE_SESSION_COOKIE_SECURE=true` 和唯一可信 HTTPS Origin；Cookie 继续使用 HttpOnly、SameSite=Lax、Secure，CORS 不使用 `*`。nginx 将 `/api/*` 直接转发 API，并用 `/_health/api`、`/_health/api-ready` 分别代理 API 的 `/health`、`/ready`，其余路径进入 Web。
- 四角色密码、PostgreSQL 密码、镜像仓库凭据和外部备份凭据只进入服务器权限为 600 的环境或 Docker secret 文件。`.env.production.example` 只保存变量名、格式和非敏感默认值。
- `EXAMFORGE_*_PASSWORD` 只用于首次 bootstrap，修改环境文件或重启 API 不会静默改密。已有账户的轮换只能在获授权维护窗口内通过 API 容器的维护 CLI 执行：新密码经标准输入传入，目标用户名重复确认，事务内更新 scrypt 散列和凭据版本、吊销全部旧会话并写入脱敏审计；成功后由受控 secrets 流程把同一新值同步为未来空库 bootstrap 输入，不能把该同步误作当前库改密；CLI 的 `--actor` 只作审计标签，不能替代服务器访问控制。
- 迁移是独立一次性容器，必须成功后才启动应用；生产启动不自动 seed 或重置业务数据。首次演示数据初始化使用明确的单次 bootstrap 命令并记录 actor 与时间。

### 4.3 数据、日志与恢复

- PostgreSQL 在线数据位于额外挂载盘的 ExamForge 独立目录；Redis AOF 可重建分发状态，但 PostgreSQL 仍是业务事实源。
- `pg_dump --format=custom` 产物同时生成 SHA-256、迁移版本和脱敏元数据；本机短期保留后复制到 COS 或用户授权的异机存储。只有完成可丢弃数据库恢复、迁移检查和核心查询后才称为有效备份。
- 容器使用 Docker 日志大小与文件数上限；nginx 使用独立 access/error log 和 logrotate。日志不得记录密码、Cookie、完整排考输入、敏感审计 payload 或数据库连接串。
- 健康检查区分 liveness 与 readiness；外部监测检查 HTTPS Web、API readiness 和证书有效期，内部检查 Publisher、Worker、scheduler、PostgreSQL 和 Redis。

### 4.4 切流与回滚

- 先在仅本机可见的生产端口启动新版本，运行迁移、smoke 和核心 E2E，再修改独立 nginx site 并执行 `nginx -t`；不直接覆盖其他站点配置。
- 应用回滚使用上一份经过验证的 digest 清单。若数据库迁移向后兼容，回滚应用不回退 schema；若迁移不兼容，发布前必须提供并验证数据恢复方案，否则禁止切流。
- 回滚验收不仅检查容器运行，还要检查四角色登录、本人作用域、任务提交/SSE、运行/草稿查询和审计追溯。
