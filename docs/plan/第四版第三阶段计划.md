# 第四版第三阶段：演示环境与体验增强实现计划

## 1. 计划定位

本阶段承接第四版第二阶段已经完成的教师二阶段 CP-SAT、`reschedule_context` 增量重排合同和规模验证，目标不是继续扩张算法模型，而是让这些能力能够通过稳定、可复现、可浏览器验收的产品路径被真实演示。

本计划按单线程顺序实施。每个任务先补最窄失败证据，再实现，再运行该任务对应的最窄验证；前一任务已经验证且之后未再修改相关代码时，不在提交或收尾时机械重复同一验证。

计划文档本次只作为活动计划落地，不单独提交。后续实施过程中的 Git 提交按用户届时指令执行，不因计划存在而自动提交或推送。

## 3. 阶段目标

本阶段完成后，应达到以下可验证状态：

1. 教务员可以在草稿工作台锁定不允许变化的考试，并从当前草稿生成一个新的增量重排运行版本。
2. API 在 `service` 层根据草稿事实生成 `reschedule_context`，Web 不自行拼装算法合同。
3. 新运行保持锁定考试的考场、时段和教师集合不变，并向 Web 展示 `frozen`、`retained`、`changed` 摘要。
4. 从干净 Docker `volume` 出发，可以通过一条明确命令完成 PostgreSQL 启动、空库迁移、演示数据 `seed`、API/Web 启动和 scheduler CLI 可用性验证。
5. 演示 `smoke` 明确证明 API 使用 PostgreSQL，而不是静默回退内存仓储，并真实完成一次排考。
6. Playwright 同时具备快速本地模式和完整 Compose/PostgreSQL 模式，重新取得核心工作流浏览器证据。
7. 本阶段不引入 CI；阶段形成的稳定命令留给第四阶段接入自动质量门禁。

## 4. 关键设计决策

### 4.1 增量重排由 API 生成合同

新增面向草稿的应用入口，建议采用：

```text
POST /api/schedule-drafts/:id/reschedule
```

其职责为：

1. 读取当前草稿和锁定考试集合。
2. 将草稿全部安排转换为 `baseline_assignments`。
3. 将未锁定考试转换为 `movable_exam_task_ids`。
4. 调用现有 `ScheduleRunService` 和 `scheduler client` 生成新运行。
5. 返回新运行以及类型明确的重排摘要。

Web 只表达用户意图和展示结果，不直接计算冻结范围，也不复制 scheduler 校验规则。

### 4.2 原草稿保持不变

增量重排生成新的 `schedule_run`，不原地覆盖当前草稿。这样可以保留人工调整证据、锁定事实和审计链，并继续复用既有运行对比、创建新草稿和发布流程。

以下语义必须由测试固定：

- `lockedExamTaskIds` 中的考试全部冻结。
- 其余考试全部进入 movable 集合。
- 当前草稿为 `published` 或 `discarded` 时禁止发起重排。
- scheduler 返回不可行结果时仍按现有运行模型持久化和展示，不伪装为成功方案。
- 发起重排不会改变原草稿状态、安排、锁定集合和变更记录。

### 4.3 本阶段不选择高成本企业化能力

第四版设计允许按需要选择真实认证、SSE/WebSocket 或 scheduler 服务化中的一项。本阶段明确不选择这三项：

- 真实认证需要用户、会话、密码策略和权限矩阵成组设计。
- SSE/WebSocket 在当前进程内作业和粗粒度进度模型上收益有限。
- scheduler 服务化会引入新的部署、超时、健康检查和服务间合同边界。

本阶段的企业化价值集中在可复现部署、依赖就绪检查、真实 PostgreSQL 证据和完整业务工作流，不扩大尚未准备好的运行时语义。

### 4.4 Compose 演示必须覆盖完整依赖链

Compose 不能只负责数据库。目标拓扑为：

```text
postgres healthy
  -> migrate completed
  -> seed completed
  -> api ready, with PostgreSQL and scheduler CLI
  -> web ready
  -> demo smoke / Playwright
```

API 容器必须包含可执行的 Python 3.12、`uv` 和 scheduler 依赖。演示环境显式设置 `DATABASE_URL`，任何数据库连接、迁移、`seed` 或 scheduler 启动错误都必须使服务或 `smoke` 失败，不允许回落到内存结果后继续报告成功。

### 4.5 Playwright 区分本地快速模式和完整演示模式

- `npm run test:e2e`：由 Playwright 启动本地 API/Web，用于快速浏览器回归。
- `npm run test:e2e:demo`：针对完整 Compose 栈运行，使用真实 PostgreSQL 和容器内 scheduler。
- 是否复用外部服务必须由显式环境变量控制，不再仅根据 `CI` 自动推断。
- 自动化模式使用隔离、可重置的数据环境；不得静默连接开发者正在使用的持久化数据库。

## 7. 任务清单

### 任务 1：建立草稿增量重排应用合同

#### 目标

让 API 能根据草稿和锁定事实生成重排请求并持久化新运行，形成 Web 可直接调用的稳定业务入口。

#### 红灯与测试

先在 `apps/api/tests/schedule-run-service.test.ts` 和 `apps/api/tests/app.test.ts` 补失败测试，至少覆盖：

1. 草稿全部安排进入 `baseline_assignments`。
2. 锁定考试不进入 movable 集合，未锁定考试全部进入 movable 集合。
3. 新运行中的锁定考试保持 `room`、`slot` 和 `teacher` 集合不变。
4. 原草稿内容、锁定集合、状态和 change events 不被修改。
5. 草稿不存在返回 `404`。
6. `published` 或 `discarded` 草稿返回 `409`。
7. `viewer` 返回 `403`，`admin` 和 `operator` 可以执行。
8. 返回值包含可由 shared 类型解析的 `frozen`、`retained`、`changed` 摘要。

如扩展真实仓储行为，再在 PostgreSQL 集成测试中补一条最窄场景，证明读取的是数据库中的草稿安排和锁定集合，而不是内存替代数据。

红灯命令示例：

```bash
npm run build --workspace @examforge/shared
cd apps/api && node --test --import tsx tests/schedule-run-service.test.ts tests/app.test.ts
```

#### 实现要求

- 在 shared 中补充请求/响应或重排摘要的稳定类型，避免 Web 对 `Record<string, unknown>` 做无保护猜测。
- 复用 `ScheduleRunService`、`PlatformRepository.getScheduleDraft()` 和现有 `scheduler client`。
- `route` 只处理鉴权、参数、状态码和错误映射；草稿到 `reschedule_context` 的转换位于 `service/use-case` 层。
- 对 `baseline` 和 `movable` ID 使用稳定顺序，保证测试、报告和演示输出可复现。
- 不把锁定草稿转换成 `fixed_assignments` 与 `reschedule_context` 双重输入，避免形成两套重叠事实。

#### 最窄验证

```bash
npm run typecheck --workspace @examforge/api
cd apps/api && node --test --import tsx tests/schedule-run-service.test.ts tests/app.test.ts
```

若修改 PostgreSQL 路径，再追加：

```bash
TEST_DATABASE_URL=postgres://examforge:examforge@localhost:5432/examforge_test npm run test:postgres
```

#### 目标

2. 锁定一场考试。

5. 断言锁定考试 `room`、`slot` 和 `teacher` 集合未变化。
6. 断言页面展示 `frozen`、`retained`、`changed` 数量或列表摘要。

### 任务 3：建立完整 Compose 演示栈和 `readiness` 合同

#### 目标

从干净 Docker 环境启动完整平台，并让自动化工具能够区分“进程已启动”和“数据库、迁移、`seed`、API、Web 已可用”。

#### 可执行失败证据

在实现容器前先确定 `smoke` 的机器可判定验收项，并让它针对现有仅 PostgreSQL 的 Compose 配置失败。验收项至少包括：

- Web 首页返回 `2xx`。
- API `liveness` 返回 `2xx`。
- API `readiness` 明确报告 `storage=postgres`。
- 基础数据接口返回 `seed` 后的批次、考试、考场、教师和时段。
- 使用 operator token 发起真实排考，结果为 `feasible`、安排数大于 0、硬冲突为 0。
- API 重启后仍能读取刚创建的运行，证明数据确实持久化在 PostgreSQL。

#### 实现要求

- 扩展 `docker-compose.yml`，加入 `migrate`、`seed`、API 和 Web 服务及健康依赖。
- 为 API 和 Web 提供多阶段 Dockerfile；最终镜像只保留运行必需内容。
- API 镜像必须能够执行现有 Python scheduler CLI，不改变 scheduler 的服务形态。
- 新增 `.dockerignore`，排除 `.git/`、`.codegraph/`、`node_modules/`、`.next/`、`dist/`、coverage、缓存和本地数据库产物。
- `/health` 保持进程 `liveness`；新增 `/ready` 或等价入口验证仓储依赖，并返回不含凭据的存储模式。
- PostgreSQL `readiness` 必须执行真实查询；存在 `DATABASE_URL` 时连接失败必须返回未就绪，不得切换内存仓储。
- `migration` 和 `seed` 使用一次性容器，失败时阻止 API 启动。
- `seed` 必须可重复执行，或由 demo `reset` 明确保证空库；不得靠人工忽略重复数据错误。
- 端口和镜像参数允许通过环境变量覆盖，默认仍保持 Web `3000`、API `4000`、PostgreSQL `5432`。

#### 演示命令

根 `package.json` 提供以下稳定入口或等价命令：

```bash
npm run demo:up
npm run demo:smoke
npm run demo:down
npm run demo:reset
```

其中 `demo:reset` 是显式破坏性命令，只删除 ExamForge 自己命名的 demo `volume`；脚本应打印目标 Compose project 和 `volume`，不得模糊删除其他 Docker 资源。

#### 最窄验证

```bash
docker compose config
npm run demo:reset
npm run demo:smoke
docker compose ps
```

验证结束后按需要运行：

```bash
npm run demo:down
```
