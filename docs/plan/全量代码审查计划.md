# 全量代码审查计划

## 1. 目标

在第三版第四阶段完成后，对 ExamForge 当前 `main` 进行一轮全量代码审查，先用本地代码阅读、CodeGraph、测试和文档证据清理确定性问题，再把后续高阶规划与架构取舍留给 `claude-fable-5` 做更聚焦的分析。

本轮审查不以继续开发新功能为目标，也不直接修复业务代码。目标是建立第三版后的质量基线，识别低级错误、回归风险、测试缺口、文档不一致和生产化前必须处理的问题，并将确认后的问题统一记录到 `docs/status/code_review_status.md`。

## 4. 问题分级

审查发现必须按以下等级记录到 `docs/status/code_review_status.md`：

- `P0 阻塞`：会导致数据损坏、发布错误、安全绕过、核心功能崩溃或验证无法可信。
- `P1 高优先级`：影响关键业务正确性、长期架构边界或后续企业化开发。
- `P2 中优先级`：影响可维护性、测试覆盖、用户体验或生产化准备。
- `P3 低优先级`：命名、文档细节、轻微重复或可延后优化。

每个问题至少包含：

- 编号
- 状态：`待解决`、`已解决` 或 `暂缓`
- 严重级别
- 所属模块
- 发现来源
- 具体文件和行号
- 问题描述
- 影响
- 建议处理
- 验证方式
- 解决记录（解决后填写提交和验证证据）

## 5. 审查步骤

1. 运行 `git status --short --branch`，确认当前分支和工作区状态。
2. 阅读第 2 节权威输入，明确第三版第四阶段后的真实实现边界和已知存留问题。
3. 使用 CodeGraph 梳理核心调用链：
   - 排考运行创建链路。
   - Python scheduler CLI 调用链路。
   - 软约束目标和评分输出链路。
   - 固定安排合同从 shared 到 scheduler 的链路。
   - 草稿创建、调整、校验、发布、废弃链路。
   - PostgreSQL JSONB 字段与关联表双写链路。
   - 已发布查询、通知和导出链路。
   - Web API client、TanStack Query 和业务面组件数据流。
4. 按第 3 节的 5 个审查面逐项审查。
5. 对每个可疑点优先用代码证据、测试证据或运行命令确认，不记录无法定位的泛泛建议。
6. 将确认后的新问题写入 `docs/status/code_review_status.md` 的“待解决问题”部分。
7. 如果发现已有存留问题已被第三版代码证明解决，将其状态改为“已解决”，并在审查记录中补充证据。
8. 审查结束后给出摘要：问题数量、P0/P1/P2/P3 分布、建议修复顺序、哪些问题应先本地修复，以及哪些高阶议题适合之后交给 `claude-fable-5` 分析。

## 6. 建议验证命令

审查过程中按需运行：

```bash
npm run test:scheduler
LOG_LEVEL=silent npm test
npm run typecheck
npm run build
npm run test:e2e
TEST_DATABASE_URL=postgres://examforge:examforge@localhost:5432/examforge_test npm run test:postgres
TEST_DATABASE_URL=postgres://examforge:examforge@localhost:5432/examforge_test npm run test:migrations
DATABASE_URL=postgres://examforge:examforge@localhost:5432/examforge_test npm run db:migrate
npm audit --audit-level=moderate
git diff --check
```

如果只做只读审查，不要求全部命令都运行；但任何“测试通过”“构建通过”“迁移通过”结论必须来自当前会话的新鲜输出。若 PostgreSQL 测试库或 Docker 环境不可用，应记录具体阻塞，不用内存仓储结果替代真实库结论。

## 7. 交付物

- 更新后的 `docs/status/code_review_status.md`。
- 中文审查汇报：问题按严重级别排序，先列问题，再列开放问题和建议。
- `claude-fable-5` 后续分析输入建议：本轮审查摘要、剩余高阶问题、第四版候选方向和不值得再投入的方向。
- 如需修复问题，另起修复计划，不在本轮审查计划中直接改业务代码。

## 8. 与 `claude-fable-5` 的衔接

本轮审查应先清理本地可确认的问题，降低后续大模型分析噪声。`claude-fable-5` 不用于第一轮低级错误排查，后续更适合聚焦：

- 第四版是否应该进入真实认证、队列、OpenAPI/SDK、部署运维或楼栋主数据。
- 当前架构中哪些兼容方案需要收敛，例如 JSONB 与关联表双写、CLI 调度器、演示 token。
- 哪些能力对课程交付足够，哪些属于企业化加分项。
- 第四版阶段边界、优先级和不做清单。

## 9. 完成标准

- 所有审查面都已覆盖。
- P0/P1 问题均有明确文件位置、影响说明和建议处理方式。
- 已知存留问题已复核，状态和证据写入 `docs/status/code_review_status.md`。
- `docs/status/project_status.md` 继续只描述开发进度和已实现内容，不维护问题清单。
- 已形成一份适合后续 `claude-fable-5` 高阶分析的精简输入摘要。
