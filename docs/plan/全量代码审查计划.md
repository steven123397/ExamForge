# 全量代码审查计划

## 1. 目标

在第二版增强方向第四阶段完成、CodeGraph 已启用之后，对 ExamForge 当前 `main` 进行一轮全量代码审查，形成可执行的问题清单和后续修复优先级。

本轮审查不以继续开发新功能为目标，而是建立第三版企业化之前的质量基线，重点识别架构边界、算法正确性、API 合同、数据一致性、前端交互、测试覆盖、文档治理和生产化风险。

## 4. 问题分级

审查发现必须按以下等级记录到 `docs/status/code_review_status.md`：

- `P0 阻塞`：会导致数据损坏、发布错误、安全绕过、核心功能崩溃或验证无法可信。
- `P1 高优先级`：影响关键业务正确性、长期架构边界或第三版继续开发。
- `P2 中优先级`：影响可维护性、测试覆盖、用户体验或生产化准备。
- `P3 低优先级`：命名、文档细节、轻微重复或可延后优化。

每个问题至少包含：

- 编号
- 状态：`待解决`、`已解决` 或 `暂缓`
- 严重级别
- 所属模块
- 发现来源
- 具体文件和行号
- 问题描述
- 影响
- 建议处理
- 验证方式
- 解决记录（解决后填写提交和验证证据）

## 5. 审查步骤

1. 核对仓库状态，确认当前分支、提交和 `.codegraph/` 可用性。
2. 阅读权威输入文档，明确当前实现边界和已知存留问题。
3. 使用 CodeGraph 梳理核心调用链：
   - 排考运行创建链路。
   - 草稿创建、调整、校验、发布、废弃链路。
   - 已发布查询、通知和导出链路。
   - Python 调度器 CLI 调用链路。
4. 按第 3 节的 5 个审查面逐项审查。
5. 对每个可疑点优先用代码证据、测试证据或运行命令确认，不记录无法定位的泛泛建议。
6. 将确认后的问题写入 `docs/status/code_review_status.md` 的“待解决问题”部分。
7. 如果发现已有存留问题已被本轮代码证明解决，将其状态改为“已解决”，并补充证据。
8. 审查结束后给出摘要：问题数量、P0/P1/P2/P3 分布、建议修复顺序和是否适合进入 Claude Fable 5 深度分析。

## 6. 建议验证命令

审查过程中按需运行：

```bash
npm test
npm run typecheck
npm run build
npm run test:e2e
cd apps/scheduler && uv run --python 3.12 --extra dev python -m pytest -q
git diff --check
```

如果只做只读审查，不要求全部命令都运行；但任何“测试通过”“构建通过”结论必须来自当前会话的新鲜输出。

## 7. 交付物

- 更新后的 `docs/status/code_review_status.md`。
- 中文审查汇报：问题按严重级别排序，先列问题，再列开放问题和建议。
- 如需修复问题，另起修复计划，不在本轮审查计划中直接改业务代码。

## 8. 完成标准

- 所有审查面都已覆盖。
- P0/P1 问题均有明确文件位置、影响说明和建议处理方式。
- 已知存留问题已迁入 `docs/status/code_review_status.md`。
- 原 `docs/status/project_status.md` 继续只描述开发进度和已实现内容，不再维护问题清单。
