# 第四版第二阶段：算法升级与规模验证实现计划

> **面向 AI 代理的工作者：** 必需子技能：使用 `executing-plans` 按任务顺序执行本计划；只有用户明确授权子代理时才使用 `subagent-driven-development`。所有行为变更使用 TDD，步骤通过复选框跟踪。

**目标：** 将监考教师分配升级为可证明的二阶段 CP-SAT 优化，建立具有基准方案和冻结范围的增量重排合同，并完成 50、100、150 场考试的可复现规模验证。

**架构：** 第一阶段 CP-SAT 继续决定 room-slot；第二阶段 CP-SAT 在 room-slot 固定后决定教师分配，统一处理固定教师、不可用时间、同时间冲突、工作量平衡、连续监考和稳定性目标。增量重排通过 `reschedule_context` 描述完整基准方案和可移动考试集合，未列为可移动的考试自动冻结；结果中的 `report.reschedule` 记录保留和变更明细。

**技术栈：** Python 3.12、OR-Tools CP-SAT、pytest、TypeScript、Zod、Fastify、Node Test Runner、JSON CLI。

## 2. 关键设计决策

### 2.1 教师分配

新增独立 `teacher_assignment.py`。对 room-slot 已确定的每场考试和每位教师建立布尔变量 `y[exam_task_id, teacher_id]`，约束如下：

- 每场考试恰好选择 `invigilator_count` 位教师。
- 教师不可用时对应变量固定为 `0`。
- 同一教师在同一时间段最多监考 1 场。
- `fixed_assignments.teacher_ids` 必须完整保留。
- 增量重排中被冻结考试的基准教师必须完整保留。

目标函数按业务权重最小化最大工作量、连续监考次数和相对基准教师变化数。教师 ID 只作为业务目标完全相同时的确定性 tie-break，不得改变业务最优解。

### 2.2 增量重排

在 Python 和 shared 合同中新增：

```python
@dataclass(frozen=True)
class RescheduleContext:
    baseline_assignments: tuple[ScheduledExam, ...]
    movable_exam_task_ids: tuple[str, ...]
```

```typescript
export const rescheduleContextSchema = z.object({
  baseline_assignments: z.array(scheduledExamSchema).min(1),
  movable_exam_task_ids: z.array(z.string()).default([]),
});
```

`scheduledExamSchema` 应移动到 `scheduleInputSchema` 之前定义；`rescheduleContextSchema` 使用 `superRefine()` 拒绝重复基准考试、重复 movable ID 和不属于基准的 movable ID。与当前 `exam_tasks` 的完整覆盖关系由 Python 合并输入校验负责。

语义固定为：

- `baseline_assignments` 必须恰好覆盖当前 `exam_tasks`，且考试 ID 唯一。
- `movable_exam_task_ids` 必须是基准考试 ID 的无重复子集。
- 未列入 `movable_exam_task_ids` 的考试冻结 room、slot 和 teacher。
- 可移动考试允许改变安排，但 `schedule_stability` 权重会惩罚 room-slot 或教师变化。
- `fixed_assignments` 仍是显式强制安排；对冻结考试给出不同固定值时输入校验失败。
- 未提供 `reschedule_context` 时行为与当前版本完全兼容。

### 2.3 规模验证

新增确定性、按可行解构造的数据生成器，不直接复用可能随机产生不可行输入的 `generate_medium_dataset()`。规模验证输出至少包含：考试数、候选变量数、求解状态、总耗时、得分、最大教师负载、教师负载差和冲突数。
