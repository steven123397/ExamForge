# 计划文档规则与模板

本目录保存可执行计划。计划描述阶段目标、垂直任务切片、依赖、验证证据和交付边界；长期设计事实写入 `docs/design/`，当前进展写入 `docs/status/`。

ExamForge 默认使用一个「第 N 版第 N 阶段计划」作为阶段执行的单一事实来源。任务不是额外的 Ticket 文档，而是阶段计划中可独立认领、验证和交付的工作单元。

## 计划级别

- 单点窄变更可以只写简短 checkpoint，不强制创建长计划。
- 多任务、阶段验收或持续交接使用正式阶段计划。
- 多 Agent 跨 worktree、多人/跨团队协作或外部跟踪器需要独立认领时，才从阶段计划导出 Ticket 或 Issue。

导出产物必须回写阶段计划，不能成为第二套事实来源。

## 生命周期

活动计划直接放在本目录下。完成后：

1. 将完成摘要追加到 `history_plan.md`。
2. 摘要记录日期、原计划、提交或证据、完成内容、验证结果和后续影响。
3. 删除对应活动计划文件。
4. 更新 `docs/status/project_status.md`。
5. 如正式文档清单变化，同步更新 `docs/index.md`。

不使用单独的 `archive/` 目录。

## 任务切片规则

- 优先端到端垂直切片，不按数据库、API、前端机械横切。
- 每项任务适合一个新 Agent 会话独立完成。
- 每项任务完成后可独立测试、演示或验证合同增量。
- 显式记录依赖、并行关系、文件边界和汇合点。
- 小步骤留在任务内部，不把测试、实现、运行命令和 commit 拆成独立任务。
- 尚未明确的问题保留在未规划区，不虚构实现步骤。
- 不默认 commit、创建 worktree、派子 Agent 或发布外部 Issue。
