# 第五版第二阶段计划：Scheduler 服务化

## 1. 阶段目标

将当前由 API 进程通过 stdin/stdout 调用的 Python scheduler 升级为可独立部署、独立探活和独立扩缩的 FastAPI 计算服务，并以 OpenAPI 和双端合同测试固定跨语言边界。生产 Compose 调用切换为 HTTP 后，CLI 继续作为本地调试、直接基准和故障定位入口，二者必须复用同一输入解析、求解和输出序列化路径。

本阶段只改变 scheduler 的部署与调用边界，不引入 Redis、BullMQ、Outbox Publisher、独立任务 Worker、SSE/WebSocket 或完整任务中心。API 现有作业 application service、持久化状态机和进程内执行器继续负责作业生命周期；不得把一次 HTTP 求解调用误报为可靠队列已经完成。

## 2. 前置条件与启动门槛

1. 第五版第一阶段已经完成并归档；作业状态、事件、幂等、真实会话、RBAC 和关系化数据单一事实源合同保持稳定。
2. `docs/status/code_review_status.md` 中 CR-008 仍为待本阶段重评的既有问题，不创建重复编号。
3. 本地 Node.js、Python 3.12/uv、Docker、PostgreSQL 16 和 Playwright 环境可用。
4. 开始修改前运行 `git status --short --branch`，保留用户已有改动；理解代码时先使用 CodeGraph。

## 3. 范围与边界

### 3.1 本阶段包含

- FastAPI scheduler 的 `/health`、`/ready` 和 `/solve` 窄接口。
- CLI 与 HTTP 共用的解析、求解、报告和序列化 application pipeline。
- OpenAPI 固定产物、生成一致性检查和 TypeScript/Python 合同样例。
- API 到 scheduler 的 HTTP 客户端、超时、取消、故障分类和安全错误映射。
- scheduler 独立镜像、Compose 服务、健康依赖和资源边界。
- CLI 与 HTTP 结果等价、服务故障、Compose 主链和性能开销验证。

### 3.2 本阶段不包含

- Redis、BullMQ、Outbox Publisher、独立 Worker、多实例任务消费和崩溃恢复。
- SSE/WebSocket、`Last-Event-ID`、事件补发和前端实时进度。
- 作业中心完整页面、前端路由重构或视觉体系扩张。
- 约束策略版本化、归一化评分、方案实验室或新的求解算法。
- scheduler 直接访问 ExamForge PostgreSQL、管理用户、会话、作业或审计数据。
- 自动发布、镜像仓库、腾讯云生产部署、TLS 或公开域名。

## 4. 核心合同决策

### 4.1 网络边界

- `POST /solve` 接收与现有 CLI JSON 输入等价的完整 `ScheduleInput`，同步返回现有 `ScheduleResult`；`infeasible` 是 HTTP 200 的成功业务结果。
- `/health` 只证明进程存活；`/ready` 证明模型、应用 pipeline 和必要运行时已可接受请求，不访问业务数据库。
- scheduler 不信任来自浏览器的请求，不暴露 CORS 通配，不接收用户 Cookie，也不承担 RBAC。
- API 是 scheduler 的唯一生产调用者；请求携带 trace/request ID，响应和日志不得泄露输入全文、内部堆栈或环境变量。

### 4.2 故障分类

- JSON/schema 或语义输入错误属于稳定的 `validation` 合同错误，API 映射为客户端可理解的 4xx，不进入 `succeeded`。
- 求解得到不可行结果仍返回 `ScheduleResult(status=infeasible)`，作业按第一阶段合同进入 `succeeded`。
- API 主动取消、调用截止时间耗尽、scheduler 不可用、协议响应损坏和内部异常必须可区分；最终作业状态继续由既有 application service 决定。
- HTTP 客户端不进行静默 CLI 回退。调试时可显式选择 CLI transport，生产 Compose 必须显式选择 HTTP transport。

### 4.3 OpenAPI 与兼容性

- FastAPI 生成的 OpenAPI JSON 是网络合同的可审查产物；生成命令必须确定性输出，CI 检查工作树中的产物未漂移。
- Python HTTP 模型和现有领域模型之间通过显式适配器转换，不在路由中复制求解逻辑。
- TypeScript 客户端必须运行时校验成功与错误响应；仅有编译期类型不构成跨进程合同证据。
- CLI 与 HTTP 对同一固定输入的规范化输出必须等价，时间、随机 ID 等非确定字段若存在应由明确规则排除。

## 5. 任务清单

### 任务 1：冻结 HTTP 协议与故障合同

**目标：** 在实现服务器前，用测试和正式文档固定端点、模型、错误 envelope、trace 和状态语义。

**预期文件：**

- `apps/scheduler/examforge_scheduler/` 下 HTTP 模型与错误合同模块
- `apps/scheduler/tests/` 下协议测试
- `apps/api/src/scheduler-client.ts` 或拆分后的客户端合同模块
- `packages/shared/src/` 中确有跨应用复用价值的最小合同
- `docs/design/第五版实现内容设计.md`

**TDD 步骤：**

1. 先写失败测试，覆盖合法输入、schema 错误、跨资源语义错误、`infeasible` 成功结果、超时、取消、服务不可用、损坏响应和内部错误脱敏。
2. 固定 `ScheduleInput`、`ScheduleResult` 和错误 envelope 的字段、HTTP 状态、稳定错误码与 request ID 传播规则。
3. 明确 API 作业状态映射，证明业务不可行不会被误记为基础设施失败，合同错误不会被误记为成功。
4. 不把 uvicorn、FastAPI exception 类型或未来 BullMQ 字段放入 shared 业务合同。

**最窄验证：**

```bash
npm run typecheck
LOG_LEVEL=silent npm test --workspace @examforge/api
cd apps/scheduler && uv run --python 3.12 --extra dev python -m pytest tests -q
git diff --check
```

### 任务 2：抽取 CLI 与 HTTP 共用 application pipeline

**目标：** 消除 transport 层的解析与求解分叉，使 CLI 和未来 FastAPI 调用完全复用一条业务路径。

**预期文件：**

- `apps/scheduler/examforge_scheduler/cli.py`
- 新增 `apps/scheduler/examforge_scheduler/transport.py` 或等价 application 模块
- `apps/scheduler/tests/test_cli.py`
- 新增 pipeline 与等价性测试

**TDD 步骤：**

1. 先用固定可行、不可行、固定安排和增量重排样例记录 CLI 规范化结果。
2. 把 JSON 到领域模型、预检/求解/报告编排和领域结果到 JSON 的逻辑移入无框架依赖的 application pipeline。
3. CLI 只负责 stdin/stdout、退出码和诊断 stderr；stdout 继续只输出机器可解析 JSON。
4. 加入重复调用和异常路径测试，证明共享 pipeline 不保存跨请求可变状态。

**最窄验证：**

```bash
npm run test:scheduler
npm run benchmark:scheduler
git diff --check
```

### 任务 4：实现 API HTTP scheduler 客户端

**目标：** 让 API 在不改变上层作业 use-case 的前提下通过 HTTP 调用 scheduler，并对网络失败作稳定分类。

**预期文件：**

- `apps/api/src/scheduler-client.ts` 及必要的拆分模块

- `apps/api/src/config.ts` 或既有集中配置入口

**TDD 步骤：**

1. 用可控测试服务器或注入式 fetch 先写红灯，覆盖 200 可行/不可行、4xx 合同错误、5xx、断连、超时、调用取消、非 JSON 和 schema 漂移。

**目标：** 在演示环境中把 scheduler 作为独立健康服务运行，并让 API 生产路径显式使用 HTTP。

**预期文件：**

- 新增 `apps/scheduler/Dockerfile`

- `apps/api/Dockerfile`
- `scripts/demo-up.sh`、`scripts/demo-reset.sh`、`scripts/demo-smoke.mjs`
- `.env.example`、`README.md`
- 必要的 Compose/E2E 测试辅助代码

**TDD 步骤：**

3. API 容器设置 `SCHEDULER_TRANSPORT=http` 和内部 URL；确认生产镜像不再依赖在请求期间启动 Python 子进程。
4. smoke 直接验证 scheduler 健康/就绪、可行与不可行样例，再通过 API 完成排考、持久化读取和 API 重启检查。
5. 在隔离 Compose project、端口和卷上执行，结束后清理测试资源，不影响用户已有栈。

## 6. 阶段验收标准

1. FastAPI scheduler 可独立启动，`/health`、`/ready`、`/solve` 和错误 envelope 有自动化合同测试。
2. CLI 与 HTTP 共用同一 application pipeline，对固定样例给出等价的规范化结果。
3. OpenAPI 产物可确定性生成，CI 能检测 Python 网络模型与受控合同漂移。
4. API HTTP 客户端对成功、业务不可行、合同错误、超时、取消、不可用、损坏响应和内部错误进行稳定分类，无静默 CLI 回退。
5. Compose 中 scheduler 是独立健康服务，API 生产调用使用 HTTP，业务数据库和身份边界仍只属于 API。
6. 真实 PostgreSQL、迁移、Node/Python 测试、构建、benchmark 和 Compose/E2E 全量验证通过，或对外部阻塞有准确记录且未虚报。
7. CR-008 已按运行与合同证据重评；CR-007、可靠队列和实时进度仍保持后续阶段边界。

## 7. 实施约束

- 默认单线程实施；只有用户或仓库规则明确要求时才使用子代理。
- 每项行为变更遵循最窄红灯、实现、回归；不得先写实现再补装饰性测试。
- 真实网络合同必须经过进程间或容器间验证，不能只以 ASGI 进程内测试代替 Compose 结论。
- 不用 CLI 成功替代 HTTP 生产路径结论，不用内存仓储成功替代 PostgreSQL 结论。
- 不修改已发布迁移；本阶段原则上不新增业务数据迁移，如确需新增必须先说明合同原因。
- 不回滚用户已有改动，不提交 `.codegraph/`、依赖、缓存、构建产物、OpenAPI 临时输出、Playwright 报告或测试结果目录。
- 除非用户明确要求，不提交、不推送；阶段完成后停在验证与文档结论，等待确认。
