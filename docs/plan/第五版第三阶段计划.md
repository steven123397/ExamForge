# 第五版第三阶段计划：可靠任务与实时事件

## 1. 阶段目标

将当前 API 进程内 `setTimeout()` 作业执行器升级为基于 PostgreSQL 事实、事务 outbox、Redis/BullMQ 分发和独立 Worker 的可靠任务系统，并通过可补发 SSE 将真实作业事件推送到 Web。API 重启、Worker 崩溃、重复投递、短暂依赖故障和浏览器断线均不得造成重复运行、终态回退或事件历史丢失。

本阶段只完成可靠任务执行与实时事件闭环。完整任务中心、约束策略版本化、跨批次评分改造、页面级前端重构和正式公开部署仍属于后续阶段；腾讯云只进行经用户单独授权的私有试部署和恢复验证。

## 2. 前置条件与启动门槛

1. 第五版第二阶段已经完成并归档；FastAPI scheduler、OpenAPI、API HTTP 客户端和独立镜像合同保持稳定。
2. CR-007 是本阶段唯一需要重评的既有审查问题，不创建重复编号；CR-002、CR-003 保持各自边界。
3. 开始前读取根目录及目标子树 `AGENTS.md`、本文、第五版设计、项目状态和审查状态；存在 `.codegraph/` 时先使用 CodeGraph 理解调用链。
4. 修改前运行 `git status --short --branch`，保留用户已有改动。行为变更先写最窄失败测试，再实现并回归。
5. Redis 和 PostgreSQL 结论必须来自可丢弃的真实隔离实例；内存仓储、mock queue 或单进程事件测试不能替代真实依赖结论。
6. 腾讯云验证涉及远程连接、镜像传输、端口、域名或服务器状态变更，执行前必须取得用户对目标主机、凭据方式和维护窗口的明确授权。

## 3. 范围与边界

### 3.1 本阶段包含

- PostgreSQL 作业请求快照、事件顺序游标、attempt 与 outbox 投递元数据的迁移和合同。
- API 提交事务、Outbox Publisher、BullMQ 队列和独立 Scheduling Worker。
- 基于幂等键、队列 job ID、数据库 CAS 和 attempt 唯一约束的重复投递保护。
- 有界退避重试、超时终态、排队取消、运行中协作式尽力取消和 Worker 崩溃后重新领取。
- PostgreSQL 历史优先、Redis 仅负责唤醒的 SSE；支持 `Last-Event-ID` 补发和终态关闭。
- Web 从轮询切换为 SSE 主路径，并保留断线时的有界兜底查询和明确错误状态。
- Redis、Publisher、Worker 的独立镜像/进程、Compose 健康依赖、资源限制和隔离 E2E。
- 经单独授权后的腾讯云私有试部署，记录资源峰值、重启恢复、SSE 重连和镜像更新证据。

### 3.2 本阶段不包含

- 用 Redis 保存业务唯一事实、替代 PostgreSQL 状态机、会话或草稿 advisory lock。
- 全站缓存、Redis 分布式锁、WebSocket、消息永久历史或伪造线性求解进度。
- 跨进程强杀 OR-Tools 后保留中间最优解；HTTP Abort 只作为协作式尽力取消。
- 完整任务中心页面、全量路由重构、视觉体系重做或新增角色模型。
- `constraint_profiles` 管理页面、策略版本快照、归一化评分或方案实验室。
- 自动发布、公开域名切流、正式生产数据、学校统一身份接入或正式对外开放。

## 4. 核心合同决策

### 4.1 PostgreSQL 是唯一事实源

- `schedule_jobs` 保存版本化请求快照，Worker 不依赖 API 进程内闭包恢复输入。
- 每次消费建立唯一 attempt；作业主状态、attempt、事件和 outbox 必须在同一数据库事务中推进。
- `schedule_job_events` 增加单调顺序游标。SSE 的 `Last-Event-ID` 使用公开事件 ID，服务端先定位其顺序再查询后续事件，不依赖 UUID 字典序或时间戳唯一性。
- outbox 允许至少一次投递。Publisher 在“队列已接收但尚未标记 published”时崩溃可产生重复发送，但 BullMQ job ID、attempt 唯一约束和数据库 CAS 必须让重复发送无副作用。

### 4.2 Redis 与 BullMQ 边界

- Redis 使用 `maxmemory-policy=noeviction`；队列名称、前缀、保留数量和重试上限集中配置。
- BullMQ 只负责分发、延迟重试、并发和 stalled 检测；作业状态与事件历史不从 Redis 反向覆盖 PostgreSQL。
- Publisher 轮询未发布 outbox，使用 `FOR UPDATE SKIP LOCKED` 分批领取。队列投递使用稳定 job ID；Redis Pub/Sub 只发布事件 ID 作为 SSE 唤醒提示，重复或丢失唤醒均由 PostgreSQL 回读消解。
- 初始 Worker 并发固定为 1。提高并发必须另有真实 CPU、内存和 PostgreSQL 证据。

### 4.3 重试、超时与取消

- 只有 `unavailable`、明确可重试的 `internal` 和临时基础设施错误进入自动重试；validation、protocol、业务 infeasible 和调用方取消不重试。
- 每次 attempt 记录开始、完成、错误分类、耗时和 BullMQ attempt 编号。有界指数退避耗尽后只允许一个终态写入。
- 排队作业取消通过 PostgreSQL CAS 直接进入 `cancelled`，Worker 收到残留队列消息时只做幂等确认。
- 运行中取消先记录 `cancellation_requested_at` 和事件；Worker 周期读取取消请求并触发 `AbortController`。若 scheduler 已进入 OR-Tools，取消是尽力语义，最终以数据库第一次合法终态为准。
- API 启动不再把历史 queued/running 作业批量标记 failed；Publisher/Worker 负责恢复投递和 stalled 重试。

### 4.4 SSE 与前端

- SSE 端点仅允许查看当前会话有权访问的内部作业；复用现有 Cookie 会话和 RBAC，不接受浏览器直连 Redis。
- 连接建立后先从 PostgreSQL 补发，再订阅 Redis 唤醒；每次唤醒重新查询游标后的事件。心跳只保活，不制造业务进度。
- 事件使用版本化 shared envelope，至少覆盖 queued、attempt_started、stage_changed、retry_scheduled、cancellation_requested、run_created 和全部终态。
- Web 以 SSE 更新 TanStack Query cache；断线显示可感知状态并按有界退避重连。仅在 SSE 不可用期间执行低频兜底查询，终态后停止连接与轮询。

## 7. 阶段验收标准

1. API 提交返回后可立即重启，作业仍由 Publisher/Worker 领取并最终落入唯一合法终态。
2. Redis 或 Worker 短暂故障不会丢失 PostgreSQL 作业；恢复后能够重新投递，重复消息不生成重复运行。
3. retryable 与 non-retryable 错误分类符合合同，每个 attempt 可追踪，重试耗尽、超时和取消只产生一次终态。
4. SSE 从 PostgreSQL 补发历史，`Last-Event-ID`、漏唤醒、重复唤醒和浏览器重连均不丢业务事件；Redis 不是历史唯一来源。
5. Web 活动作业以 SSE 为主，不再维持 1200 ms 轮询；断线、会话失效、重试、取消和终态均有可访问反馈。
6. 隔离 Compose 从空卷完成真实会话、可靠排考、SSE、持久化、API/Publisher/Worker/scheduler 重启与浏览器主链。
7. 腾讯云私有试部署只在取得授权后执行；若外部环境阻塞，准确记录阻塞，不用本地 Compose 冒充远程通过。
8. CR-007 只在可靠队列与 SSE 证据齐全后移入已解决索引；CR-002、CR-003 不因本阶段顺手改动。

## 8. 实施约束

- 默认单线程实施；只有用户或仓库规则明确要求时才使用子代理。
- 不修改已发布迁移；不以 mock queue、内存仓储或进程内 ASGI 成功替代真实 Redis/PostgreSQL/进程间结论。
- 不让 Redis 成为作业状态、结果或事件历史唯一来源，不用 Pub/Sub 到达保证替代数据库补发。
- 不记录密码、Cookie、Redis/PostgreSQL 凭据、完整排考输入或内部堆栈；错误与诊断必须可关联且脱敏。
- 不提交 `.codegraph/`、依赖目录、缓存、构建产物、Playwright 报告、测试结果或远程密钥。
- 未经用户明确要求不提交、不推送；阶段完成后停在验证与文档结论，等待确认。
