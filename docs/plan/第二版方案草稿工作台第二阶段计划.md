# 第二版方案草稿工作台第二阶段计划

## 1. 目标

在第一阶段已经完成“创建草稿、人工调整、硬约束校验、无冲突发布”的基础上，第二阶段补齐更完整的草稿治理能力：真实 PostgreSQL 路径验证、草稿废弃、草稿对比、发布确认面板、审计可读性和课程报告素材。完成后，第二版应能稳定演示“自动排考 → 草稿调整 → 冲突修复 → 对比确认 → 发布/废弃 → 查询发布结果”的运营闭环。

## 2. 范围

- 本轮包含：真实 PostgreSQL 迁移和草稿 API 验证脚本或测试记录。
- 本轮包含：`POST /api/schedule-drafts/:id/discard`，以及 Web 草稿废弃入口。
- 本轮包含：草稿与来源运行、当前发布版本的差异摘要，并在 Web 发布确认面板展示评分、冲突、变化数量和最近调整记录。
- 本轮包含：草稿审计事件展示增强，区分创建、调整、校验、发布和废弃。
- 本轮包含：课程报告可复用材料，包括流程说明、状态迁移说明、测试结论和关键取舍。
- 本轮不包含：拖拽式矩阵、局部重排、登录权限、异步队列、WebSocket、通知和导出文件。

## 3. 前置条件

- 第一阶段提交 `4154b61` 已在本地 `main`。
- 当前活动设计依据为 `docs/design/第二版实现内容设计.md`。
- Docker/PostgreSQL 如不可用，需要明确记录真实数据库验证阻塞，并保留内存仓储和 API 测试结果。
- 调度器回归仍使用 `uv run --python 3.12 --extra dev python -m pytest -q`。

## 5. 验证方式

本计划完成前运行：

```bash
npm test
npm run typecheck
npm run build
cd apps/scheduler && uv run --python 3.12 --extra dev python -m pytest -q
git diff --check
```

真实数据库路径额外验证：

```bash
psql "$DATABASE_URL" -f packages/db/drizzle/0003_schedule_drafts.sql
npm run seed --workspace @examforge/db
DATABASE_URL="$DATABASE_URL" npm run dev --workspace @examforge/api
```

随后通过 API 或 Web 验证草稿创建、调整、发布和查询。若本机 PostgreSQL 环境不可用，需要记录阻塞原因和已完成的替代验证。
