## 2. 项目背景概述

高校期末考试通常涉及课程、班级、学生群体、教师、考场、时间段和监考安排等多类资源。传统人工排考容易出现学生考试冲突、教师监考冲突、考场容量不足、考场设备不匹配和资源利用不均衡等问题。

本项目拟设计并实现一个 Web 化排考与考务资源调度系统，通过基础数据管理、自动排考、冲突检测、人工调整、方案发布和结果查询等功能，提升排考效率和排考结果质量。

## 3. 技术可行性

### 3.1 技术基础可行

本项目采用的技术路线包括：

- 前端：Next.js、TypeScript、Tailwind CSS。
- 后端：Node.js、Fastify、TypeScript。
- 数据库：PostgreSQL。
- 算法：Python、OR-Tools CP-SAT。
- 测试：pytest、Vitest、Playwright。

上述技术均为成熟开源技术，资料丰富、社区活跃、开发工具完善，适合在 WSL/Linux 环境中进行本地开发和调试。

### 3.2 算法方案可行

考场排考问题本质上属于约束满足与优化问题。系统需要在满足硬约束的前提下优化软约束评分。OR-Tools CP-SAT 能表达布尔决策变量、互斥约束、容量约束、时间窗口约束和加权惩罚项，适合用于排考、排课、排班等问题。

第一版不会直接追求所有复杂约束，而是先实现以下核心约束：

- 学生群体同时间段不能重复考试。
- 教师同时间段不能重复监考。
- 考场同时间段不能重复占用。
- 考场容量、类型和设备必须满足考试要求。
- 考试必须安排在允许时间段内。
- 教师不可用时间不得安排监考。

这些约束可以用结构化数据和 CP-SAT 模型表达，因此算法实现具有可行性。

### 3.3 工程实现可行

项目采用分阶段实施策略：

1. 先实现 Python 算法原型和测试数据生成器。
2. 再实现数据库 schema、后端 API 和前端管理页面。
3. 最后增强拖拽调整、异步任务、实时进度推送和部署能力。

这种方式可以降低一次性搭建完整全栈的风险，使核心算法和业务模型先得到验证。

### 3.4 技术风险与应对

| 风险 | 影响 | 应对方案 |
| --- | --- | --- |
| 约束模型复杂度高 | 算法原型延期 | 先实现 P0 硬约束，再逐步增加软约束 |
| 大规模数据求解耗时 | 影响演示体验 | 设置求解时间限制，输出当前最优结果和运行统计 |
| 冲突解释不够精确 | 用户难以理解失败原因 | 先实现确定性预检和规则级解释 |
| 全栈范围过大 | 开发周期失控 | 按 P0/P1/P2 分阶段推进 |
| 前端矩阵交互复杂 | UI 实现耗时 | 先实现只读矩阵和列表视图，再实现拖拽调整 |

综合来看，项目技术路线具备可行性，但必须坚持“算法原型优先、全栈分阶段落地”的策略。

## 4. 经济可行性

### 4.1 开发成本可控

本项目为课程设计项目，主要成本是个人开发时间和本地计算资源。项目使用开源技术栈，不需要购买商业软件授权。

主要工具和资源包括：

- WSL/Linux 开发环境。
- VS Code 或其他编辑器。
- Git 和 GitHub。
- Python、Node.js、PostgreSQL 等开源软件。
- 浏览器和本地测试工具。

这些工具均可免费使用，经济成本较低。

### 4.2 部署成本可控

第一版可以在本地环境完成演示。若需要云端展示，可使用一台普通云服务器部署 Docker Compose 环境。由于系统面向课程展示，不需要高并发生产环境，因此服务器配置要求不高。

部署资源可按需启用：

- 本地演示：无需额外成本。
- 云端演示：低配 Linux 服务器即可。
- 数据库持久化：使用本机 PostgreSQL 或容器卷。

### 4.3 预期收益明确

系统设计完成后可以带来以下收益：

- 明显提高课程报告技术含量。
- 展示比普通管理系统更强的算法和工程深度。
- 为后续软件工程文档提供真实设计依据。
- 形成可演示的排考流程、测试结果和可视化材料。

因此，从投入产出角度看，本项目经济可行。

## 5. 操作可行性

### 5.1 用户操作流程清晰

系统面向三类核心用户：

- 教务管理员：维护基础数据、创建考试任务、运行排考、调整方案、发布结果。
- 教师：维护不可用时间、查看监考安排和工作量。
- 学生或只读用户：查询已发布考试安排和变更通知。

这些操作符合高校考务工作的真实流程，用户角色和职责边界清晰。

### 5.2 业务流程可落地

系统业务流程包括：

1. 基础数据准备。
2. 考试任务创建。
3. 约束配置。
4. 自动排考。
5. 冲突检测与解释。
6. 人工调整。
7. 方案发布。
8. 教师和学生查询。

该流程与实际排考工作一致，具备可操作性。

### 5.3 界面交互可理解

第一版前端设计以数据密集型管理后台为主，使用列表、筛选、表格、矩阵视图和冲突面板展示信息。管理员可以从排考列表和考场矩阵中快速定位冲突；教师和学生只需查看个人相关安排，操作复杂度较低。

### 5.4 操作风险与应对

| 风险 | 影响 | 应对方案 |
| --- | --- | --- |
| 基础数据录入错误 | 导致排考失败或结果不准确 | 提供导入校验、错误行提示和数据预检 |
| 管理员误发布方案 | 教师和学生看到错误安排 | 发布前执行硬约束校验，并保留版本回滚 |
| 用户不理解冲突原因 | 无法修正数据 | 输出业务化冲突解释和处理建议 |
| 页面信息量过大 | 操作效率下降 | 提供筛选、分组、矩阵视图和摘要指标 |

综合来看，系统操作流程明确，用户学习成本可控，操作可行。

## 6. 法律可行性

### 6.1 开源软件使用可控

项目计划使用的主要技术均为主流开源软件。开发过程中应遵守各依赖组件的开源许可证要求，在课程报告或项目文档中说明主要依赖来源。

项目不计划将系统作为商业产品发布，因此开源许可证风险较低。若后续对外发布，应进一步检查各依赖许可证与分发方式是否兼容。

### 6.2 数据合规风险可控

课程设计阶段不接入学校真实教务系统，也不使用真实学生个人隐私数据。演示数据由测试数据生成器生成，或由手工构造的模拟数据提供。

第一版应避免采集以下真实敏感信息：

- 真实学生身份证号、手机号、家庭住址等个人敏感信息。
- 真实成绩、处分、学籍状态等敏感教育数据。
- 未授权的教师个人隐私数据。

系统只处理模拟考试任务、模拟教师、模拟学生群体和模拟考场，因此数据合规风险较低。

### 6.3 学术诚信要求可满足

课程要求单人独立完成，禁止抄袭文档图表。本项目的需求分析、设计文档、算法模型、测试用例和课程报告材料均应基于本人项目实现过程整理。使用 AI 工具辅助时，应对生成内容进行审阅、修改和验证，避免直接复制未经理解的内容。

### 6.4 法律风险与应对

| 风险 | 影响 | 应对方案 |
| --- | --- | --- |
| 使用真实个人数据 | 产生隐私和合规风险 | 全部使用模拟数据和生成数据 |
| 开源许可证理解不足 | 后续分发受限 | 记录主要依赖和许可证 |
| AI 生成内容未经审查 | 影响课程诚信和文档质量 | 所有文档经过人工审阅和项目化修改 |
| 误导为真实教务系统 | 产生使用边界误解 | 在报告中说明系统为课程设计与演示系统 |

综合来看，项目法律可行。

## 7. 进度可行性

课程提交时间为 8 月 5 日之前。项目采用分阶段策略，能够根据剩余时间控制范围。

建议进度安排：

| 阶段 | 内容 | 产出 |
| --- | --- | --- |
| 阶段 1 | 需求、可行性、总体设计 | 需求文档、可行性分析、总体设计 |
| 阶段 2 | 数据模型与算法原型 | 数据模型文档、Python 排考原型、测试数据 |
| 阶段 3 | 核心 Web 平台 | 基础数据、考试任务、排考结果查看 |
| 阶段 4 | 测试与报告整理 | 测试方案、测试用例、截图、Word 报告 |

如果时间不足，应优先保证算法原型、核心数据模型、关键页面截图和课程报告完整性。

## 8. 结论

从技术、经济、操作、法律和进度角度分析，本项目具备可行性。

项目的主要风险集中在约束模型复杂度和全栈实现范围上。通过“算法原型优先、核心平台其次、体验增强最后”的分阶段策略，可以在课程周期内完成一个具有技术深度、业务广度和报告完整度的排考系统。
