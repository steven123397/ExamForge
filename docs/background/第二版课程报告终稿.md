# ExamForge 第二版课程报告终稿

## 1. 项目概述

ExamForge 是面向高校期末考试的智能排考与考务资源调度系统。系统以课程、学生群体、考场、时间段、教师和考试任务为输入，自动生成排考方案，并提供冲突解释、评分、人工调整、草稿治理、发布回滚和已发布查询能力。

第二版的重点不是继续扩展普通后台 CRUD，而是完成“自动排考 -> 方案草稿 -> 人工调整 -> 硬约束校验 -> 对比确认 -> 发布治理 -> 通知导出”的人机协同闭环。

## 2. 需求与角色

系统服务三类角色：

- 管理员：维护基础数据，发起排考，发布或回滚最终方案。
- 排考员：创建草稿，调整考试时间、考场、监考教师，处理硬约束冲突。
- 查询用户：查看已发布考试安排。

核心业务要求包括：排考结果可追溯、人工调整可审计、硬约束不可绕过、发布结果可查询、课程报告可说明关键设计取舍。

## 3. 系统架构

第二版采用分层架构：

- `apps/scheduler`：Python OR-Tools 排考原型，负责自动排考、预检、评分和报告整理。
- `apps/api`：Fastify API，负责业务编排、仓储访问、草稿校验、发布治理和轻量权限护栏。
- `packages/shared`：共享领域模型、DTO 和 Zod schema。
- `packages/db`：PostgreSQL schema、迁移、seed 和持久化仓储。
- `apps/web`：Next.js 运营台，提供基础数据、排考运行、方案工作台和已发布查询界面。

Web 层只展示和提交操作，不承载约束判断；API 层统一执行硬约束校验和状态迁移；调度器保持独立可测试。

## 4. 关键功能

### 4.1 自动排考与异步作业

系统支持同步发起排考，也支持通过异步作业 API 创建后台排考任务。异步作业记录队列状态、运行进度、完成状态和生成的 `schedule_run`，Web 运营台通过轮询展示进度。

### 4.2 方案草稿与人工调整

排考运行可以复制为草稿。草稿保存独立安排、冲突记录和变更事件。排考员可以在时间段 × 考场矩阵中选择考试，调整时间段、考场和监考教师。

每次调整后，API 重新校验考场时间唯一、教师时间唯一、学生群体时间唯一、教师不可用、容量、考场类型、设备、允许时段和监考人数等硬约束。

### 4.3 锁定考试与局部再平衡

草稿支持锁定单场考试。锁定后该考试不能被人工 PATCH 修改，局部再平衡也会跳过锁定考试。局部再平衡只处理当前草稿中受冲突影响且未锁定的考试，并复用已有候选建议能力寻找无硬冲突安排。

### 4.4 发布治理

草稿发布前展示评分、冲突数量、相对来源变化、相对已发布变化和最近调整记录。存在硬冲突时禁止发布。发布成功后生成新的稳定排考运行，并更新当前批次的已发布版本；草稿也可以废弃，终态草稿禁止继续调整。

### 4.5 通知与导出

已发布方案可按教师或学生群体查询。系统提供学生群体通知预览和 CSV 导出，便于课堂演示“发布后对外分发”的业务流程。

## 5. 关键取舍

- repository 作业表替代生产级 Redis 队列：降低课程实现复杂度，同时保留作业状态、PostgreSQL 持久化路径和 Web 轮询形态。
- 内置演示账号和 Bearer token 替代 SSO：证明 API 权限边界，避免把实现重点转移到外部身份系统。
- 局部再平衡作为独立 API：避免单场人工调整接口混入自动求解逻辑。
- 硬约束优先于软约束优化：第二版重点是发布安全和人工协同，评分用于比较和解释。

## 6. 测试与验证

第二版验证覆盖：

- API 行为测试：排考运行、草稿创建、人工调整、硬约束冲突、发布/废弃、局部建议、异步作业、锁定/解锁、局部再平衡、权限、教师不可用、通知和 CSV 导出。
- 前端类型检查与构建：验证 Next.js 运营台可编译。
- Playwright 浏览器验收：验证方案工作台建议应用、矩阵拖拽、锁定/解锁、局部再平衡、异步作业和通知预览。
- 调度器回归：验证第一版排考算法原型未被第二版破坏。

## 7. 后续展望

若继续企业化，优先方向包括：持久化任务队列、真实认证授权、消息投递状态、下载审计、标准迁移执行器、部署监控和更完整的局部优化算法。
