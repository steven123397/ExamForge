# 第四版算法规模验证记录

## 1. 验证目的

本记录用于验证第四版调度器在 50、100、150 场考试规模下的完整可行性、求解耗时和教师负载分布。规模数据由确定性 witness 构造生成，不通过重复随机尝试筛选可行输入。

## 2. 验证环境

- 验证日期：2026-07-11。
- 操作系统：WSL2，Linux `6.18.33.2-microsoft-standard-WSL2`，`x86_64`。
- 处理器：13th Gen Intel Core i5-13500H，16 个逻辑 CPU。
- Python：`3.12.13`。
- OR-Tools：`9.15.6755`。
- Node.js：`v24.16.0`。
- npm：`11.13.0`。

## 3. 输入与求解口径

- 随机种子：`20260711`。
- 单阶段求解时间上限：30 秒。
- 规模：50、100、150 场考试。
- 每场考试由 witness 预先绑定一个满足容量、类型、设备和时段约束的候选安排。
- 教师数量不低于 witness 的单时段峰值监考需求。
- 启用全部硬约束、教师负载均衡、考场利用率、学生连续考试和考试分布目标。
- `teacher_consecutive_invigilation` 在规模 profile 中显式设为 `0`。连续监考目标由专项单元测试覆盖，规模基准不执行其全局最优证明。

执行命令：

```bash
npm run benchmark:scheduler
```

## 5. 结论

三个规模均返回 `feasible`，安排数与考试数一致，硬冲突数为 `0`。本机环境下，150 场考试在 1.127 秒内完成，教师负载极差为 1，满足第四版第二阶段的规模验收要求。

当前总分采用固定 100 分减累计软惩罚并截断到 0 的口径。规模增加后软惩罚会饱和，因此本次三档 `score` 均为 0；该值不表示不可行，也不适合直接用于跨规模质量比较。后续如需跨批次质量分析，应单独设计归一化评分口径。
