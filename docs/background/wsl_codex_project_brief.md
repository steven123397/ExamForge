# WSL Codex 项目背景文档

## 1. 项目背景

本项目是软件工程课程设计大作业，对应的原始课程要求文档为：

- `docs/background/软件工程课程设计大作业要求-2026.md`

课程原始要求只要求提交一份完整的软件工程课程设计 Word 报告，但本项目的实际目标高于最低要求。后续应先按真实软件项目进行分析、设计、开发和验证，再根据最终实现结果整理课程报告。

原始文档中写明的提交时间为 8 月 5 日之前。

## 2. 课程最低要求

课程最低提交物是 Word 报告，不要求提交源代码、数据库脚本或可运行程序。但本项目将主动超过这一最低标准。

报告章节顺序固定，需要覆盖软件完整生命周期：

1. 引言
2. 可行性分析
3. 需求分析
4. 系统总体设计
5. 详细设计
6. 测试方案设计
7. 软件维护规划
8. 总结与展望

课程文档中的硬性要求包括：

- 至少划分两类用户角色，并完成权限隔离设计。
- 数据库至少设计 3 张及以上有关联关系的数据表。
- 提供完整用例图，并为每条用例编写用例规约。
- 提供系统分层架构图和整体功能模块结构图。
- 提供数据库 E-R 图和完整数据表结构设计。
- 提供核心业务类图，并体现封装、继承、多态等面向对象特征。
- 提供核心业务流程时序图或活动图。
- 提供大批量标准化测试用例，覆盖正常场景、边界场景、异常错误场景。
- 提供软件维护规划，覆盖纠错性维护、适应性维护、完善性维护、预防性维护。

## 3. 用户已明确的真实目标

用户不希望只做一个满足最低报告要求的虚构系统。

本项目的目标是做出一个技术标准接近商业级的软件系统。这里的“商业级”不是指真正上线盈利或商业运营，而是指：

- 系统应具有足够的技术深度。
- 系统应具有较完整的业务广度，而不是单一演示功能。
- 架构、代码质量、可测试性、部署方式、文档记录都应认真对待。
- 最终课程报告应基于真实的设计与实现决策，而不是纯粹编造。

## 4. 已确定选题

课程原始选题为：

> 考场自动排考调度系统

建议使用的扩展项目名称为：

> 高校智能排考与考务资源调度系统

这个名称仍然对应课程文档中的“考场自动排考调度系统”，但比原题有更大的发挥空间，可以覆盖考试排程、考场分配、监考安排、冲突检测、人工调整、结果发布等完整考务流程。

## 5. 选题理由

在课程文档列出的所有选题中，该题目最符合“技术深度和技术广度都尽量高”的目标。

它不是普通的增删改查管理系统，而是天然包含：

- 约束建模
- 资源调度
- 冲突检测
- 硬约束与软约束优化
- 大规模测试数据
- 排考结果可视化
- 自动排考后的人工微调
- 排考版本管理、回滚、审计日志和管理流程

与校园二手交易、图书推荐、便利店进销存、会员管理、图书借阅、轻量桌面工具等题目相比，排考系统有更强的算法核心，同时也能扩展成完整的管理平台。

## 6. 开发环境方向

用户计划转到 WSL 中使用 Codex CLI 和 Superpowers 继续开发，因此后续建议以 WSL/Linux 作为主开发环境。

建议环境分工：

- WSL/Linux：主要代码开发、后端、前端、数据库、测试、脚本、本地服务运行。
- Windows：VS Code、浏览器预览、截图、Word/WPS 报告撰写、图表编辑、最终打包。
- 云服务器：后期可作为演示、部署、联调、HTTPS、数据库持久化和备份环境。

可考虑的技术方向如下，最终由 WSL Codex 结合可行性进一步决定：

- 前端：React、Next.js 或 Vue。
- 后端：Node.js + TypeScript，或在充分比较后选择更合适的栈。
- 数据库：PostgreSQL 优先；早期原型也可考虑 SQLite。
- 排考算法：约束规划、启发式搜索或混合调度策略。可评估 OR-Tools、Timefold/OptaPlanner 类约束求解器，或根据项目规模自研调度算法。
- 部署：Linux 服务器 + Nginx/Caddy + 后端服务 + 持久化数据库。若环境允许，可考虑 Docker。

目前没有锁定最终技术栈。WSL 侧 Codex 应根据算法可行性、开发速度、部署复杂度和课程周期进行判断。

## 7. 期望系统范围

系统应定位为完整的 Web 化排考与考务管理平台，而不是单独的排考脚本。

可作为起点考虑的功能范围包括：

- 用户登录与基于角色的权限控制。
- 教务管理员角色，用于基础数据维护、排考任务创建和结果发布。
- 教师角色，用于查看监考安排、维护不可用时间或查看相关考务信息。
- 学生或只读角色，用于查询考试安排；是否实现由后续需求分析决定。
- 课程、班级、学生群体、教师、考场、教学楼、时间段、考试任务、排考结果等基础数据管理。
- 基础数据导入导出。
- 自动排考。
- 冲突检测与冲突原因解释。
- 排考结果人工调整。
- 排考方案版本管理与回滚。
- 考场占用视图。
- 教师监考工作量统计。
- 排考结果发布与查询。
- 操作日志与审计记录。
- 小规模、中规模、大规模测试数据生成器。

以上不是正式需求分析，只是提供给 WSL Codex 的起始范围。正式需求、边界和优先级应在 WSL 环境中重新梳理。

## 8. 算法深度预期

排考模块应作为本项目的技术核心。

可考虑的硬约束包括：

- 同一学生群体不能在同一时间参加两场考试。
- 同一教师不能在同一时间监考两个考场。
- 同一考场不能在同一时间安排两场考试。
- 考场容量必须满足考试人数。
- 考场类型、校区或设备条件应满足考试要求。
- 考试必须安排在允许的考试时间段内。

可考虑的软约束包括：

- 尽量避免同一学生群体连续考试过多。
- 尽量平衡教师监考工作量。
- 在满足容量的前提下尽量合理使用考场，避免资源浪费。
- 尽量减少跨校区或跨楼栋移动。
- 尽量让考试在考试周期内分布均衡。
- 对特定课程类型可设置偏好时间段。

算法输出不应只是一张排考表，还应尽量包含：

- 是否满足所有硬约束。
- 软约束评分或解释。
- 无法生成可行方案时的冲突列表。
- 排考运行时间、迭代次数等统计信息。
- 人工调整前后的冲突与评分变化。

## 9. 工程质量标准

项目实现应尽量达到较高工程标准：

- 架构清晰，模块边界明确。
- 尽量使用类型安全的数据模型。
- 数据库表结构应有合理约束。
- API 设计应能支撑真实前端页面。
- 提供种子数据和可复现演示场景。
- 对核心排考逻辑编写自动化测试。
- 对非法数据、冲突数据进行校验。
- 错误信息和冲突解释应有实际帮助。
- 前端界面应能清晰查看、筛选和调整排考结果。
- 部署路径应有清楚记录。
- 重要设计决策应保留记录，方便后续转化为课程报告内容。

本项目不需要真的成为商业产品，但应体现接近商业级的工程思维。

## 10. 当前边界

本文档只是交接背景文档，不是正式需求分析。

完整的需求分析、系统设计、技术选型、数据模型、架构方案、实现计划和开发任务应在 WSL Codex 环境中继续完成。

Windows 侧目前只确定了以下结论：

- 课程最低要求不是本项目的真实目标。
- 真实目标是做一个技术深度和广度都较强的高质量软件系统。
- 选题确定为考场自动排考调度系统。
- 推荐扩展名称为高校智能排考与考务资源调度系统。
- WSL/Linux 应作为主开发环境。
- Windows 仍用于报告、截图、图表和最终文档整理。
- 云服务器资源后期可根据部署和演示需要使用。
