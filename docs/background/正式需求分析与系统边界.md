# 高校智能排考与考务资源调度系统正式需求分析与系统边界

## 1. 文档定位

本文档是 ExamForge 项目的正式需求分析与系统边界说明，用于承接课程要求中的“需求分析”章节，并作为后续总体设计、数据库设计、排考算法原型、后端 API、前端页面和测试方案的依据。

本项目对应课程选题“考场自动排考调度系统”，扩展项目名称为“高校智能排考与考务资源调度系统”。课程最低要求只需提交 Word 报告，但本项目按接近真实软件项目的标准推进：先完成可验证的需求、设计、实现和测试，再将成果整理为课程报告。

## 2. 产品目标

系统目标不是提供一个单独的排考脚本，而是建设一个 Web 化的高校考务排考与资源调度平台。系统应支持从基础数据维护、考试任务创建、自动排考、冲突解释、人工调整、方案发布、结果查询、统计审计到测试数据生成的完整业务闭环。

核心目标包括：

1. 降低人工排考工作量，减少学生、教师、考场和时间段冲突。
2. 对排考结果提供可解释的冲突原因、约束满足情况和软约束评分。
3. 支持教务管理员在自动排考基础上进行人工微调，并实时检测调整后的冲突。
4. 支持教师和学生查询已发布的考试与监考安排。
5. 支持排考方案版本管理、发布、回滚和操作审计。
6. 为课程报告提供完整的软件工程材料，包括角色建模、用例规约、业务流程、非功能需求、测试依据和维护边界。

## 3. 目标用户与角色边界

系统第一版正式边界采用三类核心角色，并保留后续扩展角色空间。

### 3.1 教务管理员

教务管理员是系统的主要业务操作者，负责基础数据、考试任务、排考运行、人工调整、结果发布和系统审计。

主要职责：

- 维护院系、专业、班级、课程、学生群体、教师、教学楼、考场、时间段等基础数据。
- 导入和校验考试任务数据。
- 配置排考约束和算法权重。
- 发起自动排考并查看运行结果。
- 查看冲突列表、冲突解释和软约束评分。
- 对排考结果进行人工调整。
- 保存排考方案版本，发布或回滚排考方案。
- 查看教师监考工作量、考场占用、考试分布等统计信息。
- 查看操作日志和审计记录。

### 3.2 教师

教师是考务安排的参与者和查询者，重点关注监考安排、个人不可用时间和考务通知。

主要职责：

- 查看本人监考安排。
- 维护或提交本人不可用时间段。
- 查看与本人相关的考试任务、考场和时间安排。
- 接收排考结果发布或变更通知。
- 查看个人监考工作量统计。

### 3.3 学生/只读查询用户

学生或只读查询用户用于补全排考结果发布后的业务闭环。第一版只提供只读能力，不提供复杂交互。

主要职责：

- 查询本人或所在班级/学生群体的考试安排。
- 查看考试时间、课程、考场、座位或考场说明。
- 查看已发布的考试变更通知。
- 发现查询信息异常时，通过线下或预留反馈入口联系教务管理员。

### 3.4 后续可扩展角色

以下角色不进入第一版硬性实现范围，但在权限模型中预留扩展空间：

- 系统管理员：负责用户、角色、系统配置、备份恢复。
- 院系管理员：负责本院系课程、教师和考试任务初审。
- 巡考/考务人员：查看跨考场巡考安排和异常记录。

## 4. 系统范围

### 4.1 纳入第一版正式范围

第一版采用相对激进的完整平台边界，纳入以下能力：

1. 认证与权限控制。
2. 基础数据管理。
3. 数据导入、导出和校验。
4. 考试任务管理。
5. 排考约束配置。
6. 自动排考。
7. 冲突检测与冲突原因解释。
8. 排考结果可视化查询。
9. 人工调整与调整后实时校验。
10. 排考方案版本管理、发布和回滚。
11. 教师监考工作量统计。
12. 考场占用视图。
13. 学生考试安排查询。
14. 操作日志与审计记录。
15. 小规模、中规模、大规模测试数据生成器。
16. 核心排考算法自动化测试与算法效果对比材料。

### 4.2 不纳入第一版范围

以下能力暂不纳入第一版实现和课程报告主线，只作为展望：

1. 与学校真实教务系统、统一身份认证系统、短信平台、邮件平台的正式集成。
2. 自动座位级编排、准考证打印、考场签到、缺考统计和成绩录入。
3. 移动端 App、小程序和离线桌面端。
4. 多学校、多租户、跨校区独立组织架构的复杂 SaaS 化能力。
5. 真正生产环境的数据脱敏、灾备、多活、高可用集群。
6. 复杂审批流，例如多级院系审核、教师申诉流和法定公示流程。

### 4.3 系统边界说明

系统负责“考试安排和考务资源调度”，不负责教学计划制定、成绩管理、课程选课、学生学籍管理等上游业务。上游数据在第一版中通过手工维护、模板导入或种子数据生成进入系统。

系统的核心输出是“已发布的排考方案”，包括考试时间、考场、课程、学生群体、监考教师、冲突检测结果、软约束评分、方案版本和审计记录。系统不直接承担考试执行现场的签到、监控和成绩处理。

## 5. 核心业务流程

### 5.1 基础数据准备流程

1. 教务管理员登录系统。
2. 管理员维护或导入课程、教师、学生群体、考场、教学楼、考试时间段等基础数据。
3. 系统对导入数据进行格式校验、必填校验、唯一性校验和关联完整性校验。
4. 系统提示错误数据、重复数据和潜在排考风险。
5. 管理员修正数据后确认基础数据进入可排考状态。

### 5.2 考试任务创建流程

1. 管理员创建考试批次，例如“2026 春季期末考试”。
2. 管理员录入或导入考试任务，包含课程、学生群体、人数、考试时长、考场类型、设备要求、校区要求、特殊时间要求等。
3. 系统检查任务是否缺少课程、学生群体、人数、时长或约束字段。
4. 管理员确认考试任务集合，进入排考准备状态。

### 5.3 自动排考流程

1. 管理员选择考试批次。
2. 管理员配置硬约束开关、软约束权重和算法参数。
3. 系统运行排考算法，尝试为每个考试任务分配时间段、考场和监考教师。
4. 系统输出排考方案、硬约束满足情况、软约束评分、运行时间、迭代次数和无法安排的任务列表。
5. 若无可行方案，系统输出冲突原因和建议修复方向。
6. 管理员保存排考方案为草稿版本。

### 5.4 冲突检测与人工调整流程

1. 管理员查看排考结果列表、日历视图、考场占用视图和教师工作量统计。
2. 管理员选择某一考试安排进行时间、考场或监考教师调整。
3. 系统实时检测调整后的学生冲突、教师冲突、考场冲突、容量冲突、设备冲突和时间窗口冲突。
4. 若调整产生冲突，系统标记冲突类型、影响对象和原因。
5. 管理员确认或撤销调整。
6. 系统记录调整前后差异、评分变化和操作日志。

### 5.5 方案发布与回滚流程

1. 管理员确认某一排考方案版本。
2. 系统执行发布前校验，确保不存在硬约束冲突或确认保留的例外。
3. 管理员发布方案。
4. 教师和学生可以查询已发布安排。
5. 若后续发现严重问题，管理员可以回滚到历史版本或基于历史版本创建新草稿。
6. 系统记录发布、回滚和重新发布的审计日志。

### 5.6 查询与统计流程

1. 教师登录后查看本人监考安排和不可用时间状态。
2. 学生或只读用户查看本人或所在学生群体的考试安排。
3. 管理员查看考场占用、教师监考工作量、考试时间分布、冲突数量和算法评分趋势。
4. 系统支持按课程、班级、教师、考场、时间段、校区等条件筛选。

## 6. 功能性需求

### FR-01 用户认证与权限控制

系统应支持用户登录、退出、会话管理和基于角色的访问控制。不同角色只能访问其职责范围内的页面、接口和数据。

权限边界：

- 教务管理员可以管理基础数据、考试任务、排考方案、发布状态和审计日志。
- 教师只能查看本人相关安排，维护本人不可用时间，查看个人统计。
- 学生/只读用户只能查看已发布的考试安排和变更通知。

### FR-02 基础数据管理

系统应支持维护院系、专业、班级、课程、学生群体、教师、教学楼、考场、考试时间段等基础数据。基础数据应支持新增、编辑、停用、查询、筛选、导入和导出。

系统应校验基础数据之间的关联关系，例如课程所属院系、学生群体人数、教师所属院系、考场所属教学楼、考场容量和设备标签。

### FR-03 数据导入导出

系统应提供标准导入模板，用于批量导入课程、教师、学生群体、考场和考试任务。导入时应输出成功数量、失败数量、错误行号、错误字段和修复建议。

系统应支持导出基础数据、排考结果、教师监考安排、学生考试安排和统计报表。

### FR-04 考试批次与考试任务管理

系统应支持按考试批次组织排考工作。每个考试批次包含考试周期、允许时间段、考试任务集合、约束配置、排考方案版本和发布状态。

考试任务应至少包含课程、学生群体、考试人数、考试时长、可用时间范围、考场类型要求、设备要求、校区要求、监考教师需求数量等信息。

### FR-05 约束配置

系统应支持配置排考硬约束和软约束。

硬约束包括：

- 同一学生群体同一时间不能参加多场考试。
- 同一教师同一时间不能监考多场考试。
- 同一考场同一时间不能安排多场考试。
- 考场容量必须满足考试人数。
- 考场类型、校区和设备条件必须满足考试要求。
- 考试必须安排在允许的考试时间段内。
- 教师不可用时间段不得安排监考。

软约束包括：

- 尽量避免同一学生群体连续考试过多。
- 尽量平衡教师监考工作量。
- 尽量提高考场容量利用率。
- 尽量减少跨校区或跨楼栋移动。
- 尽量让考试在考试周期内分布均衡。
- 尽量满足特定课程类型的偏好时间段。

### FR-06 自动排考

系统应提供自动排考功能。算法输入包括考试任务、考场、时间段、教师、学生群体、硬约束和软约束权重。算法输出包括排考方案、可行性状态、冲突列表、评分结果和运行统计。

系统应支持至少三种规模的演示数据：

- 小规模：用于快速演示和单元测试。
- 中规模：用于常规系统测试。
- 大规模：用于算法性能和稳定性测试。

### FR-07 冲突检测与解释

系统应对自动排考和人工调整后的方案执行冲突检测。冲突信息应包含冲突类型、冲突对象、冲突时间、影响范围、严重程度和建议处理方向。

冲突类型至少包括：

- 学生群体考试冲突。
- 教师监考冲突。
- 考场占用冲突。
- 考场容量不足。
- 考场设备或类型不匹配。
- 超出考试时间窗口。
- 教师不可用时间冲突。

### FR-08 排考结果查看

系统应支持多视角查看排考结果：

- 排考列表视图。
- 按日期和时间段组织的日历视图。
- 考场占用视图。
- 教师监考视图。
- 学生群体考试视图。
- 冲突和评分摘要视图。

系统应支持按考试批次、课程、教师、考场、班级、时间段、校区和冲突状态筛选。

### FR-09 人工调整

系统应允许教务管理员在草稿方案中调整考试时间、考场和监考教师。调整后系统应立即重新计算硬约束冲突和软约束评分变化。

每次人工调整应记录调整前值、调整后值、操作人、操作时间、调整原因和评分变化。

### FR-10 方案版本管理、发布与回滚

系统应支持同一考试批次下保存多个排考方案版本。版本状态包括草稿、待发布、已发布、已回滚和归档。

系统应支持：

- 保存自动排考结果为方案版本。
- 基于历史版本复制新草稿。
- 比较两个版本之间的安排差异。
- 发布指定版本。
- 回滚已发布版本。
- 记录发布和回滚审计日志。

### FR-11 教师不可用时间管理

教师应能提交本人不可用时间段。管理员应能查看教师不可用时间对排考的影响，并在必要时进行审核或备注。

第一版可采用简化规则：教师提交不可用时间后，系统默认作为排考硬约束参与计算；管理员可在特殊情况下标记例外说明。

### FR-12 统计分析

系统应提供以下统计能力：

- 教师监考次数和监考时长统计。
- 考场容量利用率统计。
- 考试在时间段上的分布统计。
- 冲突数量和冲突类型统计。
- 自动排考评分和人工调整评分变化统计。

### FR-13 结果发布与查询

管理员发布排考方案后，教师和学生才能查询正式安排。未发布方案只对管理员可见。

教师查询结果应包含课程、考试时间、考场、监考角色和相关备注。学生查询结果应包含课程、考试时间、考场、考试对象、考试说明和变更提示。

### FR-14 操作日志与审计

系统应记录关键操作，包括登录、导入数据、修改基础数据、创建考试批次、运行排考、人工调整、保存版本、发布、回滚、导出数据等。

审计日志应包含操作人、操作时间、操作类型、业务对象、变更摘要和结果状态。

### FR-15 测试数据生成器

系统应提供可复现的测试数据生成能力，用于生成不同规模的课程、教师、学生群体、考场、时间段和考试任务。生成器应支持固定随机种子，保证测试结果可复现。

## 9. 非功能性需求

### 9.1 性能需求

系统应支持小规模和中规模数据的交互式排考演示。小规模数据应在数秒内完成排考；中规模数据应在可接受等待时间内输出结果；大规模数据允许以批处理方式运行，并显示运行状态和统计信息。

常规查询、筛选和详情页面应保持流畅，避免因统计或冲突计算阻塞用户操作。

### 9.2 易用性需求

系统界面应围绕教务管理员的高频工作流设计，重点保证导入校验、排考运行、冲突查看、人工调整和发布流程清晰可控。

错误信息应面向业务用户，避免只显示技术异常。导入错误、排考冲突和调整冲突应尽量给出可理解的原因和修复建议。

### 9.3 安全性需求

系统应实施基于角色的访问控制，避免教师或学生访问未发布方案、其他教师的敏感安排或管理员功能。

系统应对导入文件、查询参数、登录状态和关键写操作进行校验。关键操作应记录审计日志。

### 9.4 可维护性需求

系统应采用清晰的模块边界，将认证权限、基础数据、排考算法、冲突检测、方案版本、统计分析和审计日志分离。排考算法应与 Web 接口解耦，便于独立测试和替换。

重要设计决策应以文档形式记录，便于后续生成课程报告。

### 9.5 兼容性需求

系统应优先支持 WSL/Linux 本地开发环境和现代桌面浏览器。前端应适配常见笔记本和桌面显示器。移动端不作为第一版重点，但查询页面应避免完全不可用。

### 9.6 可测试性需求

系统应为核心排考逻辑、冲突检测、导入校验、权限控制和方案版本管理提供自动化测试。测试数据生成器应支持固定随机种子，以便复现算法测试和课程报告中的效果对比。

## 10. 数据边界

第一版应围绕以下核心数据对象建模：

- 用户、角色、权限。
- 院系、专业、班级或学生群体。
- 课程和考试任务。
- 教师和教师不可用时间。
- 教学楼、考场、考场类型、设备标签。
- 考试批次和考试时间段。
- 排考方案、排考安排、方案版本。
- 约束配置、冲突记录、算法运行记录。
- 操作日志、导入记录、发布记录。

课程报告要求至少 3 张有关联关系的数据表；本项目实际设计应显著超过该最低要求，后续数据库设计阶段应给出完整 E-R 图和字段级表结构。

## 11. 算法边界

排考算法是本项目的技术核心，但第一版算法边界应保持可验证：

1. 必须完整处理硬约束冲突。
2. 必须输出软约束评分或评分明细。
3. 必须支持无法生成可行方案时的冲突解释。
4. 必须记录运行时间、尝试次数或迭代次数等统计信息。
5. 必须支持小、中、大三类测试数据的效果对比。
6. 必须允许人工调整后重新检测冲突和评分变化。

第一版可以采用启发式搜索、约束规划或混合策略。技术选型应以后续算法原型验证为准，不在本文档中锁死具体算法库。

## 12. 优先级划分

### 12.1 P0 必须完成

- 三类核心角色与权限边界。
- 基础数据、考试任务、考场、教师、时间段建模。
- 自动排考核心流程。
- 硬约束冲突检测。
- 排考结果查看。
- 人工调整与调整后校验。
- 方案发布后教师和学生查询。
- 用例图、用例规约、业务流程、非功能需求材料。

### 12.2 P1 重点完成

- 数据导入导出。
- 软约束评分和解释。
- 方案版本管理与回滚。
- 教师不可用时间。
- 教师工作量统计。
- 考场占用视图。
- 操作日志与审计。
- 测试数据生成器。

### 12.3 P2 加分完成

- 版本差异对比。
- 多维统计看板。
- 大规模数据性能评估。
- 算法参数配置界面。
- 更完整的变更通知。
- 查询页面响应式优化。

## 13. 验收标准

需求阶段完成后，应满足以下验收标准：

1. 能清楚说明系统服务的业务范围和不服务的边界。
2. 能清楚说明教务管理员、教师、学生三类角色的权限差异。
3. 能将核心业务流程转化为用例图和用例规约。
4. 能支撑后续数据库 E-R 图、类图、时序图和测试方案设计。
5. 能支撑真实开发中的模块划分、接口设计和算法原型验证。
6. 能覆盖课程对带算法业务系统的专项要求。
