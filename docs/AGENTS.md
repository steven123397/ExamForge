# docs 文档规则

## 1. 适用范围

本文件适用于 `docs/` 下所有文档。文档应服务于课程设计、项目协作和后续实现，不把短期执行状态混入长期设计文档。

## 2. 目录职责

- `background/`：外部输入和需求背景，例如课程要求、正式需求分析、系统边界、可行性分析。
- `design/`：长期设计事实，例如总体架构、技术选型、数据合同、算法方案、接口边界。
- `plan/`：短期可执行计划，例如 agent 分工、阶段任务、验证命令、交付清单。
- `status/`：当前状态事实，例如已完成内容、最新风险、下一步。

## 3. 计划生命周期

活动计划可以直接放在 `docs/plan/` 根下。计划完成后：

1. 将完成摘要追加到 `docs/plan/history_plan.md`。
2. 摘要应包含日期、原计划、完成提交或证据、完成内容、验证结果和后续影响。
3. 删除对应活动计划文件。
4. 更新 `docs/status/project_status.md`。
5. 如文档清单变化，同步更新 `docs/index.md`。

不使用 `docs/plan/archive/` 这类专门归档目录。

## 4. 写作规则

中文技术文档使用全角中文标点；英文缩写、命令、路径、代码标识使用反引号。文档标题应直接说明主题，避免把执行状态写进标题。

设计文档只写相对稳定的决策和边界；计划文档写任务清单和验证命令；状态文档写当前事实和下一步。

## 5. 修改检查

正式文档变更后至少检查：

```bash
git diff --check
```

如果新增、删除或重命名文档，同步检查 `docs/index.md` 是否仍能反映当前文档体系。
