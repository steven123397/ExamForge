# 高校智能排考与考务资源调度系统第三版实现内容设计

## 1. 文档定位

本文档定义 ExamForge 第三版的实现边界、架构取舍和验收标准。它承接 `docs/design/总体设计与技术选型.md`、`docs/design/企业级全栈平台第一阶段设计.md` 和 `docs/design/第二版实现内容设计.md`，以当前 `main` 的第二版闭环为基线。

第三版不再优先堆叠演示功能，而是处理继续演进前最影响上限的三类问题：

1. Web 运营台单文件过大，后续功能开发和回归验证成本过高。
2. 调度器的软约束评分尚未真正进入求解目标，算法输出仍主要是硬约束可行解。
3. PostgreSQL 路径、迁移和跨模块合同缺少足够自动化验证，容易出现内存仓储测试通过但真实持久化路径回归的问题。

第三版的核心价值是把第二版的“可演示闭环”升级为“可持续演进的工程基线”。

## 2. 当前基线

第三版以当前 `main` 已有能力为起点：

1. `apps/scheduler/` 已实现 Python OR-Tools CP-SAT 排考、预检、冲突解释、软约束评分和报告整理，并通过 JSON CLI 暴露给 API。
2. `apps/api/` 已实现 Fastify API、Bearer token 演示鉴权、同步排考、异步排考作业、草稿治理、发布回滚、审计、通知预览和 CSV 导出。
3. `packages/db/` 已实现 PostgreSQL schema、手写 SQL 迁移、`schema_migrations` 迁移状态表、seed 和 PostgreSQL 仓储。
4. `packages/shared/` 已承载 Zod 领域合同、API DTO、排考输入输出、草稿、作业、发布查询和基础数据类型。
5. `apps/web/` 已实现单页运营台，覆盖基础数据管理、排考运行、草稿工作台、矩阵拖拽、建议应用、锁定/解锁、局部再平衡、发布确认、通知预览和 CSV 导出。

当前仍存在的代码审查问题以 `docs/status/code_review_status.md` 为准。第三版设计将其中与架构演进直接相关的问题纳入路线，但不把所有待解决项都作为第三版硬性交付。

## 3. 第三版目标

第三版实现目标如下：

1. 拆分 Web 运营台，建立 API client、业务面组件和数据加载层边界。
2. 引入 TanStack Query 管理服务端状态，减少组件内重复的 `fetch + useState + useEffect` 模式。
3. 让调度器至少一部分软约束真正参与 CP-SAT 优化目标，避免只在求解后做后置评分。
4. 补齐 PostgreSQL 集成测试、迁移验证和调度器 CLI 契约测试。
5. 提取 API 层可复用 service 函数，降低路由文件承载的业务逻辑。
6. 为后续关联表重构、真实队列、实时进度和算法服务化留下清晰边界。

## 4. 第三版范围

### 4.1 纳入第三版

第三版纳入以下内容：

- `apps/web/app/operations-console.tsx` 拆分为多个业务面组件。
- `apps/web/lib/api-client.ts` 或等价 API client 层。
- TanStack Query 作为 Web 服务端状态管理方案。
- 异步作业、已发布查询、基础数据、运行历史、草稿工作台等业务面的组件拆分。
- 调度器软约束目标函数改造，至少覆盖可由考试、考场、时间段决策变量直接表达的惩罚项。
- PostgreSQL 集成测试套件，覆盖草稿发布、审计写入、作业状态迁移和关键查询。
- 迁移从零执行验证。
- `scheduler-client.ts` 与 Python CLI 的契约测试。
- API service 提取，优先处理发布受众、通知预览和 CSV 导出。
- 审计检索 API 的基础过滤能力。

### 4.2 选做内容

以下内容可以在第三版时间允许时纳入，但不作为必须完成项：

- 将教师分配纳入 CP-SAT 联合求解，或改为更强的负载感知匹配算法。
- 在调度器输入合同中新增 `fixed_assignments`，支持真正的固定安排增量重排。
- 将 `exam_tasks.student_group_ids`、`scheduled_exams.teacher_ids`、`draft_scheduled_exams.teacher_ids` 拆为关联表。
- 为 `rooms.buildingId` 增加 `buildings` 表或 API 层合法性校验。
- Web 面板状态同步到 URL query 参数。
- 子面板级 error boundary。
- 批量导入逐行错误报告。

### 4.3 不纳入第三版

以下内容暂不纳入第三版：

- 将 Python scheduler 服务化为 FastAPI。
- 引入 BullMQ、Redis 或完整外部任务队列。
- 用 SSE 或 WebSocket 替代当前轮询。
- 引入完整用户、会话、refresh token 和用户管理体系。
- 引入 OpenAPI 生成和 SDK 生成链路。
- 生产级多租户、多批次并行运营和复杂权限矩阵。

这些方向不是错误方向，但当前阶段投入产出比不高。它们应在第三版建立更清晰的 Web、API、测试和算法边界后再重新评估。

## 5. 总体架构演进

第三版保持现有四层边界：

```text
apps/web
  Next.js 运营台，负责交互、展示和状态编排
  |
  | REST / shared DTO
  v
apps/api
  Fastify API，负责鉴权、业务编排、仓储调用和 scheduler 调用
  |
  | JSON stdin/stdout
  v
apps/scheduler
  Python CP-SAT 调度器，负责排考算法、预检、冲突解释和报告
  |
  | Drizzle / SQL
  v
packages/db
  PostgreSQL schema、迁移、seed 和数据库客户端
```

第三版不改变部署形态，不新增独立算法服务和外部队列。架构演进重点是把现有单文件和隐式合同拆成更小的、可测试的模块。

## 6. Web 运营台设计

### 6.1 拆分目标

当前 `operations-console.tsx` 承载 API 调用、角色状态、多个业务面状态和全部 UI。第三版应将它降级为页面编排层，主要职责是：

- 组合业务面组件。
- 维护当前角色和当前激活面板。
- 提供少量跨面板共享上下文。

业务逻辑、请求封装、轮询和复杂表单状态不应继续堆在页面组件内。

### 6.2 建议目录

```text
apps/web/
├── app/
│   └── operations-console.tsx
├── features/
│   ├── async-jobs/
│   ├── draft-workspace/
│   ├── published-schedule/
│   ├── reference-data/
│   ├── run-history/
│   └── teacher-unavailable/
├── lib/
│   ├── api-client.ts
│   └── query-keys.ts
└── components/
    └── shared/
```

`features/` 下每个业务面可以包含自己的组件、局部 hook 和展示辅助函数。跨业务面的通用 UI 放入 `components/shared/`，不要把业务知识放入通用组件。

### 6.3 API client

`api-client.ts` 负责：

- 读取 `NEXT_PUBLIC_API_BASE_URL`。
- 注入 Bearer token 或角色演示 token。
- 统一处理 JSON 响应、HTTP 错误和 Zod 类型边界。
- 封装业务 API 方法，例如 `listScheduleRuns()`、`createScheduleJob()`、`publishDraft()`。

组件不得直接拼接 API URL；TanStack Query 的 query function 应调用 API client。

### 6.4 状态管理

第三版引入 TanStack Query 管理服务端状态：

- 排考运行列表。
- 审计事件。
- 草稿列表和草稿详情。
- 异步作业列表。
- 已发布方案。
- 通知预览。
- 基础数据。

Zustand 暂缓引入。只有当跨面板本地交互状态明显膨胀时，再用 Zustand 管理非服务端状态。

### 6.5 拆分顺序

Web 拆分按风险从低到高推进：

1. 提取 API client 和 query keys。
2. 拆出 `AsyncJobPanel`。
3. 拆出 `PublishedScheduleViewer`。
4. 拆出 `ReferenceDataManager` 和 `TeacherUnavailablePanel`。
5. 拆出 `ScheduleRunHistory`。
6. 最后拆出 `DraftWorkspace`。

每完成一个业务面拆分，至少运行 `npm run typecheck`、`npm run build` 和 `npm run test:e2e`。

## 7. 调度器算法设计

### 7.1 当前问题

当前 CP-SAT 求解器主要寻找硬约束可行解。`calculate_score()` 在求解后计算软约束惩罚，但这些惩罚没有驱动 CP-SAT 的目标函数。`total_score` 也仍主要体现硬冲突是否存在。

这意味着系统可以解释软约束得分，但不能保证求解器主动优化软约束。

### 7.2 第三版必做改造

第三版应在 `solve_schedule()` 中引入加权目标函数：

```text
Minimize(
  连续考试惩罚 * weight_a
  + 考场利用率惩罚 * weight_b
  + 考试分布惩罚 * weight_c
  + 其他可由 room-slot 决策直接表达的惩罚
)
```

建模原则：

1. 不破坏现有硬约束，硬约束仍是可行性的底线。
2. 软约束权重来自现有 `constraint_profile` 或明确的默认权重。
3. 求解器应设置合理时间限制，超时返回当前最优可行解。
4. 输出中保留 `score_breakdown`，并让其与求解目标有可解释映射。
5. 测试需要证明权重变化会影响解的选择或评分结果。

### 7.3 教师分配策略

当前教师分配在 room-slot 求解后由 `_assign_teachers` 贪心完成。第三版不强制一次性把教师变量纳入 CP-SAT 联合求解，但设计上必须明确边界：

- 如果第三版声称支持“教师工作量优化”，则需要将教师分配纳入 CP-SAT 目标，或至少改为负载感知匹配算法。
- 如果只完成 room-slot 维度软约束优化，则文档和 UI 不应夸大教师工作量平衡能力。

### 7.4 增量重排边界

当前局部再平衡是仓储层基于候选建议的启发式调整，不是真正的调度器级增量重排。第三版若要实现真正增量重排，应扩展 shared 合同：

```typescript
interface FixedAssignment {
  exam_task_id: string;
  room_id: string;
  time_slot_id: string;
  teacher_ids: string[];
}
```

调度器在建模时将固定安排转为固定变量，只求解未锁定或冲突考试。该能力列为选做，不阻塞第三版主线。

## 8. API 层设计

### 8.1 Service 提取

第三版不重写 API 框架，也不强制复杂分层。只提取已经明显可复用、可独立测试的业务函数：

- `buildPublishedScheduleAudience`
- `buildPublishedScheduleNotifications`
- `buildPublishedScheduleCsv`
- 后续审计过滤与导出辅助函数

建议目录：

```text
apps/api/src/
├── app.ts
├── services/
│   ├── published-schedule-service.ts
│   └── audit-service.ts
└── repository.ts
```

路由层继续负责 HTTP 入参、鉴权和响应码；service 负责业务组合；repository 负责数据访问。

### 8.2 审计检索

第三版为 `GET /api/audit-events` 增加基础过滤：

| 参数 | 说明 |
| --- | --- |
| `entityType` | 按实体类型过滤 |
| `entityId` | 按实体 ID 过滤 |
| `actor` | 按操作者过滤 |
| `since` | 查询某时间之后的事件 |
| `until` | 查询某时间之前的事件 |

列表接口应考虑分页，但第三版可以先实现有限过滤，再把全局分页作为后续平台化任务。

### 8.3 异步作业边界

第三版不引入 Redis 队列。当前 `schedule_jobs` 表继续作为作业状态源，但应补强以下能力：

- 作业创建时记录所属批次，避免后续多批次场景无法归属。
- API 启动时识别历史 `running` 作业，并标记为失败或可恢复状态。
- 作业进度不要长期硬编码为固定中间值；若调度器暂不支持真实进度，也应明确展示为“排队 / 运行中 / 完成 / 失败”状态。

## 9. 数据库与迁移设计

### 9.1 第三版必做

第三版必须补齐数据库验证，而不是先做大规模 schema 重构：

1. PostgreSQL 集成测试覆盖核心业务路径。
2. 迁移从空库顺序执行验证。
3. Drizzle schema 与 SQL 迁移保持同步检查。
4. seed 后基础数据和 API 查询结果一致。

### 9.2 关联表重构

以下 JSONB 数组是后续查询能力的主要限制：

| 当前字段 | 后续目标 |
| --- | --- |
| `exam_tasks.student_group_ids` | `exam_task_student_groups(exam_task_id, student_group_id)` |
| `scheduled_exams.teacher_ids` | `scheduled_exam_invigilators(scheduled_exam_id, teacher_id)` |
| `draft_scheduled_exams.teacher_ids` | `draft_exam_invigilators(draft_scheduled_exam_id, teacher_id)` |
| `teachers.unavailable_slot_ids` | `teacher_unavailable_slots(teacher_id, time_slot_id)` |

第三版可以先完成 `exam_tasks.student_group_ids` 和监考教师两类核心关系；`allowed_slot_ids`、`required_equipment_tags`、`rooms.equipment_tags` 可以暂缓。

该重构应在 PostgreSQL 集成测试落地后进行，避免迁移只在内存仓储路径中被间接验证。

### 9.3 `rooms.buildingId`

`rooms.buildingId` 当前是自由文本。第三版有两种选择：

1. 如果楼栋只用于展示，API 层校验非空和格式即可。
2. 如果楼栋会参与“减少跨楼栋移动”等软约束，应新增 `buildings` 表并建立外键。

第三版正式实现前需要明确楼栋字段的业务语义。

## 10. 测试与验证设计

### 10.1 新增测试

第三版新增测试应覆盖：

- Web 拆分后的 E2E 回归，至少保留草稿锁定/解锁、矩阵拖拽、异步作业和通知预览路径。
- TanStack Query/API client 的错误处理单元测试。
- API service 单元测试，覆盖通知、CSV 和发布受众。
- PostgreSQL 集成测试，覆盖草稿发布、审计写入、作业状态迁移和关键查询。
- 迁移从空库执行测试。
- 调度器 CLI 契约测试，确保 Python 输出结构符合 shared 合同。
- 调度器软约束目标测试，证明软约束权重能影响求解结果或评分。

### 10.2 默认验证矩阵

第三版完成后至少运行：

```bash
npm run typecheck
LOG_LEVEL=silent npm test
npm run build
npm run test:scheduler
npm run test:e2e
git diff --check
```

涉及数据库迁移或 PostgreSQL 仓储时，还必须运行新增 PostgreSQL 集成测试和迁移验证。

## 11. 推荐实施顺序

### 11.1 第一阶段：Web 拆分

目标：解除第三版继续开发前最大的维护风险。

1. 提取 API client。
2. 引入 TanStack Query。
3. 拆出异步作业和已发布查询面板。
4. 拆出基础数据和教师不可用面板。
5. 拆出运行历史和审计面板。
6. 拆出草稿工作台。

### 11.2 第二阶段：测试基线补强

目标：让后续算法和数据库改动有真实回归保护。

1. 建立 PostgreSQL 集成测试入口。
2. 增加迁移从空库执行验证。
3. 增加 scheduler CLI 契约测试。
4. 把 API service 提取与单元测试一并完成。

### 11.3 第三阶段：软约束入模

目标：让算法从“可行解生成”升级为“带目标函数的优化求解”。

1. 将可表达的软约束转换为 CP-SAT 目标项。
2. 增加权重变化回归测试。
3. 调整报告和评分说明，确保 UI 展示不夸大算法能力。
4. 评估教师分配是否进入联合求解或负载感知匹配。

### 11.4 第四阶段：数据模型选做重构

目标：为更复杂查询和真实业务扩展打基础。

1. 迁移考试任务与学生群体关联表。
2. 迁移监考教师关联表。
3. 补充楼栋语义和校验。
4. 更新 PostgreSQL 仓储和 shared 合同。

## 12. 验收标准

第三版完成后应满足以下标准：

1. `operations-console.tsx` 只承担页面编排职责，主要业务面已拆入 `features/`。
2. Web API 调用通过统一 API client，不再在业务组件中散落拼接 URL。
3. 服务端状态由 TanStack Query 管理，异步作业轮询封装为可复用 hook 或 query 配置。
4. 调度器软约束至少有一组目标项进入 CP-SAT `Minimize()`。
5. 调度器测试能证明软约束权重会影响解的质量或评分。
6. API 通知、CSV、发布受众逻辑可以脱离 Fastify 实例独立测试。
7. PostgreSQL 集成测试覆盖草稿发布、审计写入和作业状态迁移。
8. 迁移可以在空库上顺序执行，并与 Drizzle schema 保持一致。
9. E2E 仍覆盖第二版核心交互路径。
10. `npm run typecheck`、`LOG_LEVEL=silent npm test`、`npm run build`、`npm run test:scheduler`、`npm run test:e2e` 和 `git diff --check` 全部通过。

## 13. 开放问题

第三版正式拆计划前需要确认：

1. 第三版的主要交付对象是课程报告增强、技术展示，还是面向真实教务场景的产品化演示？
2. 楼栋字段是否会参与软约束，还是只作为展示字段？
3. 是否要在第三版支持多考试批次切换？如果需要，Web 拆分时必须提前设计批次上下文。
4. 教师工作量优化是否是第三版必须展示的算法能力？如果是，教师分配需要进入更强的优化模型。
5. PostgreSQL 集成测试是否可以依赖本机或 CI 的 Docker？如果不能，需要选择外部测试库或模拟方案。

## 14. 与后续版本的边界

第三版完成后，后续企业化阶段再评估：

- Python scheduler 服务化。
- Redis/BullMQ 或其他可靠任务队列。
- SSE/WebSocket 实时进度。
- 真实用户管理、会话生命周期和权限矩阵。
- OpenAPI/SDK 生成。
- 多租户、多批次运营和部署监控。

第三版的判断标准不是“看起来更企业级”，而是每个新增能力都能降低后续演进风险，或者显著提升算法结果的可信度。
