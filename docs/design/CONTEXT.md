# ExamForge 项目规范术语

本文档只维护 ExamForge 的规范语言、领域边界和容易漂移的概念关系。它不是实现规格、阶段计划、当前状态或决策记录；版本特有的不变量和取舍仍写入对应的 `docs/design/` 文档。

## 考务对象

### 考试批次

一组在同一业务周期内统一管理参与者语义、数据版本和排考范围的考试任务集合。考试批次决定参与者模式和快照边界，不等同于单场考试。

- 避免用词：考试项目、考试集合（未明确批次边界时）

### 考试任务

一门课程或一次具体考试活动的排考对象，包含考试要求、参与者范围和排考所需的资源约束。考试任务属于考试批次。

- 避免用词：考试批次（两者粒度不同）

### 考试场次

考试任务最终安排到一个时间段和资源组合后的实际考试事件。没有完成安排的考试任务不能称为考试场次。

## 参与者语义

### 学生群体

用于组织、筛选、统计和通知范围的学生集合。群体可以继续承担兼容职责，但在 `enrollments` 参与者模式下不自动代表学生之间存在考试冲突。

- 避免用词：班级（除非明确指教学班）、学生集合（缺少业务边界）

### 脱敏学生

用于排考约束的最小化学生参与者标识，不包含姓名、证件号、联系方式等真实个人信息。脱敏学生可以有一个主要学生群体，但跨专业、选修和重修关系通过报名表达。

- 避免用词：学生账户、学籍档案（超出当前范围）

### 考试报名

表示某个脱敏学生是否参加某个考试任务的业务事实。报名来源可以用于解释和审计，但不改变同一学生参加两场时间重叠考试时的冲突强度。

- 避免用词：选课（报名是排考事实，不是完整教务选课关系）

### 参与者模式

考试批次级的冲突语义选择。`groups_only` 使用学生群体近似互斥，`enrollments` 使用有效报名生成个体冲突；同一批次、作业快照和运行不能静默混用两种模式。

### 参与者数据状态

参与者事实对当前批次是否可用于排考的状态：未要求、草稿或完整。数据不完整时不能把覆盖率百分比当作完整性证明，也不能开始会产生正式结果的求解或发布。

## 排考与版本

### 参与者快照

在排考作业或运行创建时冻结的参与者模式、报名版本、派生冲突关系和摘要。快照是该作业、运行和发布版本的解释边界，当前报名变化不能改写历史快照。

### 排考作业

一次由 API 接收并可异步投递、重试、取消和审计的排考请求。作业负责协调执行，不等同于求解器已经实际完成的排考运行。

### 排考运行

一次使用冻结输入调用调度算法并产生结果、诊断、评分和统计的实际执行。运行必须能追溯到唯一作业、参与者快照和策略版本。

### 方案草稿

可以继续编辑、校验和生成新排考结果的工作版本。草稿不是对教师或学生可见的正式安排，也不能绕过锁定、终态和发布资格检查。

### 发布版本

经过发布资格校验、对外提供查询的不可变排考方案版本。发布版本只能通过新的版本、回滚或受控流程改变，不应由浏览器直接改写历史结果。

### 回滚

将一个已经存在且仍满足当前发布合同的历史发布版本重新设为当前对外版本。回滚不是复制并编辑旧数据，也不是恢复当前报名或策略的隐式新结果。

### 策略版本

一组约束开关、软约束权重和求解参数的不可变配置版本。排考运行和发布版本必须记录实际采用的策略版本，不能只引用当前默认策略。

## 可追溯性

### 审计事件

对创建、修改、提交、运行、发布、回滚、权限和失败等关键业务动作的结构化记录。审计事件保存足以追踪责任、版本和结果的摘要，不保存不必要的全量报名名单或敏感个人信息。

### 受众范围

用户可以读取或维护的教师、学生群体等业务范围。受众范围由服务端身份和关联事实决定，不能由 URL、查询参数或请求体临时切换。

### 设计决策

在相关 design 文档中记录的已确认取舍，包括结论、理由与证据、被拒绝的替代方案、后果和重新评估条件。设计决策不是单独的 ADR 文件。
