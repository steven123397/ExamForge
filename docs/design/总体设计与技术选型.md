# 高校智能排考与考务资源调度系统总体设计与技术选型

## 1. 文档定位

本文档是 ExamForge 项目的总体设计与技术选型初稿，承接以下背景与需求文档：

- `docs/background/软件工程课程设计大作业要求-2026.md`
- `docs/background/wsl_codex_project_brief.md`
- `docs/background/正式需求分析与系统边界.md`

本文档用于沉淀长期有效的技术边界、架构取舍、算法方案、数据模型方向和阶段路线。短期执行清单应放入 `docs/plan/`，当前进展与风险应放入 `docs/status/`。

## 2. 设计目标

本系统不是单一排考脚本，也不是只用于课程报告的虚构 CRUD 系统，而是一个以约束调度为核心的 Web 化考务管理平台。

系统设计目标包括：

1. 支持教务管理员完成基础数据维护、考试任务创建、自动排考、人工调整、发布和回滚。
2. 支持教师维护不可用时间、查看监考安排和个人监考工作量。
3. 支持学生或只读用户查询已发布考试安排和变更通知。
4. 支持硬约束冲突检测、软约束评分、冲突解释和算法运行统计。
5. 支持课程报告所需的架构图、模块图、E-R 图、类图、流程图、测试方案和维护规划素材。
6. 在课程周期内优先做出可验证、可演示、可截图、可写报告的系统，而不是追求生产级复杂度。

## 3. 设计原则

### 3.1 激进但分阶段

项目允许采用相对激进的工程方案，例如约束求解器、前后端分离、独立算法服务、队列、实时进度推送和可视化排考矩阵。但这些能力不能一次性全部作为第一阶段硬依赖。

第一优先级是验证排考核心：数据模型、约束模型、算法原型、冲突检测和可解释输出。工程基础设施应服务于这个目标，而不是先行占据大部分时间。

### 3.2 算法核心与 Web 系统解耦

排考算法应与 Web API 解耦。算法服务输入结构化数据，输出排考结果、冲突列表、评分明细和运行统计。这样可以独立测试算法，也便于后续替换求解策略。

### 3.3 真实约束优先于界面复杂度

排考系统的技术价值主要来自约束建模、冲突解释、人工调整后校验和版本管理。前端界面应围绕这些能力组织，而不是用大而全的后台页面稀释核心展示。

### 3.4 报告材料在开发过程中同步沉淀

课程报告要求覆盖完整软件生命周期。需求、架构、数据模型、算法流程、测试用例和维护规划应在开发中持续沉淀，避免最后集中补写导致材料与实现脱节。

## 4. 总体技术选型

### 4.1 推荐技术栈

| 层次 | 推荐选型 | 定位 | 阶段 |
| --- | --- | --- | --- |
| 前端 | Next.js + TypeScript | 管理后台、查询页面、排考可视化 | P1 |
| UI | shadcn/ui + Tailwind CSS | 数据密集型后台界面 | P1 |
| 客户端状态 | Zustand | 局部交互状态，例如筛选、拖拽草稿 | P1 |
| 服务端状态 | TanStack Query | API 数据缓存、加载状态和刷新 | P1 |
| 拖拽交互 | dnd-kit | 人工调整排考结果 | P1/P2 |
| 主 API | Node.js + Fastify + TypeScript | 认证、业务 API、数据校验、审计 | P1 |
| ORM | Drizzle ORM | 类型安全 schema、迁移和 SQL 可见性 | P1 |
| 数据库 | PostgreSQL | 关系数据、事务、索引、查询统计 | P1 |
| 算法服务 | Python + FastAPI | 封装 OR-Tools 求解器 | P0/P1 |
| 求解器 | Google OR-Tools CP-SAT | 硬约束满足与软约束优化 | P0/P1 |
| 队列 | BullMQ + Redis | 长耗时排考任务异步执行 | P2 |
| 实时推送 | WebSocket 或 SSE | 排考进度和结果通知 | P2 |
| 部署 | Docker Compose | 本地和云端演示环境 | P1/P2 |
| 测试 | pytest + Vitest + Playwright | 算法、业务逻辑、端到端验证 | P0-P2 |

### 4.2 阶段化落地策略

技术栈按阶段启用：

1. **算法原型阶段（P0）：** 先建立 Python 算法模块、测试数据生成器和 pytest 测试，不依赖完整 Web 系统。
2. **核心平台阶段（P1）：** 引入 PostgreSQL、Fastify API、Next.js 管理端，实现基础数据、考试任务、排考运行、结果查看和发布查询。
3. **体验增强阶段（P2）：** 引入 BullMQ、Redis、WebSocket/SSE、拖拽矩阵、版本对比和大规模性能评估。

该策略保留激进目标，但避免在第一步同时承担全栈基础设施和算法建模风险。

## 6. 排考算法设计

### 6.1 算法定位

排考问题属于约束满足与优化问题。系统需要在满足硬约束的前提下尽量优化软约束评分。

硬约束违反意味着方案不可发布。软约束违反不一定阻止发布，但应计入评分和解释。

### 6.2 约束输入

算法输入包括：

- 考试任务：课程、学生群体、人数、时长、考试要求。
- 考场：容量、楼栋、校区、类型、设备标签。
- 时间段：日期、开始时间、结束时间、是否可用于考试。
- 教师：可监考状态、不可用时间、所属院系、已有工作量。
- 约束配置：硬约束开关、软约束权重、算法时限。

### 6.3 CP-SAT 建模口径

正式实现不采用单纯的 `room_var` / `slot_var` 整数变量作为唯一建模方式。更稳妥的第一版建模建议使用布尔决策变量：

```python
x[exam_id, room_id, slot_id] = 1
```

其含义是：某场考试是否安排在某个考场和某个时间段。

基于该变量可以清晰表达：

- 每场考试只能选择一个 `(room, slot)`。
- 同一 `(room, slot)` 只能容纳一场考试。
- 不满足容量、设备、校区或时间窗口的组合直接禁止。
- 共享学生群体的考试不能处于同一时间段。
- 同一教师不能在同一时间段参与多场监考。
- 教师不可用时间段不能安排监考。

监考教师分配可以在第一版中拆成两步：

1. 先求解考试的考场和时间段。
2. 再基于结果分配监考教师并校验冲突。

如果算法原型证明联合求解可控，再升级为考试、考场、时间段、教师的联合模型。

### 6.4 硬约束

第一版必须覆盖以下硬约束：

- 同一学生群体同一时间段不能参加多场考试。
- 同一教师同一时间段不能监考多场考试。
- 同一考场同一时间段不能安排多场考试。
- 考场容量必须满足考试人数。
- 考场类型、校区和设备条件必须满足考试要求。
- 考试必须安排在允许的考试时间段内。
- 教师不可用时间段不得安排监考。
- 每场考试的监考人数必须满足最低要求。

### 6.5 软约束

软约束采用加权惩罚项建模。初始权重用于原型验证，后续可通过配置调整。

| 编号 | 软约束 | 初始权重 | 说明 |
| --- | --- | --- | --- |
| SC-1 | 避免同一学生群体连续考试 | 80 | 相邻时间段连续考试扣分 |
| SC-2 | 平衡教师监考工作量 | 70 | 监考次数或时长方差越大扣分越多 |
| SC-3 | 考试在周期内均衡分布 | 50 | 避免考试集中在少数日期 |
| SC-4 | 减少跨校区或跨楼栋移动 | 40 | 适用于同一学生群体或教师连续安排 |
| SC-5 | 提高考场容量利用率 | 30 | 避免大教室安排小人数考试 |
| SC-6 | 满足课程偏好时间段 | 20 | 例如公共课、机考课程偏好 |

### 6.6 冲突解释策略

冲突解释分两层实现：

1. **确定性预检：** 在调用 CP-SAT 前检查明显不可行的数据，例如考试人数超过所有可用考场容量、某学生群体考试数量超过可用时间段、教师不可用时间覆盖全部候选时段等。
2. **求解后解释：** 对未安排任务、硬约束冲突和人工调整冲突生成结构化解释。

CP-SAT 的不可行核心分析和最小冲突集可以作为 P2 增强项，不作为第一版硬承诺。第一版优先保证解释稳定、可测试、可写入课程报告。

### 6.7 规模策略

| 规模 | 目标 | 策略 |
| --- | --- | --- |
| 小规模（不超过 30 场） | 快速演示和单元测试 | 精确求解，输出完整解释 |
| 中规模（30-150 场） | 常规系统测试 | 限时求解，输出当前最优解和评分 |
| 大规模（150 场以上） | 性能展示和报告材料 | 限时求解，必要时按院系、校区或考试批次分块 |

具体阈值以后续算法原型测试结果为准。

## 7. 数据模型设计

### 7.1 核心数据实体

第一版建议围绕以下数据实体建模：

| 实体 | 说明 |
| --- | --- |
| users | 用户账户 |
| roles | 角色定义 |
| user_roles | 用户与角色关系 |
| departments | 院系 |
| courses | 课程 |
| course_instances | 课程教学班或学期实例 |
| student_groups | 学生群体，例如班级或考试群体 |
| teachers | 教师 |
| buildings | 教学楼 |
| rooms | 考场，包含容量、类型、设备标签 |
| time_periods | 考试时间段 |
| exam_batches | 考试批次 |
| exam_tasks | 考试任务 |
| exam_task_groups | 考试任务与学生群体多对多关系 |
| teacher_unavailable_periods | 教师不可用时间 |
| constraint_profiles | 约束配置与权重 |
| scheduling_runs | 排考运行记录和方案版本 |
| scheduled_exams | 排考结果 |
| invigilator_assignments | 监考安排 |
| conflict_records | 冲突记录 |
| import_records | 导入记录 |
| audit_logs | 操作审计日志 |

该模型显著超过课程要求的 3 张关联表最低标准，能支撑 E-R 图、字段级表结构和用例规约。

### 7.2 数据库约束落点

数据库层适合承载稳定且不依赖跨表计算的约束，例如：

```sql
-- 同一方案版本中，同一考场同一时间段只能安排一场考试
UNIQUE (scheduling_run_id, room_id, time_period_id)

-- 同一排考结果中，同一教师不能重复绑定
UNIQUE (scheduled_exam_id, teacher_id)

-- 软删除字段
deleted_at TIMESTAMPTZ DEFAULT NULL
```

以下约束不应写成普通 `CHECK` 跨表表达式：

- 考场容量是否大于考试人数。
- 考场设备是否满足考试任务要求。
- 教师不可用时间是否覆盖监考时间。
- 学生群体是否存在同时间考试冲突。

这些约束应放在应用层事务校验、算法预检、数据库触发器或物化字段校验中。课程报告中应明确说明“数据库约束”和“业务约束”的分工。

## 8. 前端体验设计

### 8.1 页面分区

第一版前端建议包含以下页面：

- 登录页。
- 管理员工作台。
- 基础数据管理：课程、教师、学生群体、考场、时间段。
- 考试批次与考试任务管理。
- 自动排考运行页面。
- 排考结果列表视图。

- 冲突解释面板。
- 人工调整页面。
- 方案版本、发布与回滚页面。
- 教师个人监考安排页面。
- 学生考试安排查询页面。
- 审计日志页面。

### 8.2 排考矩阵视图

排考矩阵是本项目的核心展示界面之一。推荐以考场为行、时间段为列展示：

```text
        时段 1      时段 2      时段 3      时段 4
教 101  高数 A      空          线代 B      空
教 102  英语四级    物理 A      空          高数 B
教 103  空          化学 B      空          英语六级
机房 1  程序设计    空          数据库      空
```

矩阵视图需要支持：

- 按批次、日期、楼栋、校区筛选。
- 高亮容量不足、教师冲突、学生群体冲突等问题。
- 查看考试卡片详情。
- 在 P1/P2 阶段支持拖拽调整。

### 8.3 进度推送

排考运行耗时较长时，前端应展示运行状态。第一版可以先用轮询实现，后续再升级为 WebSocket 或 SSE。

```json
{
  "status": "running",
  "progress": 67,
  "bestScore": 1240,
  "elapsedSeconds": 31
}
```

## 9. 后端与接口设计

### 9.1 API 服务职责

Fastify API 服务负责：

- 用户认证与权限判断。
- 基础数据 CRUD。
- 导入导出与数据校验。
- 考试批次和考试任务管理。
- 排考运行请求编排。
- 调用 Python 算法服务。
- 保存排考结果、冲突记录、版本信息和审计日志。
- 向前端提供查询、筛选和统计接口。

### 9.2 算法服务接口

算法服务保持窄接口，避免耦合 Web 业务对象。

核心接口建议：

```text
POST /solve
POST /validate-adjustment
POST /explain-conflicts
```

`POST /solve` 输入经过 API 服务整理后的纯调度数据，输出：

- `status`：`feasible`、`partial`、`infeasible`、`error`。
- `assignments`：考试、考场、时间段、监考教师安排。
- `hardConstraintViolations`：硬约束冲突。
- `softScore`：软约束总分和明细。
- `statistics`：运行时长、候选数量、搜索状态。

## 10. 测试设计

### 10.1 算法测试

算法测试是第一优先级。至少覆盖：

- 小规模可行数据能生成无硬冲突方案。
- 容量不足时能输出可解释冲突。
- 学生群体重叠时不会安排同时间考试。
- 教师不可用时间不会被安排监考。
- 软约束权重变化会影响评分。
- 固定随机种子生成的数据可复现。

### 10.2 后端测试

后端测试覆盖：

- 权限边界。
- 导入校验。
- 考试任务创建。
- 排考运行记录保存。
- 方案发布和回滚。
- 审计日志记录。

### 10.3 前端与端到端测试

前端测试优先覆盖关键路径：

- 管理员登录。
- 创建考试批次。
- 导入或创建考试任务。
- 运行排考。
- 查看冲突和排考结果。
- 发布方案。
- 教师和学生查询已发布安排。

## 11. 部署与维护设计

### 11.1 本地开发

开发环境优先支持 WSL/Linux。推荐顺序：

1. 本地直接运行 Python 算法原型和 pytest。
2. 本地运行 PostgreSQL 或 Docker PostgreSQL。
3. 启动 API 和 Web。
4. 后续用 Docker Compose 编排全栈服务。

### 11.2 云端演示

云端部署可采用 Docker Compose：

```yaml
services:
  postgres:
  redis:
  api:
  scheduler:
  web:
  nginx:
```

Redis、队列和 Nginx 属于 P2 增强。若课程周期紧张，本地演示和录屏可以优先于完整云端部署。

### 11.3 维护规划映射

| 维护类型 | 本项目方案 |
| --- | --- |
| 纠错性维护 | 通过测试、日志和审计记录定位并修复缺陷 |
| 适应性维护 | 通过配置化约束和模块化算法服务适配不同考试规则 |
| 完善性维护 | 逐步增强拖拽调整、版本对比、统计看板和通知能力 |

## 13. 阶段路线

### 13.1 P0：算法与模型原型

目标：证明核心排考问题可建模、可求解、可解释。

交付物：

- 测试数据生成器。
- Python 排考原型。
- 硬约束检测。
- 软约束评分。
- 小规模和中规模测试样例。
- 初始数据模型与约束模型文档。

### 13.2 P1：核心 Web 平台

目标：形成可演示的完整业务闭环。

交付物：

- PostgreSQL schema 和迁移。
- Fastify API。
- Next.js 管理后台。
- 基础数据、考试任务、自动排考、结果查看。
- 方案发布后教师和学生查询。
- 基础审计日志。

### 13.3 P2：体验和工程增强

目标：提升展示效果、可维护性和工程完整度。

交付物：

- 拖拽矩阵调整。
- WebSocket 或 SSE 进度推送。
- BullMQ 异步任务队列。
- 版本差异对比。
- 大规模数据性能评估。
- Docker Compose 一键启动。

## 14. 主要风险与处理策略

| 风险 | 影响 | 处理策略 |
| --- | --- | --- |
| CP-SAT 建模复杂度高 | 核心算法延误 | 先做 Python 原型，只覆盖 P0 约束，再逐步扩展 |
| 全栈工程范围过大 | 页面和服务无法按期闭环 | P0/P1/P2 分阶段，不把队列和 WebSocket 作为 P0 前置 |
| 冲突解释过度理想化 | 报告和演示难兑现 | 先做确定性预检和规则级解释，复杂 MCS 作为增强项 |
| 数据库约束跨表表达困难 | schema 设计不准确 | 区分数据库约束、应用校验和算法校验 |
| 课程报告材料滞后 | 最终报告赶工 | 每个阶段同步沉淀图表、测试用例和设计说明 |
| 前端矩阵交互复杂 | UI 开发超时 | 先做只读矩阵和列表视图，再做拖拽调整 |

## 15. 当前结论

推荐采用“TypeScript Web 平台 + Python OR-Tools 算法服务 + PostgreSQL 数据库”的总体方向，但按 P0/P1/P2 分阶段落地。

第一步不应直接搭完整全栈，而应先产出 `docs/design/数据模型与排考约束模型.md` 和 `docs/plan/排考算法原型计划.md`，随后用最小 Python 原型验证核心约束与测试数据生成能力。验证通过后，再进入 Web 平台 scaffold。
