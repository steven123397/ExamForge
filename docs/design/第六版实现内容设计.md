# 高校智能排考与考务资源调度系统第六版实现内容设计

## 1. 文档定位

本文档定义 ExamForge 第六版的长期设计边界。第六版主题是“参与者语义真实性与可复现算法证据”：在保留现有学生群体口径和可靠调度架构的前提下，引入最小化的脱敏学生与考试报名模型，使系统能够准确处理跨专业选课、重修和个别学生考试重叠，并以不可变输入快照、稳定诊断和固定实验样例证明结果。

本文档不是活动实施计划，不包含具体任务拆分、执行命令或当前完成状态。第六版开始实施前，应在第五版第六阶段完成并归档后另建 `docs/plan/` 活动计划；当前进展、备案和远程部署结论仍只维护在 `docs/status/project_status.md`。

第六版设计可以在第五版等待正式部署期间先行完成，但默认实施顺序为：

```text
第五版远程部署与证据闭环
  -> 第六版参与者语义实现与验证
  -> 功能冻结
  -> 最终课程报告、截图与答辩材料
```

如果课程时间不足以实施第六版，系统继续以第五版的 `groups_only` 语义交付，并在报告中明确其属于学生群体近似，不宣称能够处理个体选课冲突。

## 2. 基线与问题定义

### 2.1 可复用的第五版边界

第六版继续复用以下长期架构，不重新建设另一套平台：

- Web 负责工作流和展示，不承载算法核心逻辑。
- Fastify API 负责鉴权、事务、输入装配、作业、草稿、发布和审计。
- PostgreSQL 保存业务事实、作业状态、不可变请求快照和运行结果。
- Redis / BullMQ 只负责协调、投递和重试，不作为报名或排考事实源。
- Worker 只消费已冻结的作业输入并调用 FastAPI scheduler。
- scheduler 保持无业务数据库依赖，只接收纯调度合同。
- 作业请求、约束策略、运行结果和发布版本继续保持可追溯、可重放和可审计。

第六版不改变第五版的可靠任务状态机、Outbox、SSE、角色权限、草稿终态治理和 digest 部署边界。

### 2.2 当前学生冲突语义

当前 `ExamTask` 通过 `student_group_ids` 关联一个或多个学生群体。scheduler 对每个 `(group_id, time_slot_id)` 建立互斥约束，API 的草稿冲突检测也按群体判断。

该实现符合现有正式需求 FR-05 的“同一学生群体同一时间不能参加多场考试”，不是实现缺陷，但存在两类领域近似：

1. **漏判：** 同一学生因跨专业选课、选修或重修参加两个不同群体的考试时，群体不相交，系统无法发现个体重叠。
2. **误杀：** 两场考试虽然关联同一群体，但实际报名学生集合互不重叠时，群体级互斥仍禁止同槽安排。

因此，第六版是对 FR-05、FR-06 和 FR-07 的增强，不把第五版历史结果重新描述为错误，也不把报名表扩张成完整学籍或选课系统。

### 2.3 核心风险

学生个体模型会横穿 shared 合同、数据库、API 输入装配、作业快照、scheduler、草稿校验、发布资格、种子数据和 Web 展示。第六版必须优先控制以下风险：

- 部分考试有报名、部分考试无报名造成假阴性。
- 报名数据在排队、重试或草稿调整期间变化，导致同一作业不可复现。
- scheduler 使用个体冲突，但草稿或发布仍使用群体冲突，形成语义分叉。
- `group.size`、`expected_count` 和报名数量成为三个互不一致的事实。
- 为了个人化展示引入真实姓名、证件号或大规模学生账户，扩大隐私与实现范围。
- 借模型升级顺带重写仓储、实验平台或前端，导致第六版无法收口。

## 3. 目标与非目标

### 3.1 核心目标

第六版必须完成以下目标：

1. 保留学生群体组织口径，并新增最小化的脱敏学生与考试报名事实。
2. 以批次级显式模式区分群体近似和个体报名语义，禁止静默混合。
3. 在 enrollment 模式下，从完整报名数据确定性生成考试重叠边。
4. 将参与者模式、报名版本、重叠边和摘要写入不可变作业输入。
5. 让自动求解、草稿校验、建议、增量重排和发布资格使用同一份参与者快照。
6. 对名单不完整、人数不一致、重叠边非法和快照过期输出稳定诊断。
7. 用跨专业、选修、重修和群体误杀对照样例证明语义升级。
8. 增加一个简单基线与 CP-SAT 对照，形成可写入课程报告的可复现实验证据。
9. 只抽取支撑本主题的 schedule input builder，不进行无行为收益的大规模重构。

### 3.2 明确非目标

第六版不包含：

- 学生学籍、培养方案、选课审批、成绩、签到、座位或准考证系统。
- 与学校教务、CAS、OIDC、短信或邮件平台的正式集成。
- 为所有学生创建真实账户，或强制把现有学生页面改成个人学籍门户。
- Excel 解析、复杂字段映射和教务导入中台；结构化 JSON 批量输入足以满足作品演示。
- Pareto 方案集、自动策略推荐、room-slot 与教师联合全局优化。
- 一场考试拆分多个考场、多租户、Kubernetes、高可用集群或大型 BI。
- 完整修复向导、实时多人协同、LLM 自动改课表或大型算法实验室。
- 为降低文件行数而重写全部路由、内存仓储或 PostgreSQL 仓储。

## 4. 核心术语与不变量

### 4.1 参与者模式

`participant_mode` 是考试批次级语义，只允许以下值：

| 模式 | 冲突事实 | 使用场景 |
| --- | --- | --- |
| `groups_only` | `student_group_ids` 形成群体互斥 | 兼容第五版、报名数据不可用 |
| `enrollments` | 报名集合生成的 exam-exam overlap edges 形成个体互斥 | 第六版精确语义 |

同一考试批次、同一作业快照和同一运行只能使用一种模式。禁止按考试任务混用，也禁止“有报名的考试按个体、无报名的考试按群体”回退。

历史批次和历史作业默认解释为 `groups_only`。新作业必须显式保存模式，不依赖运行时默认值。

### 4.2 学生群体

学生群体继续承担组织、筛选、统计、通知范围和第五版兼容职责。在 enrollment 模式下，群体不是学生冲突的事实源。

第六版只允许学生拥有一个可选的主要组织群体。跨专业、选修和重修通过报名关系表达，不为了这些场景建立多群体学籍模型。

### 4.3 脱敏学生

`student` 是排考参与者标识，只保存：

- 稳定的脱敏 `student_id`；
- 可选的 `display_code`；
- 可选的主要学生群体；
- 启用状态和必要时间戳。

第六版不保存姓名、身份证号、手机号、邮箱、家庭信息或其他真实个人信息。固定样例和生成器使用确定性脱敏 ID，例如 `S000123`。

### 4.4 考试报名

`exam_enrollment` 表示某个学生是否参加某场考试，是 enrollment 模式下的排考事实。报名来源只用于解释和审计，不改变冲突强度：

- `regular`：常规教学关系；
- `elective`：选修；
- `retake`：重修；
- `other`：其他受控来源。

同一学生无论通过何种来源参加两场考试，只要考试时间重叠，就构成同等强度的硬冲突。

### 4.5 长期不变量

第六版必须保持以下不变量：

1. enrollment 模式下，每个考试任务都有已声明完整的有效报名集合。
2. `(exam_task_id, student_id)` 在有效报名中唯一。
3. enrollment 模式下，`expected_count` 等于有效报名数，不能由操作员独立维护第三份人数事实。
4. `groups_only` 模式下，`expected_count` 不得超过关联群体去重后的总人数；小于群体总人数允许存在，但必须可诊断。
5. 报名覆盖率百分比不能替代完整性声明；未完整时一律禁止求解和发布。
6. enrollment 模式只使用 overlap edges 约束学生冲突，不叠加群体硬互斥造成误杀。
7. 现有作业、运行和发布版本永远按创建时快照解释，当前报名变化不能改写历史语义。
8. 当前报名数据与目标运行快照不一致时，禁止发布、回滚到该运行或由其创建新的正式版本。
9. scheduler 不查询 PostgreSQL，也不接收真实个人信息。
10. 审计事件不保存全量报名名单，只保存操作摘要、版本、数量和 digest。

## 5. 数据模型设计

### 5.1 考试批次扩展

`exam_batches` 增加以下长期字段：

| 字段 | 语义 |
| --- | --- |
| `participant_mode` | `groups_only` 或 `enrollments` |
| `participant_data_status` | `not_required`、`draft` 或 `complete` |
| `participant_data_version` | 参与者数据的单调递增版本 |
| `participant_data_digest` | 完整状态下规范化参与者数据的 SHA-256 摘要，可为空 |

兼容迁移后，现有批次统一为：

```text
participant_mode = groups_only
participant_data_status = not_required
participant_data_version = 0
participant_data_digest = null
```

切换到 enrollment 模式后，状态进入 `draft`。报名增删或状态变化、已报名学生的启停、考试任务增删以及参与者模式切换都必须递增版本、清空旧 digest 并重新进入 `draft`。只修改不参与求解的展示字段时，不改变参与者版本。

### 5.2 学生表

新增 `students`：

| 字段 | 约束 |
| --- | --- |
| `id` | 主键，稳定脱敏 ID |
| `display_code` | 可空；存在时唯一 |
| `primary_student_group_id` | 可空外键，删除受限 |
| `status` | `active` 或 `disabled` |
| `created_at` / `updated_at` | 审计时间 |

学生停用不自动删除历史报名。是否参与当前批次由有效报名和批次快照决定。

### 5.3 报名表

新增 `exam_enrollments`：

| 字段 | 约束 |
| --- | --- |
| `exam_task_id` | 外键，级联删除考试任务 |
| `student_id` | 外键，删除受限 |
| `source` | `regular`、`elective`、`retake` 或 `other` |
| `status` | `active` 或 `withdrawn` |
| `created_at` / `updated_at` | 审计时间 |

主键为 `(exam_task_id, student_id)`。只有 `active` 报名进入人数、重叠边和快照；`withdrawn` 保留最小历史事实，不进入求解。

### 5.4 人数一致性

在 `groups_only` 模式下：

- `expected_count <= sum(size of distinct group IDs)` 是提交前硬门禁；
- `expected_count < sum(size of distinct group IDs)` 产生 warning，说明考试可能只覆盖群体中的部分学生；
- 该模式仍明确标注为群体近似。

在 `enrollments` 模式下：

- UI 和通用编辑 API 不允许单独修改 `expected_count`；
- seal 操作在事务内按有效报名数回写或核对 `expected_count`；
- 任何不一致都产生 `expected_count_mismatch` 并阻止完成状态；
- 不长期维护独立于报名数的“预计人数”。

### 5.5 不建立持久化重叠边表

第六版不把 overlap edges 作为新的业务事实表长期双写。边是报名快照的确定性派生物，由统一 input builder 在 seal、预检或作业提交时生成，并进入不可变请求快照。

如果后续性能证据表明每次派生成本不可接受，可以增加带 `participant_data_version` 和 digest 的缓存表；缓存不能成为报名之外的第二事实源。

## 6. 参与者数据生命周期

### 6.1 编辑与导入

第六版提供结构化 JSON 批量输入和必要的基础编辑能力，不建设 Excel 导入中台。批量输入至少支持：

- 脱敏学生；
- 考试报名；
- 报名来源；
- 撤回状态。

报名批量导入默认采用整批原子语义：先校验全部记录，再在单一事务中写入。任何非法引用、重复学生、重复报名、未知来源或跨批次考试都使整批失败，并返回稳定的行级错误；禁止把部分成功结果自动标记为完整。

### 6.2 完整性检查与 seal

管理员或排考员必须执行显式 seal，系统才可把 enrollment 数据从 `draft` 变为 `complete`。seal 至少检查：

1. 批次包含考试任务且每场考试至少有 1 条有效报名。
2. 所有学生与考试任务引用有效，考试任务属于目标批次。
3. 不存在重复有效报名。
4. 有效报名学生未被停用。
5. 每场考试报名数与 `expected_count` 一致。
6. overlap edges 可以确定性生成且全部引用当前批次考试。
7. 规范化数据摘要与 seal 事务结束时的版本一致。

seal 成功后保存 `participant_data_digest`。系统只能证明“对已声明并冻结的报名快照不存在未处理个体冲突”，不能证明该名单与学校真实教务系统完全一致。

### 6.3 修改后的失效语义

参与者数据完成后再次修改时：

- 批次立即回到 `draft`；
- 版本递增，旧 digest 失效；
- 已排队或已运行作业继续按其旧快照执行和展示；
- 尚未发布的旧运行、草稿和回滚目标标记为 participant snapshot stale；
- 已发布版本不被静默撤销，但管理员必须看到数据已变化的告警，新发布前必须重新 seal 和排考。

## 7. Schedule Input Builder

### 7.1 职责

第六版抽取单一 `schedule-input-builder`，负责：

- 按批次读取课程、教师、群体、考场、时段、考试任务和参与者数据；
- 解析批次参与者模式与完整性状态；
- 校验人数和引用一致性；
- 生成规范化 overlap edges；
- 绑定不可变约束策略快照；
- 生成 scheduler 输入、参与者快照元数据和稳定 digest；
- 为同步排考、异步作业和增量重排提供同一装配结果。

builder 属于 API/application 边界，不放入 Web、路由处理器、PostgreSQL repository 或 Python scheduler。repository 只提供结构化读取能力，不在内存仓储和 PostgreSQL 仓储各复制一套边生成规则。

### 7.2 一致性边界

异步作业提交必须在受控事务或版本 CAS 中完成以下动作：

1. 读取 `participant_data_version`、状态和 digest。
2. 读取有效报名并生成规范化边。
3. 生成完整 request snapshot 和 request digest。
4. 再次确认参与者版本未变化。
5. 创建 `schedule_jobs` 与 outbox 事件。

任何版本漂移都返回稳定冲突，不得用新报名覆盖已构建输入。

同步排考必须调用同一 builder，并把相同参与者摘要持久化到运行记录。不得为同步入口保留一个不带快照的快捷路径。

### 7.3 草稿与增量重排

草稿从来源运行继承参与者模式、overlap edges 和 snapshot digest。以下操作都使用该冻结快照：

- 拖拽或表单调整后的冲突检测；
- 调整建议生成与应用；
- 草稿校验；
- 增量重排；
- 发布资格检查。

草稿不得在后台自动切换到当前报名数据。需要刷新参与者数据时，应从新 seal 的批次创建新作业或新运行，不能原地改变旧草稿语义。

## 8. 合同与快照版本

### 8.1 Scheduler 输入合同

新的调度输入版本至少增加：

```json
{
  "participant_mode": "enrollments",
  "student_overlap_edges": [
    {
      "exam_task_id_a": "exam-a",
      "exam_task_id_b": "exam-b",
      "overlap_count": 2,
      "sample_participants": [
        {
          "student_id": "S000123",
          "exam_a_source": "regular",
          "exam_b_source": "retake"
        }
      ]
    }
  ]
}
```

边必须满足：

- `exam_task_id_a < exam_task_id_b`，按字符串稳定排序；
- 不允许自环、重复边和未知考试；
- `overlap_count` 为正整数；
- `sample_participants` 按 `student_id` 排序，最多保留 5 个脱敏样例；
- `groups_only` 模式下边集合必须为空；
- `enrollments` 模式下边集合必须由当前 seal 快照确定性生成。

scheduler 只使用考试 ID 和边建立硬约束；样例参与者只用于稳定解释，不参与求解逻辑。

### 8.2 作业请求快照 v3

历史 request snapshot v1 / v2 必须继续可读并解释为 `groups_only`。读取历史快照时，由 application/Worker 边界规范化为新的 scheduler 输入结构，但不改写数据库中的原快照或原 request digest。第六版新作业写入 v3：

```json
{
  "version": 3,
  "input": {},
  "constraintProfile": {},
  "participantSnapshot": {
    "schemaVersion": 1,
    "batchId": "batch-id",
    "mode": "enrollments",
    "dataVersion": 7,
    "digest": "sha256",
    "studentCount": 1200,
    "enrollmentCount": 1480,
    "overlapEdgeCount": 86
  }
}
```

现有 `requestDigest` 必须覆盖 v3 的完整 input、策略快照和参与者快照。Worker 在领取、重试和故障回收时只消费数据库中的 v3 快照，不重新读取当前报名表。

### 8.3 运行追溯

运行、草稿和发布版本至少可追溯：

- `participant_mode`；
- `participant_data_version`；
- `participant_snapshot_digest`；
- 学生数、报名数和 overlap edge 数；
- 约束策略版本与 digest；
- scheduler 版本和评分合同版本。

参与者摘要用于展示和审计；完整边集合保留在受保护的作业/运行输入快照中，不进入普通审计事件列表。

## 9. Scheduler 与冲突语义

### 9.1 群体模式

`groups_only` 继续使用现有 `(group_id, time_slot_id)` 互斥约束和群体冲突检测。第五版样例、历史快照和现有学生群体页面保持兼容。

UI、运行详情和报告必须标注：该模式保证学生群体不在同槽重复，不能保证跨群体个体无冲突。

### 9.2 报名模式

`enrollments` 对每条 `(exam_a, exam_b)` overlap edge 和每个时段建立：

```text
exam_a_at_slot + exam_b_at_slot <= 1
```

该模式不再叠加学生群体硬互斥。容量仍使用由有效报名数确定的 `expected_count`，其他考场、教师、设备、类型、允许时段和固定安排约束保持不变。

### 9.3 人工调整与发布校验

API 的草稿冲突检测必须复用来源快照中的 overlap edges。人工调整后若共享学生的两场考试进入同一时段，应产生 `student_exam_clash` 硬冲突并阻止发布。

发布与回滚还必须校验：

- 运行或草稿通过全部硬约束；
- assignment 与考试任务一一对应；
- 目标参与者快照不是 stale；
- enrollment 模式下，当前批次仍为 `complete`，且当前 digest 与目标运行 digest 相同；
- `groups_only` 模式保持 `not_required` 兼容语义，不要求参与者 digest。

不存在“管理员确认后带个体硬冲突发布”的例外路径。

### 9.4 软约束边界

第六版核心只升级个体硬冲突，不传入完整 student-to-exams 倒排，不实现个体连续考试软约束。

- `groups_only` 可以继续使用现有学生群体连续考试软规则。
- `enrollments` 默认关闭该群体软规则，避免把组织标签误当个体偏好。
- enrollment 模式与 group 模式的总分不能直接排名，报告必须同时展示模式和评分合同。
- 个体连续考试、跨考试日负担等优化只能在后续有明确需求时另行设计。

## 10. 预检、诊断与错误处理

### 10.1 稳定诊断码

第六版至少定义以下稳定代码：

| 代码 | 严重度 | 含义 |
| --- | --- | --- |
| `participant_mode_invalid` | error | 模式与输入字段不一致 |
| `participant_data_incomplete` | error | enrollment 数据未 seal 或已因修改失效 |
| `participant_snapshot_stale` | error | 当前批次与运行/草稿快照不一致 |
| `expected_count_exceeds_group_size` | error | group 模式人数超过群体容量 |
| `expected_count_lower_than_group_size` | warning | group 模式人数小于群体总人数 |
| `expected_count_mismatch` | error | enrollment 模式人数与有效报名不一致 |
| `student_enrollment_reference_invalid` | error | 报名引用无效或跨批次 |
| `student_overlap_edge_invalid` | error | 重叠边非规范、重复或引用未知考试 |
| `student_exam_clash` | error | 共享学生的考试被安排在同一时段 |

诊断结构继续包含资源维度、受影响 ID、缺口或 overlap 数量、展示文案和处理建议。UI 依赖代码决定行为，不解析自由文本。

### 10.2 输入失败与业务不可行

- 参与者数据不完整、引用非法或快照过期属于输入/治理错误，不进入自动重试。
- overlap edges 使问题无可行解属于正常业务不可行，应生成完整运行、诊断和审计，不标记为基础设施失败。
- Worker 网络错误、scheduler 不可用和超时继续按第五版可靠任务分类处理。
- 重试不得重新计算 overlap edges，也不得改变参与者 digest。

## 11. API 与权限边界

### 11.1 管理命令

第六版 API 至少提供以下业务能力，具体路由可在实施计划中与现有 reference-data 风格对齐：

- 设置批次参与者模式；
- 批量写入脱敏学生和报名 JSON；
- 查询参与者数据健康摘要；
- seal 参与者数据；
- 查看报名数量、来源分布和 overlap 摘要；
- 查询运行或草稿的参与者快照元数据。

模式切换、批量导入、seal 和重新打开参与者数据均为高影响操作，必须有服务端权限检查、确认交互和真实 actor 审计。

### 11.2 权限

- 管理员可以管理学生、报名、模式和 seal 状态。
- 排考员可以查看参与者健康状态、提交作业；是否允许修改报名由实施计划按现有角色权限收敛。
- 教师和学生角色不得读取报名名单、overlap 样例学生 ID 或内部诊断。
- scheduler、Worker 和发布流程不获得额外用户权限或业务数据库凭据。

### 11.3 审计

审计至少记录：

- 操作类型、批次、actor、结果和 trace ID；
- 修改前后参与者版本和状态；
- 导入成功/失败数量；
- seal 的学生数、报名数、边数和 digest；
- 模式切换和因数据变化造成的 stale 结果数量。

审计 payload 不保存完整学生列表、完整报名列表或完整 overlap edges。

## 12. Web 与展示边界

### 12.1 参与者数据健康状态

管理员基础数据或排考入口增加紧凑的参与者状态区，展示：

- 当前模式；
- `draft` / `complete` 状态和版本；
- 学生数、有效报名数、考试覆盖数和重叠边数；
- 人数不一致、无报名考试、非法引用和 stale 运行数量；
- seal 或重新排考的明确操作入口。

该区域是第六版参与者数据的必要治理界面，不扩张为通用导入质量中心。

### 12.2 冲突解释

运行详情、草稿冲突和建议面板对个体冲突至少展示：

- 两场受影响考试；
- 冲突时段；
- overlap 总人数；
- 最多 5 个脱敏学生样例及报名来源；
- 返回对应考试或草稿位置的导航。

第六版不要求实现复杂关系图或热力图。稳定的列表、计数和样例已经能够支撑答辩解释；可视化只有在核心语义完成后才作为独立加分项评估。

### 12.3 学生本人页面

第六版核心继续保留现有 `user_student_group_scopes` 和群体已发布查询，不强制新增个人学生账户。

如果主线全部完成且仍有时间，可为单个演示账号增加 `user_student_scopes`，展示一名脱敏学生的个人课表；该能力是可选展示，不得成为个体冲突正确性的前置条件。

## 13. 数据生成、样例与算法证据

### 13.1 确定性生成器

现有固定 seed 生成器增加以下参数或等价能力：

- 学生规模和群体规模；
- 常规、选修和重修报名比例；
- 跨群体重叠比例；
- exam-exam overlap 密度；
- 可行和故意不可行模式。

相同参数和 seed 必须生成相同学生、报名、边集合、digest 和期望结果。

### 13.2 对抗样例库

第六版至少沉淀以下命名样例：

1. `cross_major_elective_overlap`：不同群体的同一学生参加两场考试，旧群体模型漏判，enrollment 模式阻止同槽。
2. `retake_cross_grade_overlap`：重修学生跨年级参加考试，诊断能解释报名来源。
3. `same_group_disjoint_rosters`：两场考试关联同一群体但名单互斥，enrollment 模式允许同槽，证明减少误杀。
4. `incomplete_enrollment_snapshot`：任一考试名单未完成，作业提交被拒绝。
5. `expected_count_mismatch`：人数与有效报名不一致，seal 失败。
6. `participant_snapshot_stale`：seal 后修改报名，旧运行不能发布。
7. `retry_uses_frozen_snapshot`：报名变化后重试仍使用原作业 digest。
8. `legacy_groups_only_compatible`：第五版输入和历史快照保持原有结果。

每个样例包含固定输入、输入 digest、预期状态、预期诊断码和一句话业务说明，并进入自动化测试。

### 13.3 简单基线对照

第六版增加一个仅用于离线实验的简单确定性构造基线。基线与 CP-SAT 必须消费同一份冻结输入和 seed，输出同一套可比较指标：

- 是否可行；
- 硬冲突数量；
- 归一化软约束分数及分项；
- 求解耗时；
- 教师负载极差；
- overlap 边数量和密度。

基线不进入生产排考入口，也不包装为自动策略推荐。实验报告应诚实记录 CP-SAT 在质量、可行率和耗时上的取舍，不预设所有指标都必须优于基线。

### 13.4 规模验证

50、100、150 场固定规模继续保留，并新增可控报名重叠密度。验收至少要求：

- 预期可行样例为 `feasible` 且硬冲突为 0；
- 预期不可行样例返回稳定业务状态和诊断；
- 所有样例在既有 scheduler 时间限制内结束；
- 同一输入重复运行的参与者 digest、状态和关键统计稳定；
- 本地结果不得冒充真实学校数据或腾讯云生产上限。

## 14. 测试与验证边界

### 14.1 合同测试

- shared 与 Python 模型对参与者模式、边结构和诊断码保持一致。
- scheduler OpenAPI 门禁覆盖新输入、非法边和错误响应。
- request snapshot v1 / v2 可读，v3 新写，未知版本拒绝。
- request digest 对报名版本、边、样例解释和策略变化敏感。

### 14.2 数据库测试

- 新迁移可以从空库和现有第五版数据库应用，第二次执行为 0 项。
- 外键、主键、枚举、默认兼容值和迁移检查器均有证据。
- 批量导入、seal、版本递增和 stale 判定在真实 PostgreSQL 中验证。
- 内存仓储与 PostgreSQL 仓储遵守相同业务合同，但不复制边生成算法。

### 14.3 业务与算法测试

- 自动求解、草稿调整、建议、增量重排和发布使用同一快照。
- group 与 enrollment 两种模式分别覆盖正向、负向和兼容样例。
- 同一学生的不同报名来源不改变硬冲突强度。
- 修改当前报名不改变历史作业重试和历史运行展示。
- stale 运行、草稿和回滚目标无法发布。

### 14.4 Web 与 E2E

- 参与者状态、seal、错误摘要和冲突解释有 Playwright 覆盖。
- 非管理员角色无法访问名单和样例学生 ID。
- enrollment 模式与 group 模式有清晰、不可混淆的标签。
- 关键桌面和移动视口无文本、表格或控件溢出，并继续通过 Axe 门禁。

## 15. 分阶段设计边界

第六版实施计划可以拆为以下阶段，但每个阶段仍需另写活动计划和验证命令：

### 15.1 第一阶段：数据诚实性与合同

- 批次参与者模式、状态、版本和 digest；
- 学生与报名数据模型；
- 原子 JSON 导入、人数一致性和 seal；
- schedule input builder；
- request snapshot v3 与历史兼容。

第一阶段结束时，坏数据不能静默进入求解，但尚不能宣称 scheduler 已处理个体冲突。

### 15.2 第二阶段：个体冲突全链路

- overlap edges 生成与稳定摘要；
- scheduler 硬约束和诊断；
- 草稿、建议、增量重排与发布资格复用快照；
- stale 运行与发布门禁；
- PostgreSQL、API、scheduler 和跨语言合同验证。

第二阶段结束时，跨专业、选修和重修固定样例必须形成完整红绿证据。

### 15.3 第三阶段：证据与有限展示

- 对抗样例库；
- 简单基线对照与重叠密度基准；
- 参与者健康状态；
- 个体冲突解释界面；
- 报告可引用的 JSON、表格和截图证据。

第三阶段完成即达到第六版停止线。除非核心门禁全部稳定且课程时间明确允许，否则不再开启新的功能主线。

## 16. 可选后续与停止线

以下内容有价值，但不进入第六版核心验收：

| 候选 | 允许条件 |
| --- | --- |
| 草稿建议应用前后 delta | 核心建议合同稳定后，补相对当前草稿的冲突与分数变化 |
| 发布影响面增强 | 个体快照稳定后，补受影响考试、教师和群体统计 |
| ICS / 打印安排 | 最终展示仍有时间，且不影响核心报告冻结 |
| 时段长度与 `duration_minutes` 预检 | 作为独立窄修复，不与多考场模型绑定 |
| 单个脱敏学生演示账号 | 核心语义完成后，仅用于作品展示 |
| 证据包与答辩脚本 | 功能冻结后整理，不反向扩大产品范围 |

以下信号出现时必须停止扩张：

- 新需求需要改变 assignment 基数或重写核心状态机；
- 需要引入真实学生身份或上游教务同步才能成立；
- 需要新增完整实验平台、审批流或多目标推荐；
- 无法用一句话说明其对冲突真实性、证据或课程展示的直接收益；
- 距课程报告冻结不足以完成全量回归和证据整理。

## 17. 验收标准

第六版只有同时满足以下条件才可声明完成：

1. 现有第五版数据和历史作业无需补学生名单即可继续按 `groups_only` 读取和运行。
2. enrollment 批次未完整 seal 时，所有排考入口稳定拒绝。
3. 作业 v3 冻结参与者模式、版本、digest 和 overlap edges，重试不读取当前报名。
4. 自动求解、草稿、建议、增量重排和发布使用同一参与者快照。
5. 跨专业选修和重修样例不再漏判，同群体但名单互斥样例不再误杀。
6. enrollment 模式的 `expected_count` 与有效报名数一致，group 模式的人数异常可诊断。
7. stale 运行、草稿和回滚目标不能成为新发布版本。
8. scheduler 继续无业务数据库依赖，API/Worker/scheduler 职责不越界。
9. 全量报名和真实个人信息不进入普通日志、审计或非管理员响应。
10. shared、API、PostgreSQL、scheduler、Worker、Web、Compose 和 Playwright 风险相关门禁通过。
11. 固定样例、基线对照和规模验证产出可复现证据，报告不夸大真实学校或生产结论。
12. 第六版活动计划完成后按仓库规则归档，项目状态、设计、报告和文档索引口径一致。
