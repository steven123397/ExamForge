# 高校智能排考与考务资源调度系统第五版实现内容设计

## 1. 文档定位

本文档定义 ExamForge 第五版的长期设计方向、架构边界、阶段候选和验收口径。第五版的主题是“可靠调度执行与专业化交付”：把第四版形成的完整演示闭环升级为可在真实服务器上稳定运行、可恢复、可追踪、可面向不同角色使用的平台。

本文档不是活动实施计划。第五版各阶段开始前仍应在 `docs/plan/` 编写独立计划，明确任务拆分、文件范围、迁移顺序和验证命令。第四版完成后的全量代码审查仍是第五版实施前置门槛；审查发现维护在 `docs/status/code_review_status.md`，修复工作另行制定计划，不直接混入第五版功能阶段。

## 2. 当前基线与演进判断

截至第五版第五阶段，系统已经具备以下基础：

1. Web、Fastify API、PostgreSQL 和 Python scheduler 已形成基础数据、自动排考、草稿治理、人工调整、发布、回滚、查询、通知和导出闭环。
2. room-slot 与教师分配采用顺序二阶段 CP-SAT，增量重排合同贯通 shared、API、Python CLI 和 Web，并建立固定 seed 的 50、100、150 场规模基准。
3. 排考、草稿和发布治理进入独立 service/use-case 层；PostgreSQL 关键关联表成为优先读路径，草稿并发终态通过 advisory lock 和 CAS 治理。
4. Docker Compose 已覆盖 PostgreSQL、Redis、迁移、seed、独立 FastAPI scheduler、API、Publisher、Worker 和 Web，并建立真实 readiness、进程间 smoke 与双模式 Playwright E2E。
5. GitHub Actions 已建立快速检查、真实 PostgreSQL/迁移检查和 Compose/Playwright 三层质量门禁。
6. 约束策略已经形成身份、不可变版本、启停、唯一默认版本和审计合同；异步作业、同步排考与草稿重排均解析并持久化实际使用的策略版本、完整快照、稳定摘要、scheduler 版本和评分合同版本。
7. scheduler 与 shared 已统一保留原始违反、原始/加权惩罚、适用机会数、归一化分项和归一化总分，并以稳定结构输出容量、时段、教师、固定安排、学生群体和引用错误诊断。
8. Web 已按角色拆分十个固定路由，运营筛选和选中上下文进入 URL；教师与学生通过 PostgreSQL 作用域和 `/api/me/*` 访问本人数据，旧按 ID 查询不再匿名开放。
9. 草稿矩阵使用 dnd-kit 并保留键盘/表单等价路径；四种视口、6 张 Chromium 截图、Axe WCAG 2.1 A/AA 扫描和文本溢出检查已进入 Playwright 门禁。

第五版前五个阶段已将身份、scheduler 服务化、可靠任务、实时事件、任务治理、约束策略和多角色前端落地。当前主要上限不再是“能否可靠完成一次排考、追溯策略或按角色安全访问”，而是以下生产交付能力仍未取得真实证据：

- CI 尚未发布版本化镜像，生产 Compose、镜像摘要/SBOM、正式 secrets 和受控回滚尚未形成。
- 腾讯云上的可信域名、HTTPS、资源约束、异机备份恢复、日志轮转、监控告警和线上 smoke 尚未取得真实证据。
- `next@15.5.20` 的精确 PostCSS 依赖仍命中 2 个 moderate 公告，正式发布前必须由安全升级或明确风险接受决定处置。
- 策略版本、评分和诊断已可追溯，但尚未建设多策略批量实验、Pareto 比较或自动策略推荐；这些能力不属于第五版范围。

第六阶段应成组解决生产交付问题。部署能力必须对应明确的安全、资源、备份、故障恢复和回滚证据，且本地 Compose 结果不能冒充远程结论。

## 3. 第五版目标与原则

### 3.1 核心目标

第五版需要完成以下目标：

1. 建立可排队、恢复、重试、取消、超时和追踪的可靠调度作业系统。
2. 将 Python scheduler 升级为可独立部署的 FastAPI 计算服务，同时保留 CLI 调试入口。
3. 建立数据库账户、角色、会话和真实审计 actor，替换演示 token。
4. 将作业进度从轮询升级为可断线恢复的 Server-Sent Events（SSE）事件流。
5. 将 Web 从单页运营台重构为多角色、真实路由、数据密集但易扫描的专业工作台。
6. 在 4 核 4 GB 腾讯云 Ubuntu 服务器上完成私有试部署和正式部署验证。
7. 为约束策略版本、归一化评分和后续方案实验建立持久化基础，但不在本版扩张为完整实验平台。

### 3.2 架构原则

- **业务单体继续模块化：** Fastify API 保持模块化单体，不把基础数据、草稿、发布和审计机械拆成微服务。
- **计算边界独立：** scheduler 具有独立运行环境、资源消耗和故障模式，应作为无业务数据库依赖的计算服务部署。
- **PostgreSQL 是唯一事实源：** 业务数据、作业状态、作业事件、运行结果和审计历史均以 PostgreSQL 为准。
- **Redis 只负责协调：** Redis/BullMQ 负责消息分发、重试调度和短期通知，不保存不可替代的业务事实。
- **至少一次消费配合幂等：** 不宣称分布式 exactly-once；通过任务幂等键、结果唯一约束和可重入 service 抵御重复投递。
- **先定义状态，再实现实时通道：** SSE 只表达已经持久化的状态变化，不让易失事件反向决定业务状态。
- **前端围绕工作流设计：** 优先提升排考、冲突定位、草稿调整、任务追踪和角色查询效率，不建设营销式界面。
- **部署约束进入设计：** 资源配额、备份、日志和失败恢复从一开始作为验收条件，而不是开发结束后的补充事项。

### 4.1 组件职责

| 组件 | 负责 | 不负责 |
| --- | --- | --- |
| Next.js Web | 多角色工作台、任务中心、草稿矩阵、发布查询和实时反馈 | 业务事务、算法求解 |
| Fastify API | 认证授权、业务用例、事务边界、作业提交、结果查询、审计和 SSE | 执行 CP-SAT、承担后台长任务 |
| PostgreSQL | 业务事实、账户会话、作业状态、outbox、事件、运行结果和审计 | 高频易失消息广播 |
| Outbox Publisher | 将已提交的 outbox 事件可靠投递到 BullMQ | 创建业务任务、修改求解结果 |
| Redis/BullMQ | 任务分发、延迟重试、并发控制和短期事件协调 | 作为业务记录或事件历史唯一来源 |
| Scheduling Worker | 消费任务、调用 scheduler、处理重试/超时/取消并持久化结果 | 对外提供业务 API、承载算法实现 |
| FastAPI Scheduler | 预检、求解、校验、解释和运行统计 | 读取 ExamForge 业务数据库、管理用户和任务生命周期 |
| nginx | 复用服务器现有 HTTPS、反向代理和入口路由 | 业务鉴权、持久化应用状态 |

### 4.2 建议仓库边界

```text
apps/
├── web/
├── api/
├── worker/
└── scheduler/
packages/
├── db/
├── shared/
├── scheduling-application/
└── config/
```

`apps/worker` 是独立运行进程；API 与 worker 复用调度作业 application service 和 repository 合同，避免各自复制状态迁移与结果持久化逻辑。是否新建 `packages/scheduling-application` 应以实际依赖边界为准，不为追求目录数量提前搬迁代码。

## 5. 数据与一致性设计

### 5.1 作业数据模型

第五版建议围绕以下实体扩展：

| 实体 | 作用 |
| --- | --- |
| `schedule_jobs` | 作业主记录、请求摘要、当前状态、幂等键和最终运行引用 |
| `schedule_job_attempts` | 每次领取、scheduler 调用、失败分类、耗时和重试信息 |
| `schedule_job_events` | 可恢复的阶段事件、进度事件和终态事件 |
| `outbox_events` | 与业务事务一同提交、等待投递的队列事件 |
| `users` | 本地用户账户和身份状态 |
| `roles` | 角色定义 |
| `user_roles` | 用户与角色关系 |
| `sessions` | 可撤销、可过期的服务端会话 |
| `constraint_profiles` | 可版本化的约束权重与策略配置 |

### 5.2 作业状态机

```text
queued
  |-- worker 领取 --> running
  |                    |-- 成功 -----------------> succeeded
  |                    |-- 不可行但正常返回 -----> succeeded
  |                    |-- 临时故障且可重试 ------> queued
  |                    |-- 重试耗尽/永久错误 -----> failed
  |                    |-- 超时 -----------------> timed_out
  |                    `-- 协作取消成功 ----------> cancelled
  `-- 排队中取消 -------------------------------> cancelled
```

`infeasible` 是求解业务结果，不是基础设施失败。它应生成完整运行记录、冲突解释和审计信息，不进入自动重试。

### 5.3 投递与消费一致性

API 在同一个 PostgreSQL 事务中写入 `schedule_jobs` 和 `outbox_events`。Outbox Publisher 只发布已提交事件，并允许重复发布。BullMQ `jobId` 与数据库作业 ID 对齐，Worker 在开始和完成时使用原子状态迁移与结果唯一约束。

任何重复消息、Publisher 重启或 Worker 重启都不得生成重复 `schedule_runs`。Redis 暂时不可用时，新任务可以保留为待投递状态；Redis 数据丢失时，PostgreSQL 中的业务和审计事实仍完整，并可通过补偿投递恢复。

### 5.4 JSONB 收尾

第五版第一阶段应在真实 PostgreSQL 迁移和双向一致性证据基础上删除已由关联表取代的兼容字段。删除前必须证明：

- 正式读路径不再依赖兼容回退。
- 所有历史数据已完成回填和一致性检查。
- 迁移可在干净数据库和现有第四版数据库上执行。
- shared DTO 不再误导调用方继续写入废弃字段。

## 6. 身份、权限与审计

### 6.1 认证边界

第五版采用可独立闭环验证的本地账户体系，不依赖学校 CAS、OIDC 或外部身份提供方。密码只保存安全散列；浏览器使用 `HttpOnly`、`Secure` 和合理 `SameSite` 策略的服务端会话 Cookie。会话必须支持过期、撤销、退出登录和管理员禁用账户。

### 6.2 角色边界

| 角色 | 主要权限 |
| --- | --- |
| 管理员 | 用户角色、基础数据、排考、发布、审计和系统配置 |
| 排考员 | 基础数据、任务提交、草稿调整、校验和发布申请范围内操作 |
| 教师 | 维护本人不可用时间、查看本人监考安排和变更 |
| 学生 | 查看所属学生群体的已发布考试和变更通知 |

权限检查必须发生在 API/use-case 边界，前端隐藏按钮不能替代服务端授权。审计事件应记录真实用户 ID、角色上下文、实体、操作、结果和关联 trace/job ID，不再使用演示角色字符串作为 actor。

## 7. Scheduler 服务化设计

### 7.1 服务接口

FastAPI scheduler 保持窄接口，候选接口包括：

```text
GET  /health
GET  /ready
POST /solve
POST /validate-adjustment
POST /explain-conflicts
```

`/solve` 接收纯调度合同，输出业务状态、安排、硬约束冲突、软指标、重排摘要、阶段统计和 scheduler 版本。接口不接收数据库连接信息、用户会话或 Web 页面对象。

### 7.2 合同治理

- FastAPI 生成 OpenAPI 作为跨语言网络合同。
- `packages/shared` 继续作为 Web/API 领域合同来源，但不得与 Python 模型手工长期漂移。
- CI 应验证示例 payload、必填字段、枚举、错误响应和兼容性。
- API/worker 必须区分合同错误、业务不可行、超时、网络失败和 scheduler 内部错误。
- JSON CLI 继续保留给单元测试、离线 benchmark 和故障调试，生产调用路径改为 HTTP 服务。

### 7.3 进度与取消

进度优先表达真实阶段，不伪造线性百分比：

1. 输入校验与确定性预检。
2. room-slot 求解。
3. 教师分配求解。
4. 报告与解释整理。
5. 结果持久化。

排队任务必须可确定性取消；运行中任务采用协作式尽力取消，并明确记录取消请求、scheduler 是否确认停止和最终状态。第五版不承诺跨进程强制终止后仍保留中间最优解，除非后续原型证明该语义稳定可测。

## 8. Redis、BullMQ 与实时事件

### 8.1 Redis 使用边界

第五版引入 Redis 的直接原因是可靠队列、延迟重试、并发控制和进程间事件协调。第五版不默认使用 Redis 承担：

- 全站查询缓存。
- PostgreSQL 业务数据副本。
- 替代草稿 PostgreSQL advisory lock 的分布式锁。
- 不可恢复的唯一作业状态。
- 全部会话存储。

如果后续没有性能证据，不为“用了 Redis”而增加缓存失效和双写问题。

BullMQ 队列使用的 Redis 应设置 `maxmemory-policy=noeviction`，不能使用 `allkeys-lru` 等缓存淘汰策略删除队列键。队列容量通过作业自动清理、保留期限、内存监控和告警治理；Redis 达到内存上限时应显式失败并触发运维处理，而不是静默淘汰任务数据。

### 8.2 重试与并发

- 仅网络中断、依赖暂时不可用和明确的临时错误进入自动重试。
- 输入非法、合同不兼容和业务不可行不重试。
- 重试使用有上限的退避策略，并记录每次 attempt。
- 腾讯云初始 Worker 并发固定为 1；是否提高并发由真实负载测试决定。
- scheduler 初始最多使用 2 个 CPU 核，给 Web、API、PostgreSQL 和系统保留资源。

### 8.3 SSE 事件流

SSE 从 PostgreSQL `schedule_job_events` 恢复历史，再通过短期通知获取新事件。客户端携带 `Last-Event-ID` 重连时，API 应补发缺失事件。Redis Pub/Sub 可以用于唤醒连接，但不能成为唯一事件历史。

事件至少覆盖：作业已排队、开始尝试、阶段变化、可用进度、重试安排、取消请求、运行创建和终态。事件 payload 应版本化，避免 UI 与后端字段零散耦合。

## 9. 约束策略与算法增强

### 9.1 约束策略基础

约束策略由 `constraint_profiles` 保存身份、状态、所有者、当前版本和唯一默认标记，由 `constraint_profile_versions` 保存不可变配置、schema 版本和稳定摘要。编辑配置必须创建下一版本；禁用只阻止新作业选择，不改变历史引用。

API 在作业创建事务中解析显式选择或当前默认版本，并把版本 ID、完整配置和摘要写入版本化请求快照。Worker 只消费该冻结快照；同步排考与草稿重排同样在调用 scheduler 前解析一次策略，并把实际调用版本、快照和 scheduler 元数据原样传入运行持久化。运行结果因此保存实际使用的策略版本与快照，同时记录 scheduler 版本和评分合同版本。幂等摘要覆盖策略语义，同一幂等键不能静默复用不同策略。

本版只要求完成策略创建、版本、启停、运行绑定和历史追溯，不建设完整的多策略批量实验室。完整方案实验、Pareto 比较和策略推荐留给后续版本。

### 9.2 评分与诊断

- 评分合同 v1 同时保存规则代码、违反次数、权重、原始惩罚、加权惩罚、确定性适用机会数、`[0, 1]` 归一化惩罚和 `[0, 100]` 归一化总分；零机会分项按 0 处理。
- 原始惩罚、旧展示分数、归一化分数和各软约束分项同时保留，不能以单一分数隐藏取舍，也不能跨不同策略版本无条件排名。
- 不可行诊断使用稳定代码、资源维度、受影响 ID、缺口数量、展示文案和建议，覆盖容量、时段、教师、固定安排、学生群体与引用错误；UI 不解析自由文本决定行为。
- 可行性与发布资格继续独立于数值评分；不可行方案即使具有数值分项也不能发布。
- 第五版继续准确表述顺序二阶段 CP-SAT，不宣称 room-slot 与教师达到联合全局最优。

## 10. 前端专业化设计

### 10.1 实施时机

前端优化分为两个时点：第五版第一阶段确定信息架构、视觉语言、设计 token 和关键页面原型；核心合同与工作流稳定后，第五阶段完成页面级重构。后续阶段应维护这些边界，不重新引入根路由聚合控制台或客户端身份作用域。

多角色路由、页面状态、视觉 token、响应式和截图验收的执行基线见 `docs/design/第五版前端信息架构与视觉规范.md`。

### 10.2 页面结构

固定路由如下：

```text
/login
/admin/overview
/admin/reference-data
/scheduling/jobs
/scheduling/runs
/scheduling/drafts/:id
/scheduling/policies
/audit
/teacher/schedule
/student/schedule
```

管理员与排考员使用工作型侧边导航；教师和学生进入面向本人任务的简化工作区。URL 保存当前实体、筛选和可分享的工作上下文，刷新、前进和后退不丢失主要位置。

### 10.3 视觉与交互原则

- 采用安静、紧凑、面向高频操作的数据后台风格，不使用营销式大标题和装饰性卡片堆叠。
- 优先优化表格扫描、筛选、批量操作、冲突定位、状态反馈和错误恢复。
- 草稿矩阵继续作为核心视图，支持 dnd-kit 拖拽调整，同时保留表单编辑作为键盘和无障碍备用入口。
- 作业中心直接消费 SSE，稳定展示阶段、尝试、耗时、错误和结果链接。
- 建立统一的加载、空状态、权限不足、依赖不可用和可重试错误表达。
- 覆盖桌面、平板和基本移动端查询，不建设独立移动应用，也不牺牲桌面数据密度追求移动端完全同构。
- 关键页面应增加 Playwright 截图基线、文本溢出检查和可访问性验证。

## 11. 腾讯云部署设计

### 11.1 已知环境

- 云平台：腾讯云。
- 操作系统：Ubuntu 22.04.5 LTS (jammy)。
- 计算资源：4 核 CPU、4 GB 内存。
- 系统盘：40 GB。
- 额外挂载盘：20 GB ext4，挂载在共享根 `/srv/data/hot`，当前约 19 GB 可用；COS 挂载在 `/srv/data/cos`。
- 现有入口：nginx + Certbot，已承载 2 个低负载网站。

该配置足以承载单机 Compose 演示和小规模真实访问，但必须限制 Worker 并发、scheduler CPU/内存和日志增长。服务器不用于现场构建完整项目镜像。

### 11.2 单机服务布局

```text
nginx
├── Web
└── API

内部 Docker 网络
├── PostgreSQL
├── Redis
├── Worker
└── Scheduler
```

公网只开放 HTTPS/HTTP 和受控 SSH。Web 和 API 可按需映射到宿主机 `127.0.0.1` 供 nginx 反代；PostgreSQL、Redis、Worker 和 scheduler 只在 ExamForge Compose 内部网络暴露，不映射宿主机端口。GitHub 托管 job 负责质量门禁，本地 WSL self-hosted Runner 负责构建、扫描并推送版本化镜像到广州 TCR，服务器只负责按 digest 拉取、迁移和启动。详细边界见 `docs/design/第五版本地托管Runner发布设计.md`。

### 11.3 初始资源约束

| 服务 | 建议初始内存边界 |
| --- | ---: |
| 操作系统、Docker 与现有低负载服务 | 700 MB-1 GB |
| PostgreSQL | 512-768 MB |
| Redis | 128-256 MB |
| Web | 256-384 MB |
| API | 256-384 MB |
| Worker | 128-256 MB |
| Scheduler | 1-1.5 GB |
| nginx | 64-128 MB，复用现有进程 |

以上是初始容量假设，不是提前宣称的性能结论。勘察时 Codex/opencode 合计占用约 811 MB，属于可能退出的交互式负载；正式容量基线应在关闭非必要交互进程后重新测量。私有试部署需要记录空闲、典型排考和压力场景的 CPU、内存、磁盘和响应时间，并据此修正 Compose 配额。建议配置 2-4 GB swap 作为极端峰值保护，但不得把 swap 视为常规内存或部署可行性的替代证据。

### 11.4 磁盘、备份与日志

系统盘上的 `/srv/apps/examforge` 用于 Compose 文件、nginx 配置模板和运维脚本，不保存在线数据库数据。额外 20 GB 挂载盘使用独立子目录承载 PostgreSQL、Redis 持久化数据、上传文件和短期备份，例如：

```text
/srv/data/hot/examforge/
├── pgdata/
├── redis/
├── uploads/
└── backups/
```

系统盘承载操作系统、Docker 镜像、容器层和受控日志。Docker 日志必须轮转，旧镜像和失败构建产物应有清理策略。数据库备份不能只保存在同一服务器；正式发布前应将 `pg_dump` 等一致性备份产物复制到腾讯云 COS 或其他独立存储，并完成一次恢复演练。COS 挂载不承载 PostgreSQL 或 Redis 在线数据目录。

### 11.5 两次部署检查点

1. **私有试部署：** 第三阶段可靠任务系统完成后进行，不公开宣传，不使用正式数据；验证资源峰值、HTTPS、反向代理、任务恢复、SSE、Redis/PostgreSQL 重启和镜像更新。
2. **正式部署：** 第五阶段前端重构完成后进行；配置正式域名、演示账户、备份、日志、监控、线上 smoke 和核心 E2E，再有限开放并观察。

## 12. 验证策略

### 12.1 合同与单元验证

- scheduler 模型、预检、求解、报告和 CLI 现有测试继续保留。
- FastAPI 接口覆盖合法输入、业务不可行、合同错误、超时和内部错误。
- OpenAPI 与 TypeScript/Python 模型建立兼容性检查。
- 身份 service 覆盖登录、退出、过期、撤销和角色授权。
- 作业 application service 覆盖全部合法和非法状态迁移。

### 12.2 真实依赖集成验证

- PostgreSQL 验证迁移、outbox 原子性、幂等结果写入和事件恢复。
- Redis/BullMQ 验证重复投递、延迟重试、Redis 短暂不可用和恢复投递。
- Worker 验证进程崩溃、重新领取、scheduler 超时和重试耗尽。
- SSE 验证断线重连、`Last-Event-ID` 补发和终态事件。
- 不使用内存仓储成功替代 PostgreSQL/Redis 真实结论。

### 12.3 浏览器与部署验证

- Playwright 覆盖管理员登录、任务提交、实时进度、草稿调整、发布、教师查询和学生查询。
- 浏览器测试覆盖 401、403、依赖不可用、任务失败、取消和重试路径。
- 前端重构增加截图基线、主要视口、文本溢出和无障碍检查。
- 腾讯云验证 API/Worker/scheduler 重启、数据库备份恢复、镜像回滚和完整 smoke。
- 150 场固定基准必须在 4 核 4 GB 服务器上稳定完成；更大规模只记录真实结果，不提前承诺并发或时限。

## 13. 建议阶段边界

### 13.1 第一阶段：合同、数据与身份基础

目标是固定作业状态机、outbox、账户会话、权限边界、JSONB 收尾方案，以及前端信息架构和视觉方向。该阶段不引入队列执行，不全面重写页面。

截至 2026-07-12，第一阶段已经落地：作业状态统一为 `queued`、`running`、`succeeded`、`failed`、`cancelled`、`timed_out`，作业尝试、持久化事件和 outbox 已进入 PostgreSQL；四类关系数据已删除旧 JSONB 列并只使用关联表；本地账户、服务端会话、四角色 RBAC、真实 actor 审计和 Cookie Web 会话已经贯通；前端信息架构与视觉规范已形成正式文档，并对现有原型完成基础 token、焦点和窄屏适配。

第一阶段没有实现 Redis/BullMQ、独立 Worker、FastAPI scheduler、SSE 或完整前端路由重构。API 仍以进程内执行器调用 scheduler CLI，Web 仍以轮询获取作业状态；这些边界分别由后续阶段处理。

### 13.2 第二阶段：Scheduler 服务化

目标是建立 FastAPI、OpenAPI、API/worker 客户端合同、故障分类和阶段级进度。CLI 保留为测试与调试入口，生产调用切换应在合同测试通过后进行。

截至 2026-07-13，第二阶段已经落地：scheduler 提供独立的 `/health`、`/ready` 和 `/solve`，CLI 与 HTTP 复用同一 application pipeline；OpenAPI 可确定性生成并由 CI 检测漂移；API HTTP 客户端区分合同错误、业务不可行、超时、取消、不可用、协议损坏和内部错误，Compose 生产路径不静默回退 CLI。

scheduler 使用独立非 root 镜像并设置 CPU、内存和进程数限制，API 镜像不再包含 Python、uv 或算法源码。第二阶段仍保留 API 进程内作业执行器和 Web 轮询，没有引入 Redis/BullMQ、Outbox Publisher、独立 Worker 或 SSE；HTTP 取消只中止调用方等待，不承诺强制停止已经进入 OR-Tools 的线程。

### 13.3 第三阶段：可靠任务与实时事件

目标是引入 Redis/BullMQ、Outbox Publisher、独立 Worker、幂等消费、重试/取消/超时和 SSE。阶段末在腾讯云进行私有试部署，取得真实资源与恢复证据。

截至 2026-07-13，第三阶段已经落地：API 以 PostgreSQL 事务提交作业和 outbox，独立 Publisher 使用稳定 ID 投递 BullMQ，独立 Worker 调用 FastAPI scheduler，并以 attempt、数据库 CAS 和唯一约束治理重复投递、重试、取消、超时、stalled 回收和迟到结果。Redis 使用 AOF 与 `noeviction`，仅承担分发和唤醒，不覆盖 PostgreSQL 事实。

SSE 已支持 PostgreSQL 历史补发、`Last-Event-ID`、Redis 唤醒回读、心跳和终态关闭；Web 已切换为 SSE 主路径，并仅在连接失败期间使用低频兜底查询。隔离 Compose 从空卷完成 API、Redis、Publisher、Worker 和 scheduler 故障恢复、重复 outbox、浏览器重连及完整 E2E。腾讯云私有试部署因未取得用户对目标主机、凭据和维护窗口的单独授权而未执行，本地证据不替代远程结论。

### 13.4 第四阶段：任务中心与约束策略基础

目标是形成面向用户的任务治理页面，持久化约束策略版本与运行快照，改进归一化评分和确定性诊断，但不扩张为完整方案实验室。

截至 2026-07-13，第四阶段已经落地：PostgreSQL 保存策略身份和不可变版本，并治理启停、当前版本、唯一默认版本与审计；作业创建事务解析策略并冻结版本化请求，Worker 不读取可变策略；同步排考和草稿重排也将实际调用策略与 scheduler 元数据传入运行持久化。运行结果保存同一版本、快照、scheduler 版本和评分合同版本。迁移为旧数据生成显式 legacy 快照，不伪造历史策略引用。

shared、Python 和 OpenAPI 已固定评分合同 v1 与结构化诊断；运营台新增作业筛选、分页、基于持久化 Attempt/事件的时间线、错误解释、运行跳转和策略治理区，任务内部信息仅管理员与排考员可读。真实 PostgreSQL/Redis/Compose 验证证明默认切换、策略禁用、重复 outbox、Worker stalled、scheduler 暂停恢复和 SSE 重连不会造成历史漂移或重复运行。第四阶段没有实施正式路由拆分、多策略批量实验或远程部署。

### 13.5 第五阶段：前端重构与多角色体验

目标是按已确认的信息架构拆分真实路由和角色工作区，完成专业视觉、矩阵拖拽、状态反馈、响应式和视觉验证。该阶段结束后才进入正式公开部署。

截至 2026-07-14，第五阶段已经落地：`0014` 迁移建立教师单作用域和学生多群体作用域，本人 API 只根据服务端会话解析对象；旧按 ID 查询不再匿名开放。登录、管理员概览、基础数据、作业、运行、草稿、策略、审计、教师本人和学生本人十个路由均可深链刷新，URL 保留筛选、分页、对比和选中上下文，401、403、404 与依赖失败有独立页面状态。

第四阶段的任务/SSE/策略合同直接迁入页面，没有复制状态机；草稿使用 dnd-kit pointer/keyboard sensor 并保留表单等价入口。全局 token、布局和表单样式已拆分，四种视口、6 张截图、Axe WCAG 2.1 A/AA 扫描、200% 文本和溢出检查进入 Playwright 门禁。独立 PostgreSQL/Redis Compose 的完整回归和故障演练证明页面重构未造成重复 attempt、事件或运行。

### 13.6 第六阶段：正式部署与交付收尾

目标是建立镜像发布、nginx/HTTPS、生产 Compose、资源限制、备份恢复、日志、线上 smoke 和核心 E2E，并在腾讯云完成有限开放与正式发布。

执行范围以 `docs/plan/第五版第六阶段计划.md` 为准。阶段必须先处置 CR-002，再建立可追溯镜像和生产配置；正式发布采用 `docs/design/第五版本地托管Runner发布设计.md` 的边界，由 GitHub 托管 job 执行质量门禁，本地 WSL Runner 构建并推送广州 TCR，北京服务器只按 digest 部署。远程预检、部署、域名/HTTPS、备份恢复、150 场基准、故障恢复和回滚均需要用户明确授权目标与维护窗口。第五版全部阶段完成后再执行一次全量代码审查。

截至 2026-07-15，正式制品 `f769725` 已在腾讯云主机按 digest 完成仅回环可见的内部部署验证：迁移、bootstrap、七项 readiness、两次无故障业务 smoke、热备与 COS 备份、可丢弃库恢复及固定 seed 的 50/100/150 场基准通过，验证后内部栈已停止。备案审核期间域名保持无解析，未修改 nginx、证书、防火墙或其他服务。Scheduler 冷恢复时约 3.35 秒即耗尽 3 次自动尝试，暴露 CR-028；正式域名 E2E、HTTPS、跨版本回滚和有限开放观察仍未完成，因此第五版尚不满足本节完成判定。

CR-028 采用共享、受约束的重试策略修复。Publisher 创建 BullMQ 作业和 Worker 执行 `JobExecutionService` 必须消费同一组 `maxAttempts` 与 `retryBaseDelayMs`，禁止分别保留硬编码默认值。生产默认最多尝试 6 次、指数退避基数 1 秒，对应 1、2、4、8、16 秒延迟和约 31 秒总恢复窗口；该窗口覆盖本次远端 Scheduler 冷启动，同时仍保持明确上限。环境配置必须校验正整数、最大尝试次数和最终退避不超过状态机 30 秒单次退避上限，生产 Compose 对 Publisher/Worker 注入同一值，预检拒绝缺失或越界配置。

修复先以 Worker 真实 PostgreSQL/Redis 红灯证明连续 3 次以上 `scheduler_unavailable` 后仍应恢复，再补配置、Compose 与预检合同。成功标准为恢复后的作业仅生成 1 个运行、attempt 和事件序列完整；本地通过不替代远端结论，必须生成新正式 digest 后重新执行 Scheduler 冷恢复演练，且启动本地 Runner、提交、推送或发布前仍需用户确认。

截至 2026-07-17，本地实现已新增共享重试策略，配置加载、Publisher 队列、Worker/`JobExecutionService`、生产 Compose 和预检均使用相同的 6 次尝试与 1000 ms 默认基数，并拒绝超过 30000 ms 最终单次退避的组合。真实 PostgreSQL/Redis 测试先证明旧 3 次上限无法进入第 4 次投递，再证明修复后第 4 次成功、4 个 attempt 且仅 1 个运行；Worker 全套 `18 passed`、部署合同在补充逐镜像构建边界、Docker Engine builder 和 systemd 稳定运维目录合同后为 `37 passed`，仓库级类型检查通过。工作流 `29482599323` 因首个 API 构建中的 npm 网络等待无法证明进度而取消；逐镜像边界提交 `11ab909` 的工作流 `29557441559` 又因远端 BuildKit bootstrap 无进展而在业务镜像构建前取消。当前本地方案已移除该远端启动依赖，但 CR-028 仍需新正式 digest 和腾讯云 Scheduler 冷恢复演练后才能关闭。

## 14. 不纳入第五版

以下方向保留价值，但不纳入第五版范围：

- 生产级多租户和学校间数据隔离。
- 学校 CAS、OIDC 或完整 SSO 集成。
- Kubernetes、自动扩缩容和多节点高可用部署。
- 复杂多级审批流和跨部门协作治理。
- 完整 CSV/Excel 导入质量中心与教务系统同步。
- 多策略批量实验室、Pareto 方案集和自动策略推荐。
- room-slot 与教师联合全局优化。
- 完整 CP-SAT 最小不可行核心分析。
- 大型 BI 平台、独立移动应用和 GraphQL/gRPC 重写。

这些能力分别适合后续的数据与规则平台、协作治理和高级决策版本。第五版以“可靠执行、真实身份、专业前端、真实部署”作为一致主线。

## 15. 完成判定

第五版完成需要同时取得以下证据：

1. API 重启不会丢失已提交作业。
2. Worker 崩溃后作业能够恢复、重试或进入明确终态。
3. 重复消息不会生成重复排考运行。
4. Redis 暂时不可用或数据丢失时，PostgreSQL 业务事实和审计仍完整。
5. scheduler 超时、业务不可行、合同错误和内部故障能够被正确区分。
6. SSE 断线重连后能够补发关键事件。
7. 所有关键操作能够追溯到真实用户、策略版本、作业和调度器版本。
8. 4 核 4 GB 腾讯云服务器能够稳定完成 150 场基准和完整业务闭环。
9. 前端在真实 PostgreSQL、Redis、正式域名和真实网络环境下通过核心 E2E 与视觉检查。
10. 正式环境完成备份恢复、镜像回滚和线上 smoke 验证。

满足上述条件后，ExamForge 才从第四版的可信演示基线升级为具有明确生产化边界的专业排考平台。
