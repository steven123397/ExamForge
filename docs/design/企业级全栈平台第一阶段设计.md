# ExamForge 企业级全栈平台第一阶段设计

## 1. 文档定位

本文档定义 ExamForge 从“算法原型”升级为“企业级排考运营平台”的第一阶段实现边界。它承接 `docs/design/总体设计与技术选型.md` 和 `docs/design/第一版实现内容设计.md`，但不再以课程最小演示为约束，而是以真实 SaaS/企业内管平台的产品形态推进。

当前仓库已经完成 `apps/scheduler/` 的 Python 排考算法核心，包括数据模型、测试数据生成、预检、冲突解释、CP-SAT 求解器、软约束评分和报告整理。第一阶段全栈平台要把这些能力包装成可视化、可查询、可审计、可扩展的业务系统。

## 2. 阶段目标

第一阶段目标是打通一条完整的企业级业务主线：

1. 后端拥有清晰的业务 API 和排考运行入口。
2. 数据层拥有 PostgreSQL/Drizzle schema、迁移和种子数据口径。
3. 前端拥有可直接演示的运营台，而不是页面壳。
4. 管理员可以查看考试批次、基础数据、排考运行、冲突解释、评分和资源利用率。
5. API 可以调用现有 Python scheduler，生成真实 `ScheduleResult`。
6. 本地环境可以用一组命令启动 Web、API 和数据库。

## 3. 非目标

本阶段不追求以下能力：

- 不实现真实登录、SSO、短信、邮件和学校教务系统集成。
- 不实现完整 CRUD 后台的所有编辑弹窗。
- 不实现复杂多租户计费、审批流和生产级权限矩阵。
- 不实现 Redis 队列和 WebSocket 进度推送。
- 不把 Python scheduler 重写成 TypeScript。

这些能力保留在后续企业化阶段。本阶段要先形成“真实系统感”和“可演示闭环”。

## 4. 总体架构

```text
浏览器
  |
  | HTTP
  v

  |

  |                     |
  |                     | child_process / JSON stdin/stdout
  |                     v

  |
```

## 5. Monorepo 结构

```text
ExamForge/
├── apps/
│   ├── api/                 # Fastify API，承载企业业务接口
│   ├── web/                 # Next.js 运营台
│   └── scheduler/           # 既有 Python 调度器
├── packages/
│   ├── db/                  # Drizzle schema、迁移、seed
│   └── shared/              # 共享领域类型、API DTO、演示数据
├── docs/
│   ├── design/
│   ├── plan/
│   └── status/
├── docker-compose.yml       # PostgreSQL 本地服务
├── package.json             # npm workspace
└── tsconfig.base.json
```

本项目使用 npm workspace，避免依赖本机额外安装 `pnpm`。后续如果团队统一使用 `pnpm`，可以在不改变业务结构的情况下迁移。

## 6. 数据层设计

`packages/db` 使用 Drizzle ORM 定义 PostgreSQL schema。第一阶段至少覆盖：

- `exam_batches`：考试批次、状态、考试周期、策略配置。
- `departments`：院系。
- `student_groups`：学生群体。
- `teachers`：教师与不可用时间。
- `courses`：课程与考试类型。
- `rooms`：考场、容量、类型、设备标签。
- `time_slots`：考试时间段。
- `exam_tasks`：待排考试任务。
- `schedule_runs`：排考运行记录、状态、评分、统计。
- `scheduled_exams`：排考结果。
- `conflict_records`：冲突解释。
- `audit_events`：关键操作审计。

API 第一阶段可以在无 `DATABASE_URL` 时使用内置演示数据运行，但 schema、迁移和 Docker PostgreSQL 必须存在，保证项目不只是内存 demo。

## 7. API 边界

`apps/api` 提供以下接口：

| 方法 | 路径 | 职责 |
| --- | --- | --- |
| `GET` | `/health` | 健康检查 |
| `GET` | `/api/dashboard` | 批次概览、资源统计、最近一次排考结果 |
| `GET` | `/api/reference-data` | 基础数据、考试任务和时间段 |
| `POST` | `/api/schedule-runs` | 发起一次排考运行 |
| `GET` | `/api/schedule-runs/:id` | 查看单次排考运行详情 |

API 的调度器调用通过 Python JSON 桥接完成：Node 进程将 `ScheduleInput` 写入 Python stdin，Python 输出结构化 JSON。这样保留 Python 算法能力，也让 Web/API 层不承载算法核心。

## 8. 前端信息架构

`apps/web` 不是 landing page，而是企业运营台首屏：

- 顶部：当前考试批次、运行状态、关键操作按钮。
- 左侧：模块导航，包括概览、基础数据、考试任务、排考运行、冲突解释、资源分析、审计。
- 主区域：
  - 指标带：考试数、考场数、教师数、时间段数、冲突数、评分。
  - 排考运行控制台：运行排考、查看状态、运行耗时、候选组合数。
  - 排考结果矩阵/列表：课程、学生群体、时间段、考场、监考教师。
  - 冲突解释面板：类型、影响对象、建议处理。
  - 资源利用率：考场利用率、教师工作量。

视觉风格采用“企业运营指挥台”：高信息密度、克制色彩、清晰网格、稳定尺寸、可扫描表格，不使用营销式 hero。

## 9. Scheduler 桥接

在 `apps/scheduler` 增加 JSON CLI：

```bash
uv run --python 3.12 --extra dev python -m examforge_scheduler.cli solve
```

输入：`ScheduleInput` JSON。
输出：包含 `assignments`、`conflicts`、`score`、`statistics`、`report` 的 JSON。

CLI 内部复用：

- `run_precheck()`
- `solve_schedule()`
- `detect_assignment_conflicts()`
- `calculate_score()`
- `build_schedule_report()`

## 10. 第一阶段验收标准

1. `apps/scheduler` 全量 pytest 通过。
2. `apps/api` 的核心 API 测试通过。
3. `apps/web` 可以构建通过。
4. 前端可以展示 demo 批次和基础数据。
5. 前端可以触发 API 排考运行。
6. API 可以调用 Python scheduler 并返回真实结果。
7. 数据库 schema、迁移和 Docker PostgreSQL 配置存在。
8. `docs/index.md` 和 `docs/status/project_status.md` 与当前状态一致。

## 11. 风险与取舍

- Python 与 Node 通过进程调用集成，部署形态比纯 Node 单体复杂，但能保留 OR-Tools 与现有算法资产。
- 第一阶段允许 API 在无数据库连接时使用内置演示仓储，但不能缺失 PostgreSQL schema 和迁移。
- 前端第一版优先展示真实业务闭环，不做完整编辑体验。
- 企业级不等于一次性做完所有基础设施；本阶段用清晰边界换取可运行、可扩展、可演示。
