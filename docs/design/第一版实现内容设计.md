# 高校智能排考与考务资源调度系统第一版实现内容设计

## 1. 文档定位

本文档定义 ExamForge 第一版实现内容。它承接 `docs/design/总体设计与技术选型.md` 的总体方向，将“TypeScript Web 平台 + Python OR-Tools 算法服务 + PostgreSQL 数据库”的长期方案拆成第一轮可实现、可验证、可演示的内容边界。

第一版实现不是课程最终系统的全部功能，也不是单纯算法脚本。它的目标是先打通排考系统最关键的闭环：数据建模、测试数据生成、自动排考、冲突解释、结果查看和课程报告素材沉淀。

## 2. 第一版目标

第一版实现目标如下：

1. 证明排考问题可以用结构化数据和约束模型表达。
2. 证明核心硬约束可以被自动排考算法处理。
3. 证明系统可以输出排考结果、冲突列表、软约束评分和运行统计。
4. 证明基础数据、考试任务、排考结果之间的数据关系可以支撑后续数据库设计和 E-R 图。
5. 形成可用于课程报告的算法流程、测试用例、效果对比和总体设计材料。
6. 为后续 Web 平台实现保留清晰接口，而不是把算法逻辑写死在页面或脚本里。

## 3. 第一版范围

### 3.1 纳入第一版

第一版纳入以下内容：

- Python 排考算法原型。
- 结构化测试数据生成器。
- 核心数据模型定义。
- 硬约束检测与自动排考。
- 软约束评分。
- 冲突预检与冲突解释。
- 算法运行统计。
- 算法接口输入输出契约。
- 小规模与中规模测试样例。
- 面向课程报告的算法流程和测试结果材料。

### 3.2 第一版不纳入

以下内容不作为第一版硬性交付：

- 完整 Next.js 管理后台。
- 完整 Fastify API 服务。
- PostgreSQL 迁移和持久化数据库。
- BullMQ、Redis、WebSocket 或 SSE。
- 拖拽式排考矩阵。
- 云端部署。
- 真实学校教务系统集成。

这些内容仍属于总体设计的一部分，但应在算法原型通过后进入第二阶段。

## 4. 第一版实现形态

第一版建议以 `apps/scheduler/` 为核心实现算法原型，并用测试和命令行脚本验证能力。目录结构建议如下：

```text
apps/scheduler/
├── examforge_scheduler/
│   ├── __init__.py
│   ├── models.py              # 输入输出数据模型
│   ├── generator.py           # 测试数据生成器
│   ├── precheck.py            # 求解前确定性预检
│   ├── solver.py              # OR-Tools CP-SAT 求解
│   ├── scoring.py             # 软约束评分
│   ├── conflicts.py           # 冲突解释
│   └── report.py              # 运行统计和报告数据整理
├── tests/
│   ├── test_generator.py
│   ├── test_precheck.py
│   ├── test_solver_hard_constraints.py
│   ├── test_scoring.py
│   └── test_conflicts.py
└── pyproject.toml
```

该结构只描述第一版算法原型。Web、API 和数据库目录可在第二阶段创建。

## 5. 核心数据模型

第一版不直接落 PostgreSQL 表结构，但模型命名应与后续数据库设计保持一致。

### 5.1 输入实体

| 模型 | 关键字段 | 说明 |
| --- | --- | --- |
| `StudentGroup` | `id`、`name`、`size`、`departmentId` | 学生群体，可以是班级或考试群体 |
| `Teacher` | `id`、`name`、`departmentId`、`unavailableSlotIds` | 教师与不可用时间 |
| `Course` | `id`、`name`、`departmentId`、`examType` | 课程基础信息 |
| `Room` | `id`、`name`、`buildingId`、`capacity`、`roomType`、`equipmentTags` | 考场 |
| `TimeSlot` | `id`、`date`、`startTime`、`endTime`、`periodIndex` | 考试时间段 |
| `ExamTask` | `id`、`courseId`、`studentGroupIds`、`expectedCount`、`durationMinutes`、`requiredRoomType`、`requiredEquipmentTags`、`allowedSlotIds`、`invigilatorCount` | 待排考试任务 |
| `ConstraintProfile` | `hardRules`、`softWeights`、`timeLimitSeconds` | 约束配置 |

### 5.2 输出实体

| 模型 | 关键字段 | 说明 |
| --- | --- | --- |
| `ScheduledExam` | `examTaskId`、`roomId`、`timeSlotId`、`teacherIds` | 单场考试安排 |
| `ConflictRecord` | `type`、`severity`、`affectedIds`、`message`、`suggestion` | 冲突解释 |
| `ScoreBreakdown` | `totalScore`、`hardViolationCount`、`softPenaltyItems` | 评分明细 |
| `SolverStatistics` | `status`、`elapsedMs`、`examCount`、`roomCount`、`slotCount`、`attemptedAssignments` | 运行统计 |
| `ScheduleResult` | `assignments`、`conflicts`、`score`、`statistics` | 排考总输出 |

## 6. 约束模型

### 6.1 必须实现的硬约束

第一版必须实现以下硬约束：

1. 每场考试必须且只能安排到一个考场和一个时间段。
2. 同一考场在同一时间段最多安排一场考试。
3. 同一学生群体在同一时间段最多参加一场考试。
4. 考场容量必须大于或等于考试人数。
5. 考场类型必须满足考试任务要求。
6. 考场设备标签必须覆盖考试任务所需设备。
7. 考试时间段必须属于该任务允许的时间段。
8. 监考教师不能被安排在不可用时间段。
9. 同一教师在同一时间段最多监考一场考试。

### 6.2 第一版软约束

第一版实现以下软约束评分：

| 编号 | 软约束 | 评分方式 |
| --- | --- | --- |
| SC-1 | 避免同一学生群体连续考试 | 同一学生群体相邻时间段连续考试时扣分 |
| SC-2 | 平衡教师监考工作量 | 教师监考次数偏离平均值时扣分 |
| SC-3 | 提高考场容量利用率 | 容量利用率过低时扣分 |
| SC-4 | 考试分布均衡 | 某一日期考试过度集中时扣分 |

跨校区移动、课程偏好时间段等软约束可作为第二阶段增强项。

## 7. 求解流程

第一版求解流程如下：

1. 读取或生成测试数据。
2. 执行确定性预检。
3. 若预检发现不可行数据，直接输出冲突报告。
4. 构建 CP-SAT 布尔决策变量 `x[exam_id, room_id, slot_id]`。
5. 添加硬约束。
6. 添加软约束惩罚变量。
7. 设置求解时间限制。
8. 执行求解。
9. 提取考试安排。
10. 分配或校验监考教师。
11. 执行最终冲突检测。
12. 计算软约束评分。
13. 输出 `ScheduleResult`。

## 8. 冲突解释设计

第一版冲突解释优先采用确定性规则，不承诺完整最小冲突集。

### 8.1 预检冲突

预检阶段输出以下冲突：

- `capacity_impossible`：某考试人数超过所有候选考场容量。
- `no_available_room`：没有满足类型和设备要求的考场。
- `no_allowed_slot`：考试任务没有可用时间段。
- `student_group_overloaded`：某学生群体考试数量超过可用时间段数量。
- `teacher_unavailable`：候选监考教师在所有候选时间段均不可用。

### 8.2 求解后冲突

求解后输出以下冲突：

- `unscheduled_exam`：考试任务未被安排。
- `room_time_conflict`：同一考场同一时间段出现重复安排。
- `student_group_clash`：同一学生群体同一时间段有多场考试。
- `teacher_time_clash`：同一教师同一时间段被重复安排。
- `room_capacity_mismatch`：考场容量不足。
- `room_requirement_mismatch`：考场类型或设备不满足要求。

每条冲突应包含业务化说明和处理建议，避免只暴露技术字段。

## 9. 测试数据设计

第一版测试数据生成器必须支持固定随机种子，保证测试可复现。

### 9.1 小规模数据

用于快速演示和单元测试：

- 6-10 场考试。
- 3-5 个学生群体。
- 4-6 名教师。
- 4-6 个考场。
- 4-6 个时间段。

预期结果：可以在短时间内生成无硬冲突方案。

### 9.2 中规模数据

用于常规算法测试：

- 30-80 场考试。
- 15-30 个学生群体。
- 20-40 名教师。
- 10-20 个考场。
- 10-20 个时间段。

预期结果：可以在限定时间内生成可行方案或解释不可行原因。

### 9.3 冲突数据

用于异常场景测试：

- 考试人数超过全部考场容量。
- 学生群体考试数量超过可用时间段。
- 教师不可用时间覆盖所有候选时间段。
- 设备要求无法被任何考场满足。

预期结果：系统输出结构化冲突，不应静默失败。

## 10. 验收标准

第一版完成后应满足以下验收标准：

1. 能通过命令运行小规模和中规模排考样例。
2. 小规模可行数据能生成无硬冲突排考方案。
3. 至少 4 类不可行数据能输出结构化冲突解释。
4. 排考结果包含软约束评分和运行统计。
5. 测试数据生成器支持固定随机种子。
6. 算法核心有自动化测试覆盖。
7. 输出数据结构能映射到后续数据库实体。
8. 产出的算法流程和测试结果能写入课程报告。

## 11. 后续衔接

第一版完成并验证后，进入第二阶段：

1. 编写 `docs/design/数据模型与排考约束模型.md` 的字段级版本。
2. 创建 PostgreSQL schema 和迁移。
3. 创建 Fastify API，封装算法服务调用。
4. 创建 Next.js 管理端，实现基础数据、考试任务和排考结果查看。
5. 将第一版命令行算法输出接入 Web 界面。

## 12. 当前结论

第一版实现应聚焦算法与模型原型，而不是直接搭建完整全栈。这样既能最快验证项目技术核心，也能为课程报告提供高质量算法说明、测试材料和系统设计依据。
