# 高校智能排考与考务资源调度系统第四版实现内容设计

## 1. 文档定位

本文档定义 ExamForge 第四版的设计方向、架构边界和候选验收口径。它吸收第三版第四阶段后高阶架构与算法评审中的有效结论，但不是活动实施计划，也不替代 `docs/status/code_review_status.md` 的问题状态维护。

第四版的核心目标不是继续堆叠运营台功能，而是把第三版形成的工程闭环继续提升为更可信的排考平台基线：

1. 收敛 PostgreSQL 数据源和跨模块合同，降低 JSONB 与关联表并存带来的演进风险。
2. 明确 API route、service、repository 的职责边界，降低 `app.ts` 承载过多业务编排的回归风险。
3. 让算法能力、评分报告和 Web 表达保持一致，避免把后处理启发式包装成全局优化能力。
4. 用规模基准、迁移验证和真实 PostgreSQL 测试证明系统上限，而不是只依赖内存仓储和小规模演示数据。
5. 按投入产出比选择认证、队列、实时进度、调度器服务化和部署复现能力，不一次性企业化扩张。
6. 在第四版最后建立 CI 质量门禁，把类型检查、测试、构建、迁移和 E2E 从人工约定升级为自动验证。

第四版每个实施阶段应另行在 `docs/plan/` 编写活动计划，明确文件范围、验证命令和不做清单。

## 2. 当前基线

截至第四版第一阶段完成后，系统已具备以下能力：

1. `apps/web` 已从单体运营台拆出 `features/`、共享组件、API client 和 TanStack Query 基础设施。
2. `apps/api` 已提供 Fastify API、演示 Bearer token 鉴权、同步排考、异步作业、草稿治理、发布回滚、审计、通知预览和 CSV 导出；排考、草稿和发布流程已进入独立 service/use-case 层。
3. `apps/scheduler` 已实现 Python OR-Tools CP-SAT 排考、预检、冲突解释、软约束目标、评分和报告整理，并通过 JSON CLI 被 API 调用。
4. `packages/shared` 已承载 Zod DTO、排考输入输出合同、`fixed_assignments` 和前后端共享类型。
5. `packages/db` 已实现 Drizzle schema、手写 SQL 迁移、迁移状态表、seed、迁移检查和 PostgreSQL 仓储；关键关联表已具备主外键和 JSONB 双向一致性验证。

第三版完成后的全量代码审查已修复一批确定性问题，包括 CSV 导出鉴权和审计、Web Query 状态镜像、异步作业启动恢复、迁移检查误导性断言、API `fixed_assignments` 注入，以及 PostgreSQL 关联数据读取问题。第四版第一阶段进一步补齐了真实数据库证据和并发终态治理。

当前仍待解决的问题以 `docs/status/code_review_status.md` 为准。第四版设计只吸收其中具有长期架构价值的方向，不把所有存留问题都默认列为第四版必做。

## 3. 主要架构上限

### 3.1 数据源仍处于兼容期

第三版新增了 `teacher_unavailable_slots`、`exam_task_student_groups`、`scheduled_exam_invigilators` 和 `draft_exam_invigilators` 关联表，但 JSONB 字段仍作为兼容快照存在。第四版第一阶段已将关键读路径改为关联表优先、JSONB 回退，并补齐双向一致性检查；系统仍需确定最终删除兼容字段的时机。

第四版需要明确：

- 哪些字段以关联表为唯一事实源。
- 哪些 JSONB 字段只保留为兼容快照。
- 是否以及何时通过迁移删除冗余 JSONB 字段。
- 迁移验证是否覆盖回填正确性、外键、唯一约束、索引和关联数据一致性。

如果继续长期双写，必须补齐一致性检查和回填测试；否则后续任一导入、迁移或手工修复都可能制造调度输入与数据库事实不一致。

### 3.2 API 路由层仍承担较多业务编排

第四版第一阶段已提取排考、草稿和发布治理 service，`apps/api/src/app.ts` 主要保留 HTTP 入参解析、鉴权、状态码和错误响应。基础数据等剩余流程仍可按收益继续收敛，但不再以机械拆文件为目标。

第四版应逐步形成三层边界：

```text
routes
  只处理 HTTP 入参、鉴权、状态码和错误响应
services / use cases
  处理业务编排、事务边界、调度器调用、审计和跨仓储流程
repository
  只处理数据访问和持久化映射
```

优先提取草稿管理、作业调度和发布治理 service。不要为了“分层”做无行为变化的大搬迁；每个 service 提取都应配套最窄测试。

### 3.3 异步作业持久化了状态，但执行仍在 API 进程内

当前 `schedule_jobs` 已持久化，API 启动时也会将历史 `queued` / `running` 作业标记为失败，但新作业执行仍由 API 进程内 `setTimeout()` 触发。

这意味着当前系统适合单实例演示，不具备完整队列语义：

- 多实例部署下无法保证只执行一次。
- 进程崩溃后只能失败恢复，不能恢复执行。
- 没有取消、重试、超时策略和细粒度进度事件。

第四版不必默认引入 BullMQ、Redis 或其他队列，但必须在设计和报告中诚实标注单实例边界。如果第四版选择“真实作业系统”为主线，应先设计队列状态机、幂等键、重试策略和取消语义，再做 SSE 或 WebSocket。

### 3.4 Web 运营台已拆分，但 workflow 状态仍集中

第三版拆分后，`OperationsConsole` 已大幅收敛，但页面仍承担角色、当前草稿、局部表单、跨面板刷新和若干 workflow 编排。

第四版可按收益选择：

- 将当前面板、草稿 ID、筛选条件等同步到 URL query，刷新后保留上下文。
- 系统化 `useMutation + invalidateQueries`，减少手动 `refetch()`。
- 为草稿编辑、发布、回滚等高风险 workflow 增加局部错误边界和浏览器级测试。
- 继续拆小主组件，但不把页面拆成过多仅转发 props 的碎片。

### 3.5 鉴权、调度器服务化和实时进度仍是演示边界

当前 Bearer token 是演示级鉴权，调度器仍由 API 通过 JSON stdin/stdout CLI 调用，排考进度仍通过 HTTP 轮询获取。这些都是明确的企业化缺口，但不一定都应成为第四版主线。

第四版应按场景取舍：

- 如果目标是课程演示和本地闭环，保留演示 token、CLI 调度器和轮询，重点补强算法、数据一致性和验证。
- 如果目标是多用户或接近生产的教务场景，真实用户、会话、审计 actor、队列、实时进度和调度器服务化需要成组设计，不能零散添加。

## 4. 主要算法上限

### 4.1 CP-SAT 当前主要求解 room-slot

当前调度器以 `x[exam_task_id, room_id, slot_id]` 为核心决策变量，硬约束覆盖唯一安排、考场时段唯一、学生群体冲突、候选容量、考场类型、设备、允许时段和固定安排。

软目标已进入 CP-SAT，但主要覆盖 room-slot 维度：

- 考场利用率。
- 学生连续考试。
- 考试分布均衡。

这比第二版的后置评分更强，但仍不能覆盖教师、跨楼栋、课程偏好等依赖额外变量或主数据的目标。

### 4.2 教师分配是后处理负载感知，不是全局优化

当前教师分配发生在 room-slot 求解之后。实现会按时间段和考试顺序遍历已安排考试，并按当前负载和教师 ID 顺序选择满足不可用时间、同时间段占用和固定教师约束的监考教师。

因此系统可以称为“负载感知教师分配”，但不应称为“教师工作量全局优化”。如果第四版要把教师工作量平衡作为产品卖点，需要至少选择一种更强策略：

1. 二阶段 CP-SAT：固定 room-slot，新增教师-考试变量，优化教师工作量方差、连续监考和不可用约束。
2. 最小费用匹配：在每个时间段或全局窗口内进行教师分配，成本函数覆盖负载、连续监考和楼栋移动。
3. 联合求解：把教师变量纳入主 CP-SAT，一次性优化 room-slot 和教师分配，但要先评估变量规模和求解时间。

### 4.3 `fixed_assignments` 是强制安排，不等于增量重排

`fixed_assignments` 当前表达的是“某场考试必须固定到指定 room-slot，可选固定教师”。它是有价值的硬约束能力，但还不等于真实增量重排。

真实增量重排至少还需要：

- 基准方案：来自已发布方案、某次运行结果或草稿。
- 冻结范围：哪些考试、考场、时段或教师不可变。
- 候选空间：只允许哪些考试或资源参与重排。
- 稳定性目标：最小化相对基准方案的 room-slot 和教师变动。
- 解释输出：哪些考试被保留，哪些被重排，为什么重排。

第四版如果继续推进草稿锁定和局部再平衡，应把锁定考试转换为调度器可理解的固定安排，并新增稳定性评分；如果不做，应在 UI 和文档中避免把固定安排称为完整增量调度。

### 4.4 缺少规模基准和性能上限

现有测试以小规模演示数据为主。CP-SAT 变量数量近似为 `考试数 × 考场数 × 时段数`，软约束还会继续增加辅助变量。第四版应补充 50、100、150 场考试规模的基准输入，记录：

- 求解时间。
- 是否找到可行解或最优解。
- 软目标得分变化。
- 变量数量和约束数量。
- 时间限制下结果质量是否稳定。

规模基准成本低、价值高，应优先于复杂队列、SSO 或大规模 BI 能力。

### 4.5 冲突解释仍以确定性诊断为主

当前冲突解释主要覆盖预检和求解后校验。CP-SAT 不可行时，系统还缺少最小冲突集或更强的“如何修改数据”的诊断。

第四版可以增强诊断，但不建议把完整最小冲突集分析作为主线。更务实的方向是先扩展确定性预检和建议字段，让容量、时段窗口、教师不可用、学生群体冲突和固定安排冲突有更具体的修复建议。

## 5. 第四版推荐主线

第四版建议以“数据一致性收敛、服务层治理、算法能力对齐和规模验证”为主线。

### 5.1 数据一致性收敛

设计目标：

- 调度输入优先来自关系化事实表。
- JSONB 冗余字段只作为兼容快照或被后续迁移移除。
- PostgreSQL 集成测试覆盖关联表读写、回填和迁移幂等。
- 内存仓储测试不替代真实 PostgreSQL 结论。

最小验收口径：

- `getReferenceData()`、发布查询和草稿查询的关键关联数据均有真实 PostgreSQL 测试。
- 迁移检查覆盖关联表、主键、外键、关键唯一约束和回填数据。
- 文档明确 JSONB 字段的保留原因和删除条件。

### 5.2 API service 层治理

设计目标：

- routes 只保留 HTTP 边界。
- services 承载草稿管理、作业调度、发布治理和调度器调用。
- repository 不再承载业务流程，只做数据访问。
- service 层可以脱离 Fastify request 上下文做单元测试。

最小验收口径：

- 草稿创建、调整、校验、锁定、发布和废弃进入 `draft-service` 或等价 use case。
- 同步排考和异步作业进入 `schedule-run-service` 或等价 use case。
- 发布、回滚、通知预览和 CSV 导出进入发布治理 service。
- `app.ts` 的直接 `repository.*` 调用明显减少，关键业务分支有 service 测试。

### 5.3 算法能力与表达对齐

设计目标：

- 文档、报告和 UI 不夸大当前算法能力。
- 教师工作量如果只是后处理负载感知，就用准确措辞描述。
- 如果第四版升级教师分配，则用测试证明其优于贪心基线。
- 评分报告与 CP-SAT 优化目标保持可解释映射。

最小验收口径：

- 新增教师分配基准测试，比较贪心与新策略的工作量方差或最大负载。
- 新增规模基准测试，输出 50、100、150 场考试的求解时间和状态。
- 报告和 Web 文案与算法事实一致。

### 5.4 增量重排语义

设计目标：

- `fixed_assignments` 继续作为底层强制安排合同。
- 草稿锁定、局部再平衡和未来增量重排使用更明确的基准方案和冻结范围。
- 稳定性目标进入评分或 CP-SAT 软约束。

最小验收口径：

- 锁定考试可转换为调度器输入并在重排后保持不变。
- 非锁定考试可以在限定候选空间内重排。
- 输出中能说明保留、变更和冲突原因。

### 5.5 演示与部署可复现

设计目标：

- 本地演示不依赖口头步骤和隐式环境。
- PostgreSQL、API、Web 和 scheduler CLI 的启动方式清晰。
- Docker 或脚本入口能降低课堂演示失败概率。
- CI 在干净环境中复现核心验证，不依赖开发者或 Agent 手工记忆全部命令。

最小验收口径：

- 文档明确 PostgreSQL、Docker 和代理依赖。
- 有一条从空库迁移、seed、启动 API/Web、运行核心排考的可复现路径。
- 环境不可用时，验证命令能给出明确错误，而不是静默降级到内存仓储。
- CI 自动覆盖类型检查、API 与 scheduler 测试、生产构建、真实 PostgreSQL 集成、迁移幂等和浏览器 E2E。

## 6. 候选方向优先级

| 候选方向 | 价值 | 成本 | 建议 |
| --- | --- | --- | --- |
| PostgreSQL 单一事实源收敛 | 消除双写和读路径漂移，支撑跨资源查询 | 中 | 第四版高优先级 |
| API service 层系统化 | 降低 `app.ts` 回归风险，提高业务测试效率 | 中 | 第四版高优先级 |
| 规模基准测试 | 低成本量化算法上限 | 低 | 第四版高优先级 |
| 教师二阶段优化或最小费用匹配 | 让教师工作量平衡有算法证明 | 中到高 | 有展示价值时纳入 |
| 增量重排真实语义 | 连接草稿锁定、固定安排和稳定性目标 | 中 | 可作为算法阶段重点 |
| Web URL 状态和 mutation hooks | 改善运营台可用性和可测试性 | 低到中 | 可与 service 治理并行小步做 |
| Docker Compose 一键演示 | 降低演示环境风险 | 低 | 建议纳入 |
| CI 质量门禁 | 固化跨语言、真实数据库和浏览器验证口径 | 低到中 | 第四版最后阶段完成 |
| 真实认证授权 | 提升生产化真实性 | 高 | 课程交付可选 |
| BullMQ/Redis 持久化队列 | 支撑多实例、重试和取消 | 高 | 队列语义成为主线时再做 |
| SSE 或 WebSocket | 改善进度体验 | 中 | 建议等队列或进度模型明确后再做 |
| FastAPI scheduler 服务化 | 支撑独立部署和健康检查 | 中 | 单机演示可暂缓 |
| OpenAPI/SDK 生成 | 强化合同治理 | 中 | API/service 边界稳定后再做 |

## 7. 建议阶段边界

### 7.1 第一阶段：数据与服务边界

目标是降低最影响后续演进的结构风险。

建议内容：

- 完成 PostgreSQL 关联表读路径收敛和迁移验证。
- 提取草稿、作业和发布治理 service。
- 保留演示鉴权、CLI 调度器和 HTTP 轮询，不在本阶段扩张。

建议验证：

```bash
npm run typecheck
LOG_LEVEL=silent npm test
TEST_DATABASE_URL=postgres://examforge:examforge@localhost:5432/examforge_test npm run test:postgres
TEST_DATABASE_URL=postgres://examforge:examforge@localhost:5432/examforge_test npm run test:migrations
DATABASE_URL=postgres://examforge:examforge@localhost:5432/examforge_test npm run db:migrate
git diff --check
```

### 7.2 第二阶段：算法升级与规模验证

目标是让算法主张可被测试证明。

建议内容：

- 新增规模基准数据生成器和 50、100、150 场考试测试。
- 选择教师二阶段优化或最小费用匹配，并与贪心基线对比。
- 定义增量重排的基准方案、冻结范围和稳定性目标。

建议验证：

```bash
npm run test:scheduler
LOG_LEVEL=silent npm test
npm run typecheck
```

如果新增性能基准，应在测试输出或文档中记录规模、时间限制和结果质量，不只断言“通过”。

### 7.3 第三阶段：演示环境与体验增强

目标是降低演示风险，并按需要选择企业化能力。

建议内容：

- 优先补 Docker Compose 或脚本化本地演示路径。
- 按需要选择真实认证、SSE/WebSocket 或 scheduler 服务化中的一项，不建议三项同时推进。
- 为新增体验能力补浏览器级 E2E。

建议验证：

```bash
npm run build
npm run test:e2e
git diff --check
```

如涉及 Docker 或 PostgreSQL，必须使用真实服务验证，不得以内存仓储结果替代。

### 7.4 第四阶段：CI 与交付收尾

目标是把第四版已经稳定的验证口径固化为自动质量门禁，不在此阶段继续扩展业务功能。

建议内容：

- 建立快速检查，自动运行类型检查、API 单元测试、scheduler 测试、生产构建和 `git diff --check`。
- 在干净的 PostgreSQL 16 服务上执行空库迁移、迁移幂等检查、迁移/session 测试和 PostgreSQL 集成测试。
- 启动 API 与 Web 后运行 Playwright E2E，覆盖核心排考和草稿工作流。
- 检查禁止提交的构建产物、缓存和 `.codegraph/`；可确定的文档链接与提交格式规则逐步脚本化。
- 按执行成本拆分快速门禁和完整门禁：快速门禁用于日常提交，完整门禁用于 PR、`main` 或手动触发。

## 8. 不建议纳入第四版的方向

以下方向有价值，但不建议在第四版投入：

- 生产级多租户：当前单批次假设贯穿全栈，改造成本过高。
- GraphQL 或 gRPC 替代 REST：当前 REST + Zod 合同足够，迁移收益不足。
- 大规模 BI 看板：运营台基础统计已能支持课程演示。
- 完整 SSO/CAS：依赖外部系统，难以闭环验证。
- 复杂多级审批流：偏离排考核心，容易引入大量外部状态。
- 完整 CP-SAT 最小冲突集分析：实现复杂，建议先增强确定性诊断。
- 移动端深度适配：运营台面向管理后台，移动端不是核心路径。

## 9. 后续规划输入摘要

后续如需让高阶模型继续评审，应重点输入以下问题，而不是重复低级已修复缺陷：

1. PostgreSQL 关系化事实表、JSONB 兼容字段和 shared DTO 应如何收敛，迁移顺序和验证策略是什么。
2. API service/use-case 层应按哪些业务边界拆分，如何避免只做文件搬运。
3. 教师分配应选择二阶段 CP-SAT、最小费用匹配还是继续负载感知贪心，如何用测试证明收益。
4. `fixed_assignments`、草稿锁定和增量重排之间应如何定义基准方案、冻结范围、候选空间和稳定性目标。
5. 第四版在队列、实时进度、真实认证、scheduler 服务化和演示环境可复现之间如何取舍，哪些应明确暂缓。
