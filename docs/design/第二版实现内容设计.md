# 高校智能排考与考务资源调度系统第二版实现内容设计

## 1. 文档定位

本文档定义 ExamForge 第二版实现内容。它承接 `docs/design/第一版实现内容设计.md` 和 `docs/design/总体设计与技术选型.md`，面向当前已经完成的算法原型、Web 运营台、Fastify API、PostgreSQL 持久化仓储和发布查询闭环，继续推进到“人机协同排考运营平台”。

第一版的核心价值是证明排考算法、结构化数据、冲突解释和评分可行。第二版的核心价值是证明系统不只是一次性自动排考工具，而是支持真实教务运营流程：管理员可以生成方案、人工调整、即时校验、保留草稿、比较版本、发布方案、回滚方案，并让教师和学生查询已发布结果。

## 2. 当前基线

第二版以当前 `main` 的能力作为基线：

1. `apps/scheduler/` 已实现 Python OR-Tools 排考原型、预检、求解、冲突解释、评分和报告整理。
2. `apps/api/` 已实现 Fastify API，并通过 JSON stdin/stdout 调用 Python 调度器。
3. `packages/db/` 已实现 PostgreSQL schema、迁移、seed 和持久化仓储。
4. `apps/web/` 已实现企业运营台，可查看基础数据、运行排考、查看结果、冲突、资源利用率和教师工作量。
5. 系统已支持基础数据创建、更新、删除、批量导入，排考运行历史、审计、版本对比、发布、回滚和教师/学生已发布安排查询。

第二版设计不推翻这些能力，而是在其上增加更接近真实排考系统的版本治理和人工干预能力。

## 3. 第二版目标

第二版实现目标如下：

1. 支持从自动排考结果创建可编辑草稿，草稿与已发布方案隔离。
2. 支持管理员人工调整考试的时间段、考场和监考教师。
3. 支持每次调整后即时执行硬约束校验，并输出结构化冲突解释。
4. 支持草稿版本保存、对比、审计和发布。
5. 支持以时间段 × 考场矩阵查看和调整排考结果。
6. 支持发布前的评分、冲突、变更摘要和确认流程。
7. 为课程报告补充人机协同、版本管理、冲突校验和系统维护相关材料。

## 4. 第二版范围

### 4.1 纳入第二版

第二版纳入以下内容：

- 排考方案草稿模型。
- 从 `schedule_runs` 复制生成草稿。
- 草稿中单场考试安排的修改接口。
- 修改后的硬约束冲突校验。
- 草稿评分和冲突摘要。
- 草稿与原始运行或已发布版本的差异对比。
- 时间段 × 考场排考矩阵。
- 草稿保存、废弃、发布和审计。
- 发布前确认流程。
- 面向课程报告的流程图、状态图和测试材料。

### 4.2 不纳入第二版

以下内容不作为第二版硬性交付：

- 真实登录、SSO、短信、邮件和教务系统集成。
- 生产级多租户、计费和复杂权限矩阵。
- Redis 队列、BullMQ、WebSocket 或 SSE 的完整生产实现。
- 复杂拖拽交互的全部边界体验。
- 自动求解器的全量重新建模。
- 云端部署和生产监控体系。

这些内容可作为第三版或后续企业化增强项。第二版优先保证核心运营流程闭环。

## 5. 核心业务流程

### 5.1 自动排考到草稿

```text
基础数据
  |
  v
发起自动排考
  |
  v
生成 schedule_run
  |
  v
复制为 schedule_draft
  |
  v
人工调整与校验
  |
  v
发布或废弃
```

自动排考结果仍由 `apps/scheduler` 生成。草稿不是求解器输出的覆盖写入，而是一个可编辑工作副本。这样可以保留自动求解版本、人工调整版本和已发布版本之间的关系。

### 5.2 人工调整闭环

管理员在 Web 矩阵中选择一场考试后，可以调整：

- 时间段。
- 考场。
- 监考教师。

API 接收到调整后执行：

1. 更新草稿内对应安排。
2. 调用共享冲突检测逻辑校验硬约束。
3. 重新计算评分摘要。
4. 写入审计事件。
5. 返回最新草稿、冲突列表和评分。

Web 端不承载约束判断，只负责展示校验结果和交互状态。

### 5.3 发布确认

发布草稿前，系统展示：

- 草稿来源。
- 当前评分。
- 硬冲突数量。
- 与原始运行或当前发布版本的差异摘要。
- 最近调整审计事件。

默认策略是：存在硬冲突时不允许直接发布。后续可以增加“带确认发布”能力，但第二版先采用更清晰的安全规则。

## 6. 数据模型设计

第二版建议新增以下持久化实体。

### 6.1 `schedule_drafts`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | text | 草稿 ID |
| `batch_id` | text | 所属考试批次 |
| `source_run_id` | text | 来源自动排考运行 |
| `base_published_run_id` | text nullable | 创建草稿时的已发布版本 |
| `status` | enum | `editing`、`validated`、`blocked`、`published`、`discarded` |
| `score` | integer | 当前评分 |
| `conflict_count` | integer | 当前冲突数量 |
| `assignment_count` | integer | 当前安排数量 |
| `created_by` | text | 创建人，第二版可固定为 `system` 或 `admin` |
| `created_at` | timestamp | 创建时间 |
| `updated_at` | timestamp | 更新时间 |

### 6.2 `draft_scheduled_exams`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | text | 草稿安排 ID |
| `draft_id` | text | 所属草稿 |
| `exam_task_id` | text | 考试任务 |
| `room_id` | text | 考场 |
| `time_slot_id` | text | 时间段 |
| `teacher_ids` | jsonb | 监考教师 |
| `updated_at` | timestamp | 更新时间 |

### 6.3 `draft_conflict_records`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | text | 冲突记录 ID |
| `draft_id` | text | 所属草稿 |
| `type` | text | 冲突类型 |
| `severity` | enum | `error` 或 `warning` |
| `affected_ids` | jsonb | 影响对象 |
| `message` | text | 业务化说明 |
| `suggestion` | text | 处理建议 |

### 6.4 `draft_change_events`

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | text | 变更事件 ID |
| `draft_id` | text | 所属草稿 |
| `exam_task_id` | text | 被调整的考试任务 |
| `before` | jsonb | 调整前安排 |
| `after` | jsonb | 调整后安排 |
| `actor` | text | 操作人 |
| `created_at` | timestamp | 操作时间 |

`audit_events` 继续作为全局审计日志；`draft_change_events` 用于草稿内部的高频调整明细。

## 7. API 设计

第二版 API 保持 REST 风格，继续由 `apps/api` 承载业务边界。

| 方法 | 路径 | 职责 |
| --- | --- | --- |
| `POST` | `/api/schedule-runs/:id/drafts` | 从某次排考运行创建草稿 |
| `GET` | `/api/schedule-drafts` | 查看草稿列表 |
| `GET` | `/api/schedule-drafts/:id` | 查看草稿详情 |
| `PATCH` | `/api/schedule-drafts/:id/assignments/:examTaskId` | 调整单场考试安排 |
| `POST` | `/api/schedule-drafts/:id/validate` | 重新校验草稿 |
| `GET` | `/api/schedule-drafts/:id/compare` | 对比草稿与来源运行或已发布版本 |
| `POST` | `/api/schedule-drafts/:id/publish` | 发布草稿 |
| `POST` | `/api/schedule-drafts/:id/discard` | 废弃草稿 |

### 7.1 调整安排请求

```json
{
  "room_id": "r-main-101",
  "time_slot_id": "slot-2026-06-21-am",
  "teacher_ids": ["t-zhang", "t-li"]
}
```

请求字段可以部分提交。API 合并现有安排后进行完整校验。

### 7.2 草稿详情响应

```json
{
  "draft": {
    "id": "draft-001",
    "status": "validated",
    "sourceRunId": "run-001",
    "score": 86,
    "conflictCount": 0,
    "assignmentCount": 36
  },
  "assignments": [],
  "conflicts": [],
  "changeEvents": []
}
```

响应中的 `assignments` 应使用和 `ScheduleResult.assignments` 一致的字段，以便前端矩阵、版本对比和发布流程复用展示逻辑。

## 8. 约束校验设计

第二版不要求将所有人工调整纳入 CP-SAT 重新求解，而是优先使用确定性校验。

调整后必须校验：

1. 同一考场同一时间段不能多场考试。
2. 同一学生群体同一时间段不能多场考试。
3. 同一教师同一时间段不能多场监考。
4. 教师不可用时间不能安排监考。
5. 考场容量必须满足人数。
6. 考场类型和设备必须满足考试要求。
7. 时间段必须在考试任务允许范围内。
8. 每场考试监考教师数量必须满足要求。

其中第 1-6 项可复用第一版 `detect_assignment_conflicts()` 的思想；第 7-8 项需要在第二版补充为人工调整专用校验。

## 9. 前端信息架构

第二版 Web 重点新增“方案工作台”。

### 9.1 方案工作台

方案工作台包含：

- 左侧：草稿列表和来源版本。
- 顶部：草稿状态、评分、冲突数、发布按钮。
- 主区域：时间段 × 考场矩阵。
- 右侧：选中考试详情、调整表单、冲突解释和变更记录。

### 9.2 排考矩阵

矩阵按以下方式组织：

```text
              考场 A       考场 B       考场 C
时间段 1      考试卡片      空           考试卡片
时间段 2      空           考试卡片      空
时间段 3      考试卡片      考试卡片      空
```

考试卡片展示：

- 课程名称。
- 学生群体。
- 人数。
- 监考教师。
- 冲突标记。

第一版矩阵可以先用点击选择和表单调整，不强制实现拖拽。拖拽可作为体验增强项。

### 9.3 发布确认面板

发布确认面板展示：

- 当前草稿摘要。
- 硬冲突数量。
- 评分变化。
- 安排变化数量。
- 最近 10 条调整记录。

硬冲突数量大于 0 时，发布按钮禁用，并展示阻塞原因。

## 10. 算法与服务边界

第二版继续遵守“算法核心与 Web/API 解耦”原则：

- 自动排考仍由 `apps/scheduler` 完成。
- 人工调整校验由 API 层组织数据后调用共享校验逻辑。
- Web 层不得自行判断约束是否通过。
- 数据库只保存事实和版本，不嵌入求解逻辑。

如果第二版需要从草稿重新优化局部方案，应新增独立接口，例如“基于当前草稿重排未锁定考试”，不应把它混入单场调整接口。

## 11. 测试方案

第二版测试分为 4 层。

### 11.1 API 单元测试

必须覆盖：

- 从运行创建草稿。
- 读取草稿详情。
- 调整单场考试安排。
- 调整后产生冲突。
- 修复冲突后草稿恢复可发布。
- 有硬冲突时发布失败。
- 无硬冲突时发布成功并更新当前发布版本。

### 11.2 仓储测试

必须覆盖：

- 草稿、草稿安排、草稿冲突、变更事件写入。
- 草稿状态迁移。
- 从 `schedule_run` 复制安排到草稿。
- 发布草稿时生成或更新正式运行版本。

### 11.3 前端构建与交互验证

必须覆盖：

- `npm run typecheck`。
- `npm run build`。
- 手动或自动验证矩阵页面可以展示草稿、选择考试、提交调整。

### 11.4 调度器回归测试

第二版不应破坏第一版算法能力，仍需保持：

```bash
cd apps/scheduler
uv run --python 3.12 --extra dev python -m pytest -q
```

## 12. 验收标准

第二版完成后应满足以下验收标准：

1. 可以从任意可读排考运行创建草稿。
2. 草稿中的单场考试可以调整时间段、考场和监考教师。
3. 调整后系统可以即时输出硬约束冲突和处理建议。
4. 无硬冲突草稿可以发布为当前方案。
5. 有硬冲突草稿不能直接发布。
6. 发布、废弃、调整均写入审计。
7. Web 运营台提供可演示的时间段 × 考场矩阵。
8. 当前已发布查询仍可读取最新发布方案。
9. 第一版算法测试、API 测试、前端类型检查和构建保持通过。
10. `docs/status/project_status.md` 能反映第二版进展、风险和验证结果。

## 13. 风险与取舍

### 13.1 草稿与运行版本的边界

草稿是可编辑工作副本，运行版本是自动排考或发布后的稳定结果。两者不能混用，否则会导致版本追溯困难。

### 13.2 删除基础数据后的历史一致性

当前系统已经支持基础数据删除。第二版草稿和已发布方案需要尽量保留安排中的 ID，并在展示时允许引用对象缺失，避免历史版本因基础数据变化无法打开。

### 13.3 矩阵交互复杂度

拖拽体验价值高，但边界复杂。第二版先实现点击选择和表单调整，保证课程验收和业务闭环；拖拽作为后续增强。

### 13.4 软约束优化深度

第二版重点是人工调整和硬约束校验，不强制把软约束全部纳入 CP-SAT 优化目标。评分可以继续作为解释和比较指标。

## 14. 后续衔接

第二版完成后，第三版可继续推进：

1. 异步排考任务队列和实时进度推送。
2. 拖拽式矩阵调整。
3. 局部重排和锁定考试。
4. 登录、角色和权限。
5. 教师不可用时间自助维护。
6. 学生变更通知和导出。
7. 课程报告终稿、部署说明和演示脚本。

## 15. 当前结论

第二版应聚焦“人机协同排考”和“可审计发布治理”。这比继续扩展普通后台 CRUD 更能体现排考系统的业务价值，也能承接总体设计中关于方案版本、冲突检测、人工调整、统计分析和审计日志的长期方向。
