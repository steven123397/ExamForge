# 设计文档规则与模板

本目录保存长期有效的系统设计事实，包括领域边界、架构、数据合同、算法模型、接口语义和跨模块约束。短期执行清单写入 `docs/plan/`，当前进展和风险写入 `docs/status/`。

`CONTEXT.md` 只维护规范术语和概念边界。版本特有的不变量、关键取舍、失败语义和迁移影响仍写入对应设计文档。

## 形成流程

1. 读取 `CONTEXT.md`、相关 design 和必要的代码事实。
2. 区分已知事实、待做决策和未定义术语。
3. 对关键歧义按需进行逐决策 Grill、domain modeling、研究或最小原型。
4. 将确认后的规范术语同步到 `CONTEXT.md`。
5. 将已收敛结论综合写入 design，不保存问答实录。
6. 完成术语、场景、决策、不变量、失败边界和验证原则自检。
7. design 达到可执行状态后再编写阶段计划。

新设计和当前活跃设计采用该流程；历史文档只在实际触及时逐步收敛，不批量重写。

## 建议文件命名

- `总体设计与技术选型.md`
- `第N版实现内容设计.md`
- `<模块>设计.md`
- `<主题>接口与数据合同.md`

## 模板

以下章节按风险选用，不要求机械填满：

```markdown
# <设计主题>

## 1. 范围与背景

说明问题、事实证据、约束来源，以及本设计与现有系统的关系。

## 2. 目标与非目标

- 目标：
- 非目标：

## 3. 领域术语与关键场景

引用 `CONTEXT.md` 中的规范术语，补充本设计特有的使用场景和边界案例。

## 4. 状态与长期不变量

记录状态迁移、必须始终成立的规则，以及禁止的隐式回退。

## 5. 方案与模块边界

说明职责、公共接口、隐藏复杂度、依赖、数据流和所有权。

## 6. 关键决策

### 决策：<题目>

- 结论：
- 理由与证据：
- 被拒绝的替代方案：
- 后果与重新评估条件：

## 7. 失败、安全与迁移

说明错误语义、权限、兼容、数据迁移、回滚和真实依赖失败。

## 8. 测试接缝与验证原则

说明调用方和测试共同使用的公共接缝，以及需要的机械、行为和共识证据。

## 9. 延后决策与停止线

记录尚不可规划的内容、明确非目标和重新评估触发条件。

## 10. 后续影响

说明受影响的计划、状态、接口、数据和课程报告材料。
```

## 自检

- 规范术语一致，没有同义词漂移。
- 每个目标、场景和不变量都有明确边界。
- 关键决策包含结论、理由、替代方案和后果。
- 模块以小而稳定的公共接口隐藏复杂度。
- 调用方和测试通过相同公共接缝观察行为。
- 没有混入任务清单、当前进度、聊天实录或未解释的占位符。
- 没有单独创建 ADR；决策直接归属于相关 design。
