# 第五版前端信息架构与视觉规范

## 1. 文档定位

本文档固定 ExamForge 第五版页面级重构的信息架构、角色导航、视觉语言、组件状态和响应式验收口径。它是第五阶段实现完成后的长期前端基线，不替代 API 权限、领域状态机或阶段实施计划。

第五版第五阶段已经完成真实路由拆分、URL 状态、教师/学生服务端作用域、自助维护、可访问拖拽和完整页面重构。本文列出的固定路由均已落地；后续部署或功能演进不得退回根路由聚合控制台，也不得用客户端参数替代本人作用域。

## 2. 设计方向

设计主题为“排考控制面”：安静、工业化、紧凑，优先支持长时间扫描、异常定位和重复操作。

- 第一视觉优先级是草稿矩阵、作业状态、冲突和待处理操作，不是欢迎语或品牌展示。
- 页面采用中性灰白工作面、深色导航、青绿色主操作、琥珀色提醒、红色阻断和绿色成功，避免米色或单一蓝紫色主导。
- 信息分组依赖标题、分隔线、表格列和固定工具栏，不把每个区块包装成悬浮卡片。
- 交互密度以桌面运营场景为主；移动端保证查询、通知和基本状态查看，不强求完整复刻桌面矩阵操作。
- 动效只用于状态变化、抽屉和焦点转移，默认 120 至 180 ms；尊重 `prefers-reduced-motion`。

## 3. 角色与核心任务

| 角色 | 默认入口 | 高频任务 | 服务端权限边界 |
| --- | --- | --- | --- |
| 管理员 | `/admin/overview` | 数据治理、排考、发布、审计、账户与策略 | 删除、发布、回滚和角色管理必须由 API 再授权 |
| 排考员 | `/scheduling/jobs` | 提交作业、跟踪进度、运行对比、草稿调整 | 不拥有管理员专属删除、账户和系统配置权限 |
| 教师 | `/teacher/schedule` | 查看本人监考、变更通知、维护不可用时间 | 不能读取全量运营数据或修改排考结果 |
| 学生 | `/student/schedule` | 查看所属群体考试与变更通知 | 不能读取其他群体或任何运营数据 |

未认证访问内部路由统一跳转 `/login?returnTo=...`。认证后按服务端角色决定默认入口；客户端路由守卫只改善体验，不能替代 API 权限检查。

## 4. 路由与导航

### 4.1 固定路由

| 路由 | 页面职责 | 可见角色 | 导航组 |
| --- | --- | --- | --- |
| `/login` | 登录、错误密码、停用账户和会话恢复 | 匿名 | 无 |
| `/admin/overview` | 数据健康、当前批次、作业与冲突摘要 | 管理员、排考员 | 管理 |
| `/admin/reference-data` | 课程、考场、教师、学生群体、时段和任务 | 管理员、排考员 | 管理 |
| `/scheduling/jobs` | 作业阶段、尝试、耗时、错误和结果入口 | 管理员、排考员 | 排考 |
| `/scheduling/runs` | 运行历史、对比、发布和回滚 | 管理员、排考员 | 排考 |
| `/scheduling/drafts/:id` | 草稿矩阵、检查器、建议、锁定、校验和发布 | 管理员、排考员 | 排考 |
| `/scheduling/policies` | 约束策略、权重和版本差异 | 管理员、排考员 | 排考 |
| `/audit` | actor、实体、动作、结果和 trace/job 关联 | 管理员 | 治理 |
| `/teacher/schedule` | 本人监考、不可用时间和变更 | 教师 | 本人任务 |
| `/student/schedule` | 所属群体考试和变更 | 学生 | 我的考试 |

### 4.2 URL 状态

筛选、分页、排序、选中实体和可分享上下文写入 URL：

- 基础数据：`resource`、`id`。
- 作业：`status`、`submittedBy`、`constraintProfileVersionId`、`from`、`to`、`page`、`pageSize`、`jobId`。
- 运行：`status`、`runId`、`compareTo`、`page`、`pageSize`。
- 草稿：路径保存 `draftId`，查询参数保存 `examTaskId`、`view=matrix|list`、`conflict`。
- 审计：`actor`、`action`、`entityType`、`entityId`、`traceId`、`from`、`to`、`page`、`pageSize`。
- 教师和学生查询不接受可越权的用户 ID；身份范围由服务端会话确定。

刷新页面必须保留主要位置。无权访问的深链显示权限不足页；实体不存在显示 404；会话失效保存安全的 `returnTo` 后进入登录页。

### 4.3 导航结构

桌面管理员与排考员使用 240 至 264 px 侧边导航，顶部固定当前批次、全局作业入口和会话菜单。二级筛选放在页面标题下的紧凑工具栏，不重复放入侧栏。

教师与学生不显示运营侧栏，使用窄顶栏和最多两个一级入口。移动端采用顶栏加抽屉导航；教师、学生的查询与通知保持第一屏可用，草稿矩阵在移动端默认切换为列表并提供“使用桌面调整”的非阻断提示。

## 5. 关键页面原型

### 5.1 登录与概览

- 登录使用 360 至 420 px 单列表单，品牌只作为身份确认，不设置营销英雄区。
- 用户名和密码使用可见标签；错误贴近表单并由 `role="alert"` 播报。
- 提交中锁定重复提交但保留按钮尺寸；会话恢复不闪现运营内容。
- 概览首屏依次为标题与主操作、紧凑指标带、当前作业与阻断冲突、数据健康摘要。
- 指标只显示能触发判断的数据，并提供目标页面入口，不使用装饰性图表。

### 5.2 作业中心

- 默认列为状态、阶段、进度、批次、尝试、排队耗时、运行耗时、提交人、更新时间和结果。
- 状态和阶段分列。`succeeded` 可同时包含 `feasible` 或 `infeasible` 业务结果。
- 失败行展开后显示错误类别、错误码、是否可重试、最近尝试和 trace ID。
- SSE 断开时保留最后持久化状态，显示实时连接与重连状态，不伪造作业失败。

### 5.3 运行与草稿

- 运行页以版本表格和双版本对比为主；发布、回滚等高影响动作使用确认对话框并显示目标 ID。
- 草稿桌面布局为 240 px 草稿列表、弹性矩阵、300 至 340 px 检查器。矩阵至少获得剩余宽度的 55%。
- 冲突筛选、锁定状态和未保存变更在矩阵上方固定工具栏显示。
- 拖拽只是快捷方式；键盘用户可在检查器中通过时段、考场和监考教师表单完成相同调整。
- 长课程名、教师名和 ID 不得扩大网格轨道；截断时提供完整可访问名称。

### 5.4 策略与审计

- 策略页比较当前生效版本与草稿版本；权重使用数字控件，二元规则使用开关，枚举使用选择菜单。
- 审计以筛选工具栏和高密度表格为主；actor、动作、实体、结果、时间、trace/job ID 均可筛选或复制。
- 原始 payload 放在按需展开区，不在主表直接铺开 JSON。

### 5.5 教师与学生查询

- 第一屏显示下一场安排、变更数量和按日期分组的列表。
- 教师页面额外提供本人不可用时间维护；学生页面不暴露教师工作量或内部冲突解释。
- 375 px 宽度下使用单列日程项，日期、时间、地点和对象均可换行，禁止页面级水平滚动。

## 6. 设计 Token

### 6.1 颜色

| Token | 值 | 用途 |
| --- | --- | --- |
| `--paper` | `#f3f5f4` | 页面工作面 |
| `--panel` | `#ffffff` | 表格、表单和工具区背景 |
| `--ink` | `#1d2522` | 主文本 |
| `--muted` | `#626d68` | 次级文本，不能用于关键状态 |
| `--line` | `#d4dbd7` | 分隔与输入边界 |
| `--rail` | `#202825` | 运营导航 |
| `--teal` | `#0f766e` | 主操作与选中状态 |
| `--copper` | `#a34b08` | 提醒、阶段与待处理 |
| `--rose` | `#be123c` | 错误、阻断和危险动作 |
| `--green` | `#15803d` | 成功和已发布 |
| `--focus` | `#087ca7` | 键盘焦点，不与状态色混用 |

状态不能只依赖颜色，必须同时显示文本或图标。正文与背景对比度至少 4.5:1，大号文本和非文本控件至少 3:1。

### 6.2 排版、间距与尺寸

- 中文正文优先 `Noto Sans CJK SC` 或 `Source Han Sans SC`，拉丁和数字优先 `IBM Plex Sans`；代码和 ID 使用仓库现有等宽字体。
- 页面标题桌面 28 px、移动 24 px；区块标题 18 px；正文 14 px；表格与辅助信息 12 至 13 px。字号不随视口宽度线性缩放。
- 字间距固定为 0。正文行高 1.5，标题行高 1.2。
- 间距基线为 4 px；常用间距为 8、12、16、24、32 px。控件高度为 38 至 44 px，图标按钮固定 38 px。
- 圆角以 6 px 为主、8 px 为上限；阴影只用于对话框、弹出层和原型期有限分组。

## 7. 组件与状态

### 7.1 表格、筛选与表单

- 表头在长列表中可粘附；数值右对齐，动作列宽度稳定，空值统一为 `--`。
- 筛选使用真实表单控件并支持清除全部；已应用条件以紧凑摘要显示，不堆叠胶囊标签。
- 移动查询把表格转换为带字段名的行组；运营表格允许容器内横向滚动，但不得造成页面级横向滚动。
- 输入必须有持久标签、帮助或错误文本和程序化关联。占位符不能替代标签。
- 每个工具区最多一个主按钮；熟悉的关闭、退出、刷新等命令可用带 tooltip 的图标按钮。
- 删除、发布、回滚、废弃等操作使用确认对话框，展示实体名称和 ID，并防止重复提交。

### 7.2 标准页面状态

| 状态 | 表达 |
| --- | --- |
| 加载 | 保持稳定尺寸的骨架或局部加载文本，不清空已成功数据 |
| 空态 | 说明当前集合为空，只在有权限时给出下一条有效命令 |
| 可重试错误 | 在受影响区块内显示原因摘要和“重试”按钮 |
| 权限不足 | 显示缺少的访问范围和安全返回入口，不展示危险操作 |
| 会话失效 | 清理私有查询缓存并进入登录页，保留安全 `returnTo` |
| 依赖不可用 | 区分数据库、队列、实时事件和 scheduler，保留可用只读能力 |
| 冲突阻断 | 同时显示红色图标、数量、原因和定位入口 |

键盘焦点使用统一 3 px 外轮廓。对话框打开后焦点进入标题或首个安全控件，关闭后回到触发器。动态错误和作业终态分别使用合适的 `aria-live` 级别，避免连续播报进度百分比。

## 8. 响应式与验收

| 视口 | 目标 |
| --- | --- |
| 1600 x 1000 | 完整运营布局、三栏草稿矩阵、稳定工具栏 |
| 1280 x 800 | 运营导航和主要表格不遮挡，矩阵仍可操作 |
| 768 x 1024 | 单列内容、抽屉导航、查询和审批可用 |
| 375 x 812 | 登录、教师/学生查询、通知和基本状态无页面级横向滚动 |

每个关键页面验证最长中文名称、64 字符 ID、无数据、加载、403、404、500、依赖不可用、200% 文本缩放、键盘遍历、焦点可见、对比度、控件遮挡和文本溢出。截图基线至少覆盖登录、概览、作业、草稿桌面、教师移动和学生移动。

## 9. 当前落地边界

### 9.1 路由与权限

- `/login`、管理员、排考、审计、教师本人和学生本人路由均已拆分；根 `/` 只按服务端会话角色分流。
- 管理员/排考员使用工作型侧栏，教师/学生使用本人任务顶栏；筛选、页码、对比和选中实体进入 URL，可深链、刷新和前进后退。
- 401 会清理私有 Query cache，错误角色深链显示 403，不存在页面或实体显示 404；客户端 RouteGuard 只改善体验，API/use-case 继续独立授权。
- 教师和学生页面只调用 `/api/me/*`。教师单作用域、学生多群体作用域由 PostgreSQL 关联决定，页面不接受可切换其他对象的 ID。

### 9.2 工作流组件

- `/scheduling/jobs` 与 `/scheduling/policies` 复用既有 Query cache、SSE 事件恢复、持久化 attempt/事件、策略不可变版本和服务端权限合同。
- `/scheduling/runs` 与 `/audit` 分别承载运行、对比、发布/回滚和审计筛选；跨任务、运行、草稿和审计使用真实链接。
- `/scheduling/drafts/:id` 是草稿唯一编辑入口。dnd-kit 提供 pointer 与 keyboard sensor，检查器表单提供等价调整能力；锁定、终态和建议响应代次共同约束 mutation。
- `/teacher/schedule` 支持本人不可用时间维护；`/student/schedule` 只呈现所属群体考试，不展示内部冲突、教师负载或策略事实。

### 9.3 视觉门禁

- 样式按 token、布局、表单和 feature CSS module 分层；页面保持中性工作面、紧凑信息密度、清晰状态色和可见焦点。
- Playwright 固定 1600 x 1000、1280 x 800、768 x 1024、375 x 812 四种视口，并保存登录、概览、作业、草稿桌面及教师、学生移动端基线。
- Axe 自动扫描覆盖登录与全部关键角色页面；键盘拖拽、跳到主内容、200% 文本、长 ID、403、404、依赖失败、对比度和页面级溢出另有专项断言。
- 移动端优先保证登录、本人查询、基本任务状态和无横向溢出；草稿矩阵在窄屏使用列表/检查器路径，不强行缩放桌面矩阵。

## 10. 变更治理

前端实现若偏离本文档，应先确认角色任务、URL 状态或可访问性是否发生长期变化。长期变化更新本文档；阶段任务和验证命令写入 `docs/plan/`；已完成事实与实际验证结果写入 `docs/status/project_status.md`。
